### S4: 스킬 이용권 카드

기존 쿠폰 카드 형식을 따름.

**스킬 이용권 카드 렌더 분기 규칙 (클라이언트 공통)**

api-spec.md는 `GET /api/coupon` 응답의 두 필드를 **의미적으로 독립**하도록 정의한다 — `fixedMenuSeq`는 "단일 스킬에 고정된 쿠폰의 스킬 식별자"(api-spec.md:249), `isUnlimited`는 "유효기간 무제한"(api-spec.md:251). 현재 스킬 이용권은 우연히 두 속성을 모두 가지지만, 두 필드는 서로 직교(orthogonal)한 개념이다. 따라서 스킬 이용권 카드의 분기도 **두 종류로 나누어 적용**한다:

1. **"스킬 이용권 상품 자체를 식별"하는 분기** — 판별 기준 `isUnlimited === true && fixedMenuSeq != null` (AND)
   - **배지 "이용권" 노출** (ISS-040)
   - **부가 설명 행 숨김** (ISS-041/046, invisible reserve)
   - 사유: "이 쿠폰이 스킬 이용권 상품 그 자체다"라는 **의미론적 판단**. 두 속성이 모두 있어야 스킬 이용권으로 분류.

2. **"개별 필드 속성에 반응"하는 분기** — 각 필드 단독 조건 (api-spec.md 적용 규칙 §247-258 준수)
   - **만료일 / 만료임박 숨김** (ISS-021/022): `isUnlimited === true` 단독 — 향후 "무제한 일반 쿠폰"이 도입되어도 만료일이 자연스럽게 숨겨지도록 (api-spec.md:251 "카드 UI에서 만료일/만료임박 행 미표시 권장")
   - **"스킬 보러가기" 링크 노출** (ISS-019/024): `fixedMenuSeq != null` 단독 — 향후 "스킬 지정 + 유한 만료" 쿠폰이 도입되어도 스킬 상세 숏컷이 유지되도록 (api-spec.md:257 "존재 시 쿠폰 카드 탭 → SkillDetail(fixedMenuSeq) 네비게이션")

> ⚠️ 과거(~2026-04-23) "모든 분기 동일 기준(AND)" 서술은 ISS-040 배지 derive 문구의 과잉 일반화였으며, api-spec.md의 필드 독립성 계약과 모순이라 폐기. 2026-04-24 `/architect`가 api-spec.md 정합성 방향으로 재정의. 코드(iOS/Android)는 이미 api-spec 규약을 따르고 있어 코드 수정 영향 없음. Web은 회귀 확인 대상.

**레이아웃** (Figma node `10:12563`, `10:12592`, `10:12594`, `10:12622` 공통):

```
┌─────────────────────────────────────────┐
│ 100%                      [이용권][앱전용]  │  상단: 좌=할인율, 우=태그 (앱전용은 조건부)
│ {{스킬명}} 이용권                          │
│ (부가 설명 자리 — invisible reserve)        │  ← 스킬 이용권은 공간만 유지, 문구 미노출
│ ┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈┈ │  점선 구분선 1px `#EDEDEE`
│ [N일 남음 │] 유효기간        스킬 보러가기 > │  하단: 좌=기한, 우=링크
└─────────────────────────────────────────┘
```

- **상단 영역**: 좌측에 할인율(`100%`), 우측에 태그들(`이용권`, 필요시 `앱전용` 추가) — `space-between` 정렬
- **부가 설명 행 (invisible reserve)**: 스킬 이용권 카드에서는 문구 자체(`"0하트 이상 결제 시 사용 가능"`)가 무의미하므로 **시각적으로 숨기되 레이아웃 공간은 유지**. 일반 쿠폰과 동일한 카드 높이/수직 리듬을 확보하여 쿠폰함 리스트에서 두 카드 타입이 섞여도 시각적 어긋남이 없도록 함. 스크린리더(TalkBack/VoiceOver)는 해당 행을 **읽지 않아야 함**.
- **하단 영역**: 좌측에 (만료 임박 "N일 남음" + 구분선 `│`)? + 유효기간, 우측에 "스킬 보러가기 >" 링크 — `space-between` 정렬
- "스킬 보러가기 >" 링크는 **반드시 하단 우측 정렬** (좌측 정렬 금지)
- **무제한 쿠폰 (`isUnlimited === true`)**: 하단 좌측 영역을 비워 렌더링하고, 점선 구분선도 숨김(또는 링크만 표시되는 단일 행으로 대체). sentinel 날짜(`2099-12-31T23:59:59.000Z`) 직접 비교 금지 — 반드시 `isUnlimited` 플래그로 분기
- **각 요소의 표시 조건**: 위 "스킬 이용권 카드 렌더 분기 규칙" 참조. 요소별로 `isSkillVoucher`(AND)·`isUnlimited` 단독·`fixedMenuSeq` 단독 조건이 **의도적으로 다름**

**플랫폼별 구현 레시피 (부가 설명 invisible reserve)**:
- **Web**: Tailwind `invisible` 클래스 (display 유지, visibility hidden). 또는 `visibility: hidden`. — ISS-031 구현 참조.
- **Android (Compose)**: `Modifier.alpha(0f)` + `Modifier.clearAndSetSemantics { }` 체이닝. `alpha` 단독은 TalkBack이 여전히 읽으므로 semantics 차단 필수. — ISS-046 구현 참조.
- **iOS (UIKit)**: `descriptionLabel.isHidden = true` + **FlexLayout `isIncludedInLayout = true` 유지**(기본값, FlexLayout에서 `isHidden`만으로는 공간이 collapse되지 않음). `isIncludedInLayout = false` 조합(collapse)은 금지 — 카드 높이가 축소되어 일반 쿠폰과 불일치. ISS-041 재조정(2026-04-23)으로 이미 적용 완료. 접근성은 `isHidden = true` 상태에서 VoiceOver가 자동 제외함.

> ⚠️ collapse(`display: none` / `isHidden = true` + `flex.isIncludedInLayout = false`) 방식 **금지**. 스펙 위반. 플랫폼 관계없이 reserve로 통일 (2026-04-24 /architect 결정, 3 플랫폼 구현 완료 상태).

### 쿠폰 등록 버튼 로딩 상태 (2026-04-22 ISS-042 확정)

등록 플로우에서 응답 대기 동안의 버튼 UI를 다음과 같이 정의합니다. "공통 컴포넌트 — 등록 버튼"의 로딩 상태 스타일을 사용합니다.

| 단계 | 버튼 상태 | 설명 |
|------|---------|------|
| 탭 전 (입력 유효) | 활성 (`#FFE967` / `#242526`) | 텍스트 "등록" |
| 탭 직후 ~ 응답 수신 전 | 비활성 + 회색 (`#C6C8CC` / `#EDEDEE`) | **스피너/로딩 인디케이터 없음**. 탭 불가. |
| 응답 수신 (성공/실패) | 활성 또는 비활성(empty) 복귀 | 성공 시 입력 초기화 → empty 비활성. 실패 시 입력 유지 → 활성 복귀. |

- **스피너 노출 금지 원칙**: 회색 비활성 톤 자체가 진행 상태 피드백. 버튼 내부 인라인 스피너, 풀스크린 오버레이, `CircularProgressIndicator` 등 일체 신규 노출 금지.
- **색상 토큰**: 기존 input empty 상태의 disabled 색상(`#C6C8CC` 배경 / `#EDEDEE` 텍스트)을 재사용 — 신규 색상 토큰 도입 불필요.
- **중복 탭 방지**: 버튼 비활성화로 UI 레벨에서 차단. throttle/debounce와 별개로 상태 플래그 기반 차단이 필수.
- **레거시 예외 (점진 정합)**:
  - Web — 풀스크린 `<Loading />` 오버레이는 제거 완료 (2026-04-22). 단, 동시 도입된 버튼 내 인라인 스피너(`animate-spin` 16×16)는 본 정책(스피너 금지)과 미정합 — 후속 정합 대상.
  - Android — ISS-042 초기 해결(2026-04-22)에서 도입된 `CircularProgressIndicator(16dp, Gray900)`는 2026-04-23 rollback으로 제거 완료 — 회색 비활성 톤만 유지 (본 정책 정합).
  - iOS — ISS-042 미해결. 본 확정에 따라 **버튼 비활성 + 회색 톤만** 적용하고 스피너 도입하지 않음.
