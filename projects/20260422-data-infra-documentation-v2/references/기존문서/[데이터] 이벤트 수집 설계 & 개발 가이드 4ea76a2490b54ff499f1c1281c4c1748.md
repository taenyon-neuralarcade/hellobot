# 헬로우봇, 스플 사용자 개념과 서비스 내 로직

### 익명 사용자 vs 가입 사용자 개념

- 헬로우봇, 스플 서비스에서 **미가입 상태에서도 스킬이나 작품을 사용할 수 있도록 하기 위해 익명 사용자의 개념이 도입**되었음. (익명 사용자 상태에서도 서비스 후반부의 콘텐츠 사용이나 구매 등의 전환이 이루어질 수 있음)
- **익명 사용자:** 미가입 사용자의 서비스 이용을 위해 서버에서 임시로 만드는 계정 (사용자의 개인정보만 없고, user_id 를 가지고 있음)
    - 미가입 사용자가 챗봇 콘텐츠인 스킬(헬로우봇), 작품(스플)을 이용하기 위해, 구매 등의 기록을 위해 사용자 계정이 필요.
- **가입 사용자:** 사용자가 회원가입 절차를 진행하여 개인정보를 입력한 계정
- 해당 구조로 인한 특징
    1. 서비스 최초 접속시 앱/웹(프론트)에서 서버에 최초 요청할 때 사용자 계정이 없는 상태(최초 접속, 로그아웃/회원탈퇴 후)라면 **서버에서 신규 익명 사용자 계정을 생성**
    2. 회원가입 시 회원가입 이전에 사용하고 있던 익명 사용자 계정에 사용자가 입력한 개인정보를 업데이트 하여 가입 사용자로 전환 (user_id가 변경되지 않음. 사용자의 속성값(가입일, 회원타입 등)이 변경됨)
    3. 서비스에 새로 접속(익명 사용자 계정 생성)하여 기존 가입 사용자 계정으로 로그인 하면, 로그인 전의 익명 사용자 계정은 더이상 사용할 수 없는 상태가 됨
        1. 실제로 계정을 삭제하지는 않지만 재사용되지 않음
    4. 가입 사용자 계정으로 로그인하여 사용하다가 로그아웃하면 새로운 익명 사용자 계정이 생성됨
        1. 앱에서 익명 사용자 계정 생성 요청하는 방식
    5. 익명 사용자 계정은 회원가입, 로그인 등의 액션이 없는 한 앱/웹(프론트)의 캐시 수명 주기만큼 유지됨. (종료 후 재실행 정도까지는 유지됨)

- **사용자 계정의 상태 변화** (**헬로우봇, 스플** 공통 사항)

**User ID, User Property는 발송하는 모든 이벤트에 적용되는 기능이라 중요도가 매우 높음**

- 사용자의 상태가 변할 때 user_id와 user_property가 **현실의 상태를 반영하도록 변경**되어야 함
    - 주로 회원가입, 로그인, 로그아웃 등 계정의 상태가 변할 때
    - 로그인 후 설정 등에서 user_property에서 추적하는 값을 변경할 때, 해당 속성값을 맞게 변경해 주어야 함. (사용자 상태가 실제 ON인데 장애등의 이유로 이벤트가 OFF인 상태가 많으면 해당 필드값은 사용하지 못함.)
    - 아무리 잘해도 여러가지 이유로 실제 상태와 깨진 상태 발생 가능 ⇒ 최초 접속 시점에 현재 계정의 상태를 확인하고 업데이트 해주는 등의 보완 코드가 추가되면, 정확도를 좀 더 올릴 수 있음
- 속성값 제거 방법: 속성 값을 초기화 (들어가 있던 값을 삭제)하는 기능은 없는 것 같음 ⇒ null이나 ‘’(empty string) 등으로 세팅하는 형태로 제거
- 최초 접속시부터 user_id는 존재 - 기능적 이유로 앱/웹(프론트) 접속 직후부터 서버 요청 1회 진행 전까지는 user_id 가 없을 수 있음 (예시. first_open, 웹에서 상세 페이지로 바로 접근하는 경우 조회 이벤트)
    - user_id는 알게되는 시점에 최대한 빨리 세팅해주면 좋음

### Firebase Analytics - User Property 동작 방식

- **사용자 속성(user properties)은 사용자의 user_id 기준으로 상태가 변경되는 시점에 업데이트를 반드시 해 주어야 합니다. (모든 이벤트에 영향)**
- user property는 1명의 사용자의 속성(상태)을 나타내는 전역 변수 개념
    - user_id
    - user_property - 앱/웹 서비스에서 세팅하는 정보
- Firebase로 리포트하는 모든 이벤트에 현재의 user_property 정보를 붙여서 리포트함
- user_id / user_property 내 모든 정보들은 각각 독립적 ⇒ 전체를 1명의 사용자 상태로 사용하고 있으므로 업데이트 시 전체 항목을 함께 고려해야 함
- **⚠️ 주의!** 추가된 정보 unset 기능 없음 ⇒ setUserProperties(analytics, {삭제할 정보 항목: null}) 처럼 null로 set 해주는 방식으로 정보 제거
    - null로 set이 안될 수도 있음. 확인필요. 안될 시 ‘’(empty string)으로 set

# 이벤트 설계 및 구현 가이드

- **이벤트가 1개 발송되었다는 것은 사용자가 정상적으로 이 액션을 1회 완료했다는 것을 의미**하도록 구현해야 함 (정상적으로 완료라는 부분이 사람마다 생각이 다를 수 있음 - 조금이라도 헷갈리면 크로스 체크)

### **기본 원칙**

- 모든 액션은 **해당 액션이 기술적으로 완전히 완료된 시점의 이벤트 사용**
- 최대한 해당 액션이 정상적으로 완료되었음이 인지되는 타이밍에 이벤트를 발송해야 함
    - ex) 회원가입 시 서버에서 회원가입 완료 응답을 정상적으로 받았을 때 sign_up 이벤트 발송
        - 서버 장애시 = 회원가입 수는 급감해야 함.
        - 액션 시작 시점에 발송하면, 회원가입이 안 될 경우 사용자는 회원가입 버튼을 반복적으로 클릭 ⇒ 서버 장애시 회원가입 지표가 급증할 수 있음
- bool 타입 파라미터 가급적 사용하지 않기([히스토리](https://thingsflow.slack.com/archives/C01CDC7CL0J/p1711087066084269?thread_ts=1711069224.962379&cid=C01CDC7CL0J))
    - 개발을 bool 타입으로 해도 ga에서 웹만 string으로 변환을 해서 수집함

### 이벤트 수집 중요도

- **레벨 1 (Regression Test 필수) - 전사 목표, 본부 목표 관련 지표**
    - 반복적으로 모니터링해야 하는 지표 관련 데이터. (전사 성과 지표, 서비스 운영시 모니터링 지표)
    - → 모든 배포 전에 QA하여 이상없음 확인 (X. 이것도 어려움)
    - → 데이터팀내 이벤트 모니터링 대시보드 운영 + 해당 데이터 사용하는 대시보드 수치 이상시 확인하는 방식으로 관리
- **레벨 2 공통 이벤트 - 다양한 목적으로 활용될 수 있는 이벤트**
    - 모든 페이지 뷰
    - 필수 이벤트의 앱/웹 이벤트 (GA console 활용 목적)
- **레벨 3 1회성 성과 측정, 분석 목적용 이벤트**
    - 기능 출시 성과 측정, 단기적 기능 개선을 위한 데이터 분석 목적 지표 관련 데이터 → 해당 배포전 정상 동작 QA.
    - 매 스프린트 regression test는 하지 않으나, 한동안 안쓰던 지표 데이터는 사용 전 QA하여 정상동작 확인하고 사용. (반복적 regression test 비용을 줄이고, 필요할 때 사용하는 비용이 올라가는 구조)
- 중요도 순서: 분석시 활용도가 높아서 정확도가 높아야 함
    - **액션 이벤트 (회원가입, 구매, 결제 등 주요 액션 완료 이벤트) > 화면 조회 이벤트 > 클릭 이벤트 > 그 외 이벤트**

### 이벤트 설계 순서 + 중요도

**필수 이벤트**

**: 이 기능 자체의 성공 여부 측정을 위한 데이터**

(특별히 view_ 이벤트로 수집, 판단하기 어려운 경우 외에는 화면 view_ 이벤트로 설계. 성과 판단에 중요한 화면에 대한 view_ 이벤트, 결제 등 핵심 액션 이벤트가 누락되지 않았으면 아래 항목은 거의 커버 됨)

- 누락 여부 판단을 위한 체크 포인트
1. **특정 기능에 대한 모든 진입점 포인트의 view_* 이벤트**
    1. 해당 기능의 유입 모수 역할을 함
    2. ex) 스토리 게임에 유입된 전체 사용자 수는?
        1. + 스토리 게임 탭 진입한 사용자
        2. + 스토리 게임 상세 화면으로 랜딩한 사용자
        3. + 스토리 게임 채팅 화면으로 랜딩한 사용자
        4. + 홈 랜딩한 사용자? 등
        5. 이라면 각 화면에 진입했다는 view_ 이벤트가 모두 있어야 함
2. **해당 기능의 최종 목표 완료 이벤트 (view_ or 액션 이벤트)**
    1. 작품 조회까지 됐으면 성공이라고 판단할 수 있으면 조회를 했는지 여부가 최종 Goal
3. **진입점 → 최종 목표 사이의 주요 이탈 지점 확인용 이벤트**
    1. 여정이 긴 퍼널이면 경로가 다양할 수 있음.
    2. ex) 작품 구매, 매출 목표라면,
        1. 진입 → 작품 상세 → 채팅방 → 구매 완료

**필수 외 이벤트**

: 핵심 기능 성공을 도와주는 부가 기능의 개별 성과 측정을 위한 데이터

: 부가 기능 각각 개별적으로 판단. (어떤 부가 기능이 중요한 기능인가? 부가 기능의 임팩트와 중요도 기준)

1. 진입점 이벤트 - 이 기능에 진입한 사용자 전체 판단
2. 최종 목표 완료 이벤트
3. 퍼널 주요 단계 이벤트

**공통**

1. 성과 측정을 위한 데이터 수집(사용자가 핵심 퍼널의 단계에 도달 했는가) 이벤트가 1순위. 필수.
    1. Key 값 필수
        1. 스플의 경우 user_id, 작품 관련 이벤트에는 story_id
        2. 또 있나?
    2. 기존 유사 이벤트의 네이밍, 규칙 등을 고려하여 설계
2. 세부 분석을 위한 데이터 수집 이벤트는 2순위. 필수는 아님.

        - 예시)
            - 이벤트명 : touch_chatbot

            - 문제 : 검색결과에서 해당 이벤트가 발생했을 때는 referral_tab_name 값이 NULL이라서 디버그 뷰에서 보이지 않는다.
- 파라미터가 `boolean` 타입이면, 디버그 뷰에서 `FALSE`는 0, `TRUE`는 1로 잡히는 점을 유의한다.
