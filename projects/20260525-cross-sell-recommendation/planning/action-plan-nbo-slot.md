## 1. Executive Summary — 최종 결론

| # | 결론 | 한 줄 근거 |

## 2. 결론에 이르는 근거 (분석 종합 / Evidence Chain)

### 2.1 기회 진단 — 왜 1→2 전환인가
- 지난 12개월 결제 ~79만 건·고객 ~28.8만 명. 매출 **59%가 웹**, 그 웹의 **88%가 사주** — 광고로 들어와 사주 한 편 사고 이탈하는 구조.
- 신규의 **익월 재구매 7.4%**, 6개월 잔존 ~6%. 열에 아홉이 한 번 사고 떠남.
- 재구매를 가르는 건 상품 종류가 아니라 **"두 번째 구매를 했는가"** ('쿠팡 가설' 기각: 소비주기 短/中/長 재구매율 21.5/20.0/19.8% 동일).
- 이탈 낭떠러지는 **1→2 한 곳(전환 31.5%)**, 넘기면 **2→3이 48%**. → 모든 레버를 이 한 지점에.
- 매출은 웹+앱 단골 **2.6%가 26%** 책임 → 신규를 단골로 키우는 게 최대 기회.

### 2.2 추천 방법 — 왜 co-purchase인가
- 5개 방법 backtest 적중률: **co-purchase 36.6%** > 인기순 32.5% > 같은주제 15.2% > 재구매점수순 1.7%(표본부족 노이즈).
- 주제 정보 섞어도 36.6→36.7 (무의미) — co-purchase가 이미 주제 이동을 학습. **정교한 개인화보다 단순 co-purchase가 우월.**
- 같은 세부주제 연속 구매는 ~18%뿐 — 대부분 다음 고민으로 이동(같은주제 좁히기가 실패한 이유).

### 2.3 실현 가능성 검증 — 실제 NBO 테이블 도출
- A 앱퍼스트 98,392명의 `f_menu→s_menu`로 **스킬 단위 NBO 테이블 도출 성공** (BQ 없이 로컬).
- **설계 파라미터 실증**: 최소표본 n≥30 → **234 앵커가 첫구매 93.9% 커버**, 나머지는 인기순 fallback (재구매점수순 1.7% 실패는 이 컷 부재 탓).
- **강신호 확인**: 최상위 추천이 토픽이 아니라 특정 형제 스킬 — 41402→41698 lift 41.8, 병오년→`[추가구매] 병오년` lift 241. 토픽 레벨 매트릭스로는 안 보이던 신호 → "콘텐츠 단위 추천" 결론의 실증.

### 2.4 비판 리뷰가 드러낸 것 — 왜 정제가 필요한가
5개 독립 분석가가 CSV 재계산으로 수렴 (상세 [`review-affinity-nbo-critical.md`](review-affinity-nbo-critical.md)):
- **affinity가 '관심' 아닌 '인세션 노출/번들' 학습** — 학습 풀의 **68%가 당일 구매**. 결제직후 슬롯이 과거 추천 surface를 재학습 → **자기강화(closed loop)**.
- **출력 오염** — rank-1의 **24%가 할인·충전·추가구매 SKU**, **24~35%가 판매불가**(을사년 시즌 14건 전부 판매중지).
- **방법 결함** — 양방향 pooling이 추천 시간방향 오염(top-1 5/7 변동), small-n 무의미(12개 중 10개 top-1이 통계적으로 1위 아님), lift 분모 오류(1.3~1.5배 오스케일).
- **모집단 불일치** — 앱 학습→웹 적용. 단 C(웹)가 데이터 3~7배·동일 클러스터 → 방법은 유효, 정제·모집단 정합만 필요.
- **Credit** — 연령 교란 거의 없음(age-표준화 변화 5.1%) → "demographic 개인화 무용" 뒷받침. affinity 순위가 인기로 환원 안 됨(Spearman 0.32) → 앵커 고유 신호 존재.

### 2.5 종합 → 액션 정의
방법(co-purchase)은 옳고 기회(1→2)는 크지만, **현 데이터는 노출 편향·판매불가·소표본으로 오염**돼 있다. 따라서 "데이터를 정제 → 가장 싼 채널로 증분 검증 → 제품화"가 합리적 경로다. 노출 자체가 부족하므로 **노출 확보가 1순위**이고, 증분은 **A/B로만** 확정한다.

## 3. 품질 개선 포인트 (리뷰 → 빌드 필수 요구사항)

> 아래는 "있으면 좋은 것"이 아니라 **적용 전 충족해야 할 산출/서빙 요구사항**이다. 각 항목은 리뷰 P# 근거.

### 3.1 데이터 정제 (NBO 산출 단계)
| 요구 | 내용 | 근거 |
|---|---|---|
| **판매상태 하드게이트** | `is_open=true AND is_stop_selling=false AND buyers_30d>0` 통과만 추천 후보 | P1 (rank-1 26% 판매불가) |
| **비콘텐츠·업셀 blocklist** | 질문권충전·하트·패키지·`(추천할인)`·`[추가구매]`·프리패스 menu_seq 제외 | P2 (rank-1 24% 머천 SKU) |
| **보유·동일스킬 제외** | 앵커 자기추천 + 유저 보유 스킬 제외(소모성은 화이트리스트) | P10 (rank-1 9.4% 자기추천) |
| **NaN 드롭 로깅** | menu_seq null 페어 2% 손실 정량·정체 확인(미태깅 SKU) → ISS 등록 | P11 |
| **카탈로그 window 일치** | 이름·상태 룩업을 학습창과 동일 window로 재추출(미상 8.6% 해소) | P12 |

### 3.2 방법론 (공식·구조)
| 요구 | 내용 | 근거 |
|---|---|---|
| **방향 보존** | NBO는 정방향만(`pairs[f==X].s`). 양방향 pooling 폐기(또는 별도 "연관묶음" 용도 명시) | P4 |
| **표면별 분리 산출** | 결제직후 슬롯 = `s_menu`(당일) 풀 / 재방문 푸시·홈탭 = `r_menu`(진짜재방문) 풀. 단일 엔진 금지 | P3 (당일/재방문 top5 48%만 일치) |
| **표본 게이트 코드화** | `pair_n≥30 AND anchor_n2≥50` CSV 산출단계 강제 + Wilson 하한 정렬 + rank1-rank2 CI 분리 시만 "추천" | P5 (12개 중 10개 top-1 무의미) |
| **lift 분모 정정** | `P(Y as 2nd)` 또는 PMI/Jaccard 단일 채택, self-pair 분모 제외 | P7 (1.3~1.5배 오스케일) |
| **단일 랭킹 기준** | conf/lift 혼용 금지 — 증분이면 lift 정렬, conf는 신뢰필터. R3 교정 재적용 | P8·P9 |

### 3.3 모집단·채널
| 요구 | 내용 | 근거 |
|---|---|---|
| **채널 분리 산출** | A(앱) / C(웹) 별도 NBO. **웹 적용엔 C 사용**(데이터 3~7배). 앱↔웹 top-N 교집합만 고신뢰 | P6 |
| **시즌물 별도 트랙** | `temporal`·`open_date`로 신년·연도 운세 분리, 작년→올해 SKU 치환 룰 | P14 (시즌→시즌 lift 9.8 vs 2.1) |

### 3.4 운영·실험
| 요구 | 내용 | 근거 |
|---|---|---|
| **A/B 증분 게이트** | 출시 = 노출군 vs 비노출군 실험. 적중률은 보조지표로 강등 | P15 (holdout 없음) |
| **이탈자 fallback** | 1회성 이탈자(26.5%, 취향신호 0)엔 인기 fallback 우선 분기 | P13 |
| **다양성·탐색** | per-skill 노출캡 + ε-greedy 탐색(롱테일 82% 노출 0 방지) | P16 |
| **갱신주기** | rolling window(최근 6~12개월) 정기 재집계(월 1회, 시즌 전환기 주 1회). 콜드스타트 fallback에도 판매상태 게이트 | P2·P16 |

### 3.5 근본 개선 (별도 트랙)
- **노출 로그 기반 lift** — co-purchase에 노출 분모가 없어 "관심 vs 노출"을 못 가른다. impression 로그로 "노출됐는데 안 산 것" 대비 lift를 재정의해야 진짜 관심 신호. `/dev-data` 신규 작업.

## 4. 액션 기획

### 4.1 무엇을 — 정의
앱에서 **스킬 결제 완료 / 결과 확인 직후 화면**에 "이 운세 본 분들이 다음으로 많이 본 운세" 슬롯(top-3) 노출. 추천 = §3 요구를 반영한 **정제 NBO**. 데이터 부족/신규 스킬은 인기순 fallback.

### 4.2 데이터 파이프라인 — 정제 NBO 산출
1. 소스: `mart_use_skill_se` (앱 구매 시퀀스). 채널 A/C 분리.
2. 정제: §3.1~3.3 전부 적용 (판매상태·blocklist·보유제외·방향·표면분리·표본게이트·lift정정·채널분리).
3. 산출물: `(anchor_menu) → [rec_menu, rank, conf, lift_corrected, pair_n, wilson_lb]` (표면별 2벌: 당일/재방문) + fallback(게이트 통과 인기순).
4. 갱신: rolling window 월 1회(시즌기 주 1회). → `/dev-data`.

### 4.3 서빙
- API: `GET /reco/next?anchor={menu}&surface={inssession|revisit}&exclude={owned}` → 게이트·보유제외·fallback 적용 top-N.
- NBO 테이블 작음 → Redis 캐시 즉답. **서빙 시점 판매상태 실시간 재확인**(배치 stale 대비). → `/dev-server`.

### 4.4 노출 표면 & UX
- 위치: 결제완료 화면 + 스킬결과 하단(6/9 결정). 카드 top-3, 탭 시 결제 플로우.
- 카피: "이 운세 본 분들이 다음으로 많이 본 운세". → `/design` 스펙 + `/dev-ios`·`/dev-android`.

### 4.5 측정 & 실험 설계
- **북극성 KPI**: 첫 구매자 → 2번째 구매 전환율 (당일/재방문 분리).
- 슬롯 지표: 노출수·CTR·슬롯 경유 2번째 전환.
- **A/B**: 노출군 vs 비노출군의 **증분** 2번째 구매율. 분할·기간·최소표본은 6/9 결정.
- 이벤트: 추천 노출(impression)·클릭·전환 — 이벤트 스펙 신설(노출 로그는 §3.5 근본개선의 입력도 됨). → `/dev-data` event-spec.

### 4.6 단계적 롤아웃
| Phase | 내용 | 개발량 | 산출/게이트 |
|---|---|---|---|
| **P0. 데이터 정제** | §3 처방 반영해 NBO 정제본 재생성 + 검증 | 데이터만 | 정제 NBO 2벌 + 검증 리포트 |
| **P1. CRM A/B (Tier 0)** | 정제 NBO top-1을 카플친/Braze로 미재구매자에 푸시, **노출군 vs 대조군 증분 검증** | 앱·서버 0 | 2주 내 증분 1차 확인 |
| **P2. 인앱 슬롯 (Tier 1)** | 결제 직후 슬롯 제품화 + A/B | 서버+앱+디자인 | 슬롯 전환·증분 |
| **P3. 재방문 확장** | 푸시·홈탭(TODO-012)에 `r_menu` 풀 적용 | +웹 | 재방문 전환 |

## 5. 리스크 & 가드레일

| 리스크 | 영향 | 가드레일 |
