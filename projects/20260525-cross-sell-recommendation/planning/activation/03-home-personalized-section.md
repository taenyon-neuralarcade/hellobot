## ① 한 줄 요약 — 무엇을

> 앱 **홈화면에 사용자 구매 이력 기반 개인화 추천 스킬 섹션**(top-N 카드)을 상시 노출한다. 카피 = "○○ 보신 분들이 다음으로 많이 본 운세". 추천 = 정제 NBO 세트(co-purchase), anchor = 사용자 과거 구매 스킬. **단일 북극성 KPI = 첫 구매자 → 2번째 구매 전환율.** 출시 게이트 = 섹션 노출군 vs 비노출(기존 홈) **A/B 증분** — 적중률은 보조지표.

형제 결제직후 인세션 슬롯이 "**막 산 직후**" 표면이라면, 이 홈 섹션은 "**다른 날 다시 와서 둘러보는**" 상시 표면이다. 둘은 서빙 API(`GET /reco/next`)를 공유하되 **표면(surface) 파라미터로 추천 세트를 분리**한다.

## ② 배경 & 문제

| # | 문제 | 근거 |
|---|---|---|
| B1 | **상시 발견 표면 부재** | 현재 앱 홈은 사용자가 "다음에 뭘 볼지" 개인화 추천으로 안내받는 상설 자리가 없다. 1회 구매자가 다른 날 앱을 다시 열어도 다음 구매로 이어줄 표면이 없음 |
| B2 | **1→2 전환이 최대 레버** | 재구매 낭떠러지는 1→2 단 한 곳(전환 31.5%). 넘기면 2→3이 48%. 매출은 단골 2.6%가 26% 책임 → 신규를 단골로 키우는 게 최대 기회 (baseline.md) |
| B3 | **재방문 리텐션은 인세션과 다른 표면** | 1회 구매자의 2번째 결제 중 상당수가 다른 날 재방문 경로. 결제직후(인세션) 슬롯만으로는 이 재방문 경로를 못 잡는다 |
| B4 | **반복 노출 표면의 위생 부담** | 홈은 매 진입 시 반복 노출 → 보유 스킬 추천·판매중지 상품·소표본 노이즈가 인세션보다 신뢰를 더 빠르게 갉아먹는다 |

**가설 (A/B로 검증)**: 홈에 개인화 추천 섹션을 얹으면 재방문 사용자의 홈경유 2번째 구매가 증가하고, 1→2 전환율이 비노출 대비 증분으로 오른다. — **검증된 사실이 아니라 실험 가설.** 적중률(co-purchase backtest 36.6%)은 "이미 살 사람을 맞힌 상관 수치"일 뿐 증분 매출이 아니다.

## ③ 목표 & 성공지표

| 층 | 지표 | 정의 | 역할 |
|---|---|---|---|
| **북극성** | **1→2 전환율** | 첫 구매자의 2번째 구매 발생률 (당일/재방문 분리) | 출시 판정 기준 (A/B 증분) |
| 섹션 | 섹션 노출수 / 도달률 | 홈 진입 중 섹션이 실제 노출된 비율 | 노출이 모든 실험의 출발점 |
| 섹션 | 섹션 CTR | 노출 대비 카드 클릭률 | 추천 적합도 보조 |
| 섹션 | **홈경유 2번째 구매 전환** | 섹션 클릭 → 결제 완료 | 섹션 기여 핵심 |
| 보조(강등) | 추천 적중률 | co-purchase backtest hit-rate | **보조지표로 강등** (상관 수치, ROI 아님) |

### 가드레일 (악화 시 롤백)
- 홈 전체 전환·세션당 구매가 비노출 대비 **하락하지 않을 것** (기존 모듈 잠식 방지)
- 섹션 내 **보유 스킬 추천 = 0** (이미 산 스킬 추천 금지 — 신뢰 직결)
- 섹션 내 **판매중지·비콘텐츠(충전/질문권/하트/패키지) 노출 = 0**
- 반복 노출 피로: 동일 카드 반복 노출률 상한 (다양성캡·ε-greedy로 관리)

## ④ 타겟 & 세그먼트

| 세그먼트 | 정의 | 본 섹션 처리 |
|---|---|---|
| **A. 앱-퍼스트 (모집단)** | `user_first_paid_date == user_first_app_paid_date`. 가장 깨끗한 인앱 cross-sell 타깃 | **1차 노출 모집단** — v2 직접 추천 우선 |
| B. 웹→앱 이동자 | 웹 먼저 → 앱 이동 | 2차 (앱 구매 이력 있으면 동일 로직 적용 가능, A/B는 A 우선) |
| C. 웹온리 | 앱 구매 없음 | 본 섹션 밖 (앱 미진입 → 인앱 추천 레버 밖. 앱전환은 별도 트랙) |
| **thin-signal 분기** | 1구매·추천 신호 약한 사용자 | **fallback 분기** (§5) — 인기순/관심사 백오프 |

**모집단 정합**: 이 섹션은 **앱 구매 이력자**가 대상. anchor 자체가 "과거 구매 스킬"이므로 0구매 신규는 대상 아님(별도 온보딩 영역). 1구매자는 anchor 1개뿐이라 신호가 얇음 → fallback 비중 큼.

## ⑤ 추천 로직 연결 (핵심)

### 5.1 사용하는 NBO 세트 — 인세션 v2 (확정본), 재방문은 오픈

확정된 추천세트는 **인세션 v2**(`nbo_v2_combined.csv`)다. 실측:

| 항목 | 값 (CSV 실측) |
|---|---|
| 앵커(첫 스킬) 수 | **1,045개** |
| 추천 행 | 5,168행 (앵커당 top5) |
| `source=직접` (실제 1→2 co-purchase) | 397행 / **127개 앵커** |
| `source=관심사` (롱테일 백오프) | 4,771행 |
| **사용자 가중 직접 커버리지** | **83.2%** (81,656 / 98,088) |
| 직접 추천 score_pct | median 5.1%, max 50.3% (pair_n 10~523) |
| 관심사 백오프 score_pct | median 3.0% |

→ 사용자의 83%는 실제 함께구매 데이터로 직접 추천, 나머지 17% 롱테일은 관심사(주제+의도) 그룹 백오프. **방법=2층**: 1층 직접(s_menu, 인세션 포함, Wilson, anchor≥50·pair≥10) → 2층 관심사그룹 백오프.

### 5.2 ⚠ 표면 정합 주의 — 홈은 재방문 의미에 더 가깝다

홈은 사용자가 **다른 날 앱을 다시 열어** 둘러보는 상시 표면이므로, 의미상 **결제직후 인세션보다 재방문(r_menu) 에 가깝다.** 그러나 확정된 건 인세션 v2 세트뿐이다(재방문 P0 = `nbo_table_a_revisit_p0.csv` 는 **위생 정제 미반영·74앵커만**). 두 세트는 실제로 추천이 다르다 — CSV 실측 비교:

| anchor | 인세션 v2 (s_menu) rank-1 | 재방문 P0 (r_menu) rank-1 | rank-1 일치? |
|---|---|---|---|
| 400 "다시 만날 수 있을까" | 전연인 어떻게 지낼까 (16.8%) | **재회확률 높이는 법** (8.0%, lift3.91) | ✗ **flip** |
| 42952 "내 사주에 쓰인 운명의 배우자" | 내 사주에 연애 상대 몇 명? (3.5%) | **내 사주의 사랑운** (3.4%, lift2.42) | ✗ **flip** |
| 2141 "그 사람의 속마음" | 그 사람이 연락 안 하는 이유 (12.9%) | 동일 (5.4%, lift2.62) | ✓ |

관찰: ① **인세션 score 가 체계적으로 높다**(같은 앵커 16.8% vs 8.0%) — 같은 세션 노출로 함께 산 인세션 편향. ② 재방문은 **lift 기준 재정렬**로 다른 순위가 뜬다(예: 2141 의 "내가 관심 있는 거 눈치챘을까?" insession rank-5 → revisit rank-4 lift4.69). 형제 task2 검증과 정합: 두 표면 top5 가 **48%만 겹치고 앵커 32%가 1순위가 바뀜** → **단일 엔진 금지.**

**→ 결정 필요 (§12 오픈)**: 홈에 (a) 인세션 v2 그대로 쓰되 표면 의미 차이를 감수, (b) 재방문 P0 를 **v2급으로 정제 후** 병용/대체, (c) v2 를 기본으로 깔고 재방문 정제세트 준비되면 교체. **권장 = (c)** — 확정본(v2)으로 빠르게 출시·A/B 증분 먼저, 재방문 정제세트(P0→v2급 위생 게이트)는 `/dev-data` 후속으로 준비해 홈 표면만 교체. 출시 지연 없이 표면 정합을 점진 개선.

### 5.3 anchor keying 정책 — 핵심 설계 결정

사용자는 보유 스킬이 여럿일 수 있으므로 "어느 anchor 로 추천할지"가 설계의 핵심.

| 옵션 | 정의 | 장점 | 단점 |
|---|---|---|---|
| **O1. 마지막 구매 1개** | 가장 최근 구매 스킬을 anchor | 단순·"방금 관심사" 반영, 인세션 슬롯과 동형 | 1개 의존 → 직접커버 못 받으면 바로 fallback |
| **O2. 최다 관여 1개** | 구매·조회 최다 스킬을 anchor | 안정적 대표 관심사 | 오래된 관심사 고착, 최근성 약함 |
| **O3. 다중 앵커 블렌드** | 보유 스킬 N개 각각의 top5 추천을 점수 병합(보유제외 후 상위) | 신호 최대 활용·직접커버↑·다양성↑ | 구현 복잡·중복 추천 dedup·가중치 설계 필요 |

**권장 = O3(다중 앵커 블렌드) 를 기본, 단 단순 가중**:
- 보유 스킬별 v2 top5 를 모아 `rec_menu` 단위로 score 합산(최근 구매에 가중 ↑), 보유·중복 제외 후 상위 top-N.
- 1구매자(anchor 1개)는 자동으로 O1 과 동일하게 동작 → 별도 분기 불필요.
- **이유**: 직접커버 83% 는 "앵커 1개" 기준 수치인데, 보유 스킬이 여럿이면 그중 하나라도 127개 직접앵커에 들 확률이 올라 **fallback 비중을 낮춘다**(thin-signal 완화). 홈은 반복 노출이라 다양성도 유리.
- **단순 시작 옵션**: 1차 PoC 는 O1(마지막 구매)로 시작해 블렌드는 2차 — 구현 리스크를 낮추고 싶으면. (§12 오픈)

### 5.4 서빙 시점 위생 (홈 특화 — 반복 노출이라 특히 중요)

§3(형제 기획서)의 품질 게이트를 **그대로 상속**하되, 홈은 아래가 특히 critical:

| 게이트 | 내용 | 홈에서 왜 더 중요한가 |
|---|---|---|
| **보유·자기 스킬 제외** | 서빙 시점 실시간으로 유저 보유 스킬 + anchor 자기 제외 | **반복 노출** — 이미 산 스킬을 홈에서 계속 보면 신뢰 즉시 붕괴 |
| 판매상태 하드게이트 | `is_open & !is_stop_selling & buyers_30d>0`, 서빙 시점 재확인 | 매 진입 노출 → 죽은 상품 반복 노출 위험 |
| blocklist | 질문권·충전·하트·패키지·[추가구매]·(추천할인)·프리패스 제외 | 동일 |
| 표본 게이트 | `pair_n≥30·anchor_n≥50` + Wilson 하한 | 소표본 추천을 매번 강신호처럼 노출 방지 |
| **다양성캡 + ε-greedy** | per-skill 노출캡 + 탐색으로 롱테일 노출 | 반복 노출 피로 완화 + 롱테일 82% 노출 0 방지 |

### 5.5 fallback 분기 (thin-signal / 전량 보유)

| 상황 | 분기 |
|---|---|
| 직접커버 있음 (블렌드 후 top-N 충족) | 정제 NBO 직접 추천 |
| thin-signal (직접 부족) | **관심사(주제+의도) 백오프** → 그래도 부족하면 **판매중 인기순** |
| 1구매·신호 약함 | 위와 동일 분기 (anchor 1개 → 직접커버 못 받으면 관심사/인기순) |
| **전량 보유** (추천 후보가 보유로 다 빠짐) | 관심사 백오프 → 인기순으로 채움. 그래도 0이면 **섹션 미노출**(빈 섹션 금지) |

## ⑥ 동작 / UX

```
홈 진입
  → [서버] 유저 보유 스킬 조회 → anchor keying(블렌드/마지막)
  → GET /reco/next?surface=home&anchors={보유}&exclude={owned}
  → [서버] 정제 NBO 룩업 + 표면세트 + 서빙위생(보유/판매상태/blocklist/표본) + 다양성·탐색 + fallback
  → top-N 카드 반환
  → [앱] 홈 섹션 렌더 + impression 이벤트
  → 카드 탭 → 스킬 상세/결제 플로우 + click 이벤트
```

- **섹션 헤더 카피**: "○○ 보신 분들이 다음으로 많이 본 운세" (anchor 스킬명 치환). 블렌드 시 = "회원님께 어울리는 다음 운세" 등 anchor 비노출 카피 (오픈 — §12).
- **카드**: 스킬 썸네일 + 제목 + (옵션) 짧은 사회적 증거. top-N = 3~6 (오픈 — §12).
- **빈 섹션 금지**: fallback 으로도 카드가 안 차면 섹션 자체를 숨김.

> 노출 이벤트는 §3.5(형제) **노출로그 기반 근본개선**의 입력도 된다 — 홈 섹션 노출 로그가 쌓이면 "노출됐는데 안 산 것" 대비 lift 정의에 기여.

## ⑧ 측정 & 실험

- **실험 = 섹션 노출군 vs 비노출(기존 홈) A/B.** 측정 = 1→2 전환·홈경유 2번째 구매의 **증분**.
- **적중률 강등**: co-purchase backtest 적중률은 보조지표. "추천 섹션 = 매출 증가"로 단정 금지 — A/B 로만 증분 확정.
- 분할·기간·최소표본: 형제 기획서 실험 프레임 상속(재방문 +3pp 검정에 ~4,174/팔, 풀링+CUPED+순차검정 6~10주 — baseline redesign 근거). 구체 수치는 출시 결정 시 확정.
- **롤아웃 원칙**: 가장 싼 채널로 증분 먼저 → 검증되면 제품화. 홈 섹션은 인앱 개발이 드는 표면이므로 **형제 P1(CRM A/B)·P2(인앱 인세션 슬롯)로 co-purchase 증분이 1차 검증된 뒤** 본격화(§10).
