**작성**: 2026-06-09 (코디네이터/PO) · **상태**: 6/9 데이터 미팅 리뷰용 기획안 (확정 추천세트 기반 · 효과는 A/B 미검증)
**SSOT 참조**: 분석=[`../baseline.md`](../baseline.md) (REVISION 3 · C 섹션) · 형제(결제직후 인세션 슬롯)=[`../action-plan-nbo-slot.md`](../action-plan-nbo-slot.md) · 추천세트=[`../../data/nbo_web12_named.csv`](../../data/nbo_web12_named.csv) · [`../../data/web_hero_c_nbo_p0.csv`](../../data/web_hero_c_nbo_p0.csv)

## 1. 한 줄 요약 — 무엇을

웹(hellobot-web/webview) 신규 첫 구매자가 결과/결제 직후 보는 **아웃트로 화면**에 추천 스킬을 노출해 두 길로 다음 행동을 유도한다:

| 메커니즘 | 무엇 | 노리는 것 | 정합 표면 |
|---|---|---|---|
| **(a) 웹 인세션 추가구매** | 아웃트로에 cross-sell 카드 → 웹에서 바로 2번째 구매 | 즉시 매출 (당일 cross-sell) | 결제직후·당일 |
| **(b) 앱전용 할인쿠폰 웹→앱 유도** | 추천 스킬을 "앱에서 더 싸게" 쿠폰으로 제공 → 앱 설치·전환 후 redemption | 리텐션·LTV·채널이전 | 재방문·채널이동 |

> **핵심 전략 가설**: C(웹온리) 분석 결론은 "웹온리의 본질 = 리텐션·앱전환 문제, 인앱 추천 레버 밖 → 앱전환·재방문 유도가 핵심". 따라서 **(b) 앱전용 쿠폰 웹투앱이 이 문제를 정면 공략하는 강수**이고, (a)는 즉시 매출 보조다. 두 메커니즘의 관계는 §6에서 명시 설계.

## 2. 배경 & 문제

- 매출의 **59%가 웹**, 그 웹의 **88%가 사주** — 광고로 유입돼 사주 한 편 사고 이탈하는 구조 (북극성 진단).
- **C(웹온리) = 289,983명**(A 앱퍼스트의 3배), rev90 재구매 **13.1%**(A의 1/3), **75.5%가 1회성**. 웹 1회 구매 후 절대다수가 다시 안 온다.
- 그런데 웹 사용자에게 **앱 행동 데이터(A세트)를 그대로 추천에 쓰면 모집단 불일치**가 크다 → 웹엔 웹 모집단(C) 또는 "앱 행동으로 관심사를 추정한" 정합 데이터가 필요.
- **1→2 전환이 최대 레버**: 1→2 전환율 31.5%가 유일한 낭떠러지, 넘기면 2→3은 48%. 매출은 단골 2.6%가 26% 책임. 웹 1회성 이탈을 1→2로 넘기는 것이 회사 최대 기회.
- **앱전환 갭**: B(웹→앱 이동자) 분석은 "채널이동 자체가 강한 리텐션 신호"라고 본다. 즉 웹 구매자를 앱으로 넘기는 것은 단순 채널 이동이 아니라 **리텐션 전환의 프록시**다. → 아웃트로는 이 전환을 만들 수 있는 가장 이른 접점.

**문제 정의**: 웹 신규 첫 구매자는 (1) 한 번 사고 즉시 이탈하고(즉시 매출 누수), (2) 앱으로 넘어오지 않아 리텐션 모수에 들어오지 못한다. 아웃트로 한 화면에서 (a)즉시 cross-sell과 (b)앱전환의 두 갈래를 동시에 시도한다.

## 3. 목표 & 성공지표

**북극성 KPI (회사 공통)**: 첫 구매자 → 2번째 구매 전환율 (1→2). 본 아이템은 **웹 신규 첫구매자**의 1→2를 두 경로로 끌어올린다.

| 구분 | 지표 | 정의 | 판정 |
|---|---|---|---|
| 북극성 | 웹 신규 첫구매자 1→2 전환율 | (a)+(b) 합산, 첫 구매 후 N일(30/60/90) 내 2번째 구매 | **A/B 증분으로만 확정** |
| (a) 메커니즘 | 웹 인세션 2번째 구매율 | 아웃트로 노출자 중 같은 세션/당일 2번째 구매율 | 노출군 vs 비노출(holdout) 증분 |
| (b) 메커니즘 | **웹→앱 전환율** | 아웃트로 쿠폰 발급자 중 앱 설치·로그인 도달율 | 쿠폰군 vs 비쿠폰군 |
| (b) 메커니즘 | **앱 내 2번째 구매·재방문** | 쿠폰 redemption + 앱 내 2번째 구매율 + 익일+ 재방문율 | 쿠폰군 vs 비쿠폰군 증분 |
| **가드레일** | **쿠폰 순효과(마진 차감)** | (쿠폰군 증분 매출) − (할인 원가·발급비용) > 0 | **순증분이 음수면 중단** |
| 보조(강등) | 아웃트로 CTR·적중률 | 추천 클릭률·co-purchase 적중률 | 보조지표 — ROI로 오해 금지 |

> **효과 caveat (필수)**: 추천 적중률은 "이미 살 사람을 맞힌 상관 수치"일 뿐 증분 매출이 아니다. (a)·(b) 모두 **노출/쿠폰군 vs 비노출/비쿠폰군 A/B 증분 게이트** 통과가 출시 조건. "쿠폰 주면 산다"가 아니라 "A/B로 검증할 가설"로 다룬다.

## 4. 타겟 & 세그먼트

| 세그먼트 | 정의 | 본 아이템에서 역할 |
|---|---|---|
| **웹 신규 첫구매자** | 웹에서 진짜 첫 결제 (`user_first_paid_date`), 앱 구매 이력 없음 | **주 타겟** — 아웃트로 노출 대상 |
| **C (웹온리)** | 앱 구매 없음 (289,983명) | **모집단 정합 기준** — 추천 소스·기대 전환율의 베이스라인. 본질=리텐션·앱전환 문제 → (b) 1차 대상 |
| **B (웹→앱 이동자)** | `user_first_paid_date < user_first_app_paid_date` | **신호 기준** — 채널이동=강한 리텐션 신호. (b)가 성공하면 신규 첫구매자를 B 경로로 이동시키는 셈 |
| (A 앱퍼스트) | 앱 첫구매자 | **직접 적용 금지** — 모집단 불일치. 앱 행동은 "관심사 추정"의 재료로만 사용(web12 방식), A 추천세트를 웹에 직접 노출하지 않음 |

**모집단 정합 원칙**: 웹 표면에는 ① 인기 12스킬엔 `nbo_web12_named.csv`(앱 행동으로 관심사 추정), ② 더 넓은 커버리지엔 **C 모집단 NBO**(`web_hero_c_nbo_p0.csv`, 정제 선결)를 쓴다. **앱 A세트(`nbo_v2_combined.csv`)는 웹에 직접 적용하지 않는다.**

## 5. 추천 로직 연결 (확정 데이터 → 서빙)

### 5.1 두 소스의 역할 분담

| 소스 | 파일 | 앵커 수 | 특성 | 본 아이템 용도 |
|---|---|---|---|---|
| **web12 (인기 12)** | `nbo_web12_named.csv` | 12 | 앱 행동으로 관심사 추정 · **위생 적용됨** · Wilson 정렬 | **1차 서빙** — 웹 인기 앵커 |
| **C P0 (웹온리 모집단)** | `web_hero_c_nbo_p0.csv` | 8(현재) | C 모집단 직접 co-purchase · anchor_n 2~3배 큼 · **위생 미적용(정제 선결)** | **백오프 후보** — 모집단 정합 검증·롱테일 확장의 씨앗 |

> ⚠ **현재 C P0의 한계 (정직하게)**: P0 파일은 아직 **8 앵커**뿐이고, 그 8개가 전부 web12의 직접 앵커와 겹친다(43117·49816·55723·57670·21964·27376·47704·2141). 즉 **지금 C P0는 web12보다 넓지 않다**. C의 "더 넓은 커버리지"는 **잠재력**(전체 C 웹온리 모집단으로 롱테일 앵커까지 확장 가능)이지 이 P0에서 실현된 것이 아니다. → **C 정제 NBO 풀빌드가 (b) 롱테일·web12 밖 커버리지의 선결 과제**(/dev-data).

### 5.2 web12 세트 — 직접 9 + 토픽 백오프 3 (CSV 실측)

컬럼: `anchor,anchor_name,anchor_topic,anchor_n_app,rank,rec_menu,rec_name,source,score_pct,lift,co_n`

- **직접 9개** (`source=직접`, 앱 표본 205~6,046, lift·co_n 보유) — 관심사 클러스터가 선명:
  - `2141 그 사람의 속마음`(n=6,046) → rank1 `17212 그 사람이 연락을 안 하는 이유` (score 9.2%, lift 2.92, co_n 555) → 감정 흐름 여정.
  - `49816 재회 사주`(n=908) → rank1 `28894 재회 사주: 다시 만날 운명일까?` (10.0%, lift 2.09) → 재회→재회 클러스터.
  - `27376 결혼OX궁합`(n=1,360) → rank1 `2166 그 사람과 나의 사주 궁합` (7.8%, lift 2.36) → 궁합→궁합.
- **토픽 백오프 3개** (`source=관심사`, `anchor_n_app` 극소 = 순수 웹획득형, lift 공란):
  - `60706 결혼OX진단`(앱 n=**6**), `60013 내운트이는동네`(n=**8**), `60244 연애상대스포`(n=**4**). 앱 organic 신호 부재 → 주제(결혼/일반운세/연애)로 백오프해 추천 생성.

> web12는 첫 구매(아웃트로 진입) 앵커용으로 **바로 서빙 가능**. 단 `wilson_lb` 컬럼이 세트에 노출돼 있지 않으니 서빙 전 표본·신뢰 메타를 확정(§9 /dev-data).

### 5.3 C P0 세트 — 모집단 정합 강함, 위생 미적용 (CSV 실측)

컬럼: `anchor,anchor_name,anchor_n,rank,rec_menu,rec_name,rec_cat,pair_n,conf_pct,lift,wilson_lb_pct`

- **장점**: anchor_n이 web12 앱프록시보다 크다 — 예 `43117`(C anchor_n 2,431 vs web12 앱프록시 2,089), `47704`(C 3,940). `wilson_lb_pct`·`pair_n`·`rec_cat`까지 보유해 표본 게이트·신뢰필터에 바로 쓸 형태.
  - 예: `43117 사랑운` → rank1 `47704 삼신할매 인연` (pair_n 215, conf 8.8%, lift 1.91, wilson_lb 7.8).
- **위생 미적용 리스크(P0)**: `is_open`/`is_stop_selling`/`buyers_30d` 게이트 미적용. 예 — `57670 2026 병오년 운세`가 rec로 노출(시즌물 → 판매중지 위험), 고-lift 일부(`56648` lift 23.13, `35527` lift 17.55)는 small pair_n에서 나온 인플레 가능 → **풀빌드 시 위생 게이트·표본 게이트·lift 분모 정정 필수**.

### 5.4 앵커 키잉 & 서빙 위생 (형제 §3 처방 동일 적용)

| 항목 | 규칙 | 근거 |
|---|---|---|
| **앵커 키잉** | 웹 아웃트로 진입 스킬(`menu_seq`)을 앵커로 web12 우선 매칭 → 미스 시 C(정제본) → 둘 다 미스 시 토픽 백오프 → 최후 인기 fallback | web12 12 + C 정제본 |
| **위생 게이트** | 판매중·유료·콘텐츠만(`is_open & !is_stop_selling & buyers_30d>0`), blocklist(질문권·충전·하트·패키지·[추가구매]·(추천할인)·프리패스) 제외 | 형제 §3.1 |
| **자기·보유 제외** | 앵커 자기추천 제외 + 유저 보유 스킬 제외(**서빙 시점**) | 형제 §3.1 |
| **표본 게이트** | `pair_n≥30 · anchor_n≥50` + Wilson 하한 정렬, rank1-rank2 CI 분리 시만 "추천" | 형제 §3.2 |
| **방향 보존** | 정방향만(앵커→다음). 양방향 pooling 폐기 | 형제 §3.2 |
| **앱 A세트 금지** | 웹 표면엔 web12/C만. `nbo_v2_combined.csv`(앱) 직접 서빙 금지 | 모집단 정합 |

> web12는 위생 적용됨. **C P0는 미적용** → (b) 롱테일 서빙 전 정제 풀빌드가 선결.

## 6. 동작 / UX

### 6.1 두 메커니즘의 관계 (명시 설계 — 권장안)

오픈 결정이지만 **권장 = 순차 분기**:

```
[웹 아웃트로 트리거: 결과 확인 / 결제 완료 직후]
        │
        ▼
(a) 인세션 추가구매 카드 (top-3, web12/C 정제 추천)
        │
   ├─ 클릭·구매 ─────────────▶ 웹 즉시 2번째 결제 (즉시 매출)   ← 측정: 웹 인세션 2구매율
        │
   └─ 미전환(이탈 시그널/스크롤 이탈/N초) 또는 명시 CTA
        │
        ▼
(b) "앱에서 이 운세 OO% 할인" 앱전용 쿠폰 제안
        │
   ├─ 앱 설치자 ─▶ 딥링크 → 앱 쿠폰 랜딩 → redemption → 앱 내 구매   ← 측정: 웹→앱 전환·앱 2구매
        │
   └─ 앱 미설치자 ─▶ (오픈 결정: 설치 유도 vs 웹 즉시구매 폴백)
```

대안 옵션(오픈 결정):
- **세그먼트 분기**: 즉시구매 의향 높은 앵커(고-conf 인기 연애)는 (a) 우선, 1회성 이탈 위험 큰 앵커/시즌물은 (b) 우선.
- **병행 노출**: (a)카드 + (b)쿠폰 배지 동시 — 단 선택 피로·측정 혼선 위험.
- **순수 A/B**: (a)군 vs (b)군 vs holdout 3-arm으로 어느 경로가 1→2를 더 키우는지 직접 비교(가장 깨끗하나 표본 소요 큼).

| 요소 | 내용 |
|---|---|

## 8. 측정 & 실험

- **설계**: 아웃트로 노출/쿠폰을 **홀드아웃(비노출·비쿠폰)과 분리한 A/B**. 권장 3-arm = ①holdout ②(a)인세션만 ③(a)→(b)순차. 단순화 시 2-arm(노출 vs holdout)부터.
- **판정**: 
  - (a) = 노출군 vs holdout의 **웹 인세션 2구매 증분**.
  - (b) = 쿠폰군 vs 비쿠폰군의 **웹→앱 전환 + 앱 2구매·재방문 증분**, 여기서 **쿠폰 마진(할인 원가+발급비)을 차감한 순증분**으로 GO/NO-GO.
- **적중률 강등**: co-purchase 적중률·CTR은 보조지표. ROI 근거로 쓰지 않는다.
- **검정력**: 형제 문서 기준(재방문 +3pp ≈ 4,174/팔, 풀링+CUPED+순차검정 6~10주) 준용. C 모수가 크므로 웹 표면 표본 확보는 상대적으로 유리.
- **롤아웃 원칙**: 가장 싼 채널로 증분 먼저 → 검증되면 제품화. (b) 쿠폰은 마진을 직접 태우므로 **소규모 A/B로 순효과 부호부터** 확인 후 확대.

## 12. 오픈 결정사항 (6/9 미팅)
