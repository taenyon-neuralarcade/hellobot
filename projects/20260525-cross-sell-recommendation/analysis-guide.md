# 분석 프로세스 가이드 — 세그먼트 기반 Cross-Sell 추천

> **목적**: 1회 구매자 → 두 번째 구매 전환을 위한 추천 SKU 도출의 표준 분석 방법론.
> **범위**: 본 프로젝트 (cross-sell-recommendation) 의 분석 절차 정의 + 향후 유사 cross-sell·추천 분석 시 재사용 가능한 가이드.

---

## 0. 분석 의도 정리

이 분석이 풀려는 질문:

> **"어떤 사용자가 첫 결제로 A 를 샀을 때, 어떤 두 번째 SKU 를 추천해야 재구매로 이어지는가?"**

이를 풀기 위한 표준 분석 프레임:

```
Segmentation → Profiling → Next-Best-Offer (NBO)
```

세그먼트로 사용자를 나누고, 각 세그먼트가 "무엇이 다른가" 를 정량화한 뒤, 그 차이를 근거로 "다음에 무엇을 추천할까" 를 도출.

---

## 1. 문제 정의 (Framing) — 분석 착수 전 4개 결정

분석에 들어가기 전 반드시 픽스해야 하는 항목. 이게 빠지면 그 다음 모든 분석이 흔들림.

| 항목 | 결정 내용 (본 프로젝트) |
|---|---|
| **KPI** | 첫 결제 후 30/60/90 일 이내 두 번째 결제 발생률 (재구매율). **신규/기존 결제자 분리** |
| **분석 단위** | 사용자 1명 = 1행. `first_paid_date` (사용자 최초 결제일) 를 t=0 으로 정규화 |
| **관찰 윈도우** | `first_paid_date` 기준, t+90 일까지 행동 관찰 |
| **타겟 변수** | y = (t+N 일 이내 두 번째 결제 = 1, 아니면 0). 단, **2회차 타이밍을 2분기로 분리** (1.3 참조) |

> 데이터 분석가가 가장 많이 시간을 쓰는 부분. 후속 단계에서 "이게 뭘 의미하는 지표지?" 라는 질문이 반복되면 1단계가 부실한 것.

### 1.1 첫구매 채널 분리 (최상위 분리축)

웹/앱은 유입 동기·사용 방식이 다름. 특히 **웹 마케팅 유입 첫구매자는 1회성 구매 후 이탈 성향이 강함** — 외부 광고가 트리거이기 때문. 따라서 채널을 합쳐 보면 두 가지 문제 발생:

1. **오분류** — 웹에서 먼저 산 사람이 앱에서 처음 사면, 앱 기준으로는 "첫구매자"로 잡히지만 실제론 경험자 (재참여)
2. **평균 무의미** — 이탈 성향 다른 두 모집단의 평균 재구매율은 어느 쪽도 대표하지 못함

두 필드로 3그룹 판별:

| 필드 | 의미 |
|---|---|
| `user_first_paid_date` | 웹/앱 통합 — 진짜 첫 구매일 |
| `user_first_app_paid_date` | 앱 한정 — 앱에서의 첫 구매일 (웹 구매 제외) |

| 그룹 | 판별식 | 정체 | "두 번째 구매" 의미 | 추천 레버 | 스코프 |
|---|---|---|---|---|---|
| **A. 앱-퍼스트** | `user_first_paid_date == user_first_app_paid_date` | 진짜 앱 첫 구매자 | 순수 인앱 재구매 | 홈탭·푸시·결제직후 추천 | **메인** |
| **B. 웹→앱 이동자** | `user_first_paid_date < user_first_app_paid_date` | 웹 먼저 → 앱 이동 | 앱 첫 구매가 사실 전체 2번째 = 채널이동 성공 | 무엇이 앱 이동을 유발했나 (역이용) | **격리 분석** |
| **C. 웹-온리** | 앱 구매 없음 (`user_first_app_paid_date` null) | 웹 마케팅 전환 모집단 | 웹 재전환 | 마케팅 스킬 선정 | **후속 트랙 (제외)** |

> C(웹온리)는 "마케팅 콘텐츠로 어떤 스킬을 내보낼까" 라는 별도 문제 (마케팅 스킬 선정 = 추천의 일종). 본 분석(인앱 cross-sell)과 레버가 달라 후속 트랙으로 분리.

**실행 원칙**: Phase 1 맨 앞에서 **3그룹 각각의 인원수부터 쿼리** (거의 공짜). 크기 분포를 보고 A 집중 정도·B 분석 비중을 확정 — 추측으로 미리 못 박지 않음.

### 1.2 첫 결제 시점 — `user_first_paid_date` 사용

데이터에 사용자별 최초 결제일 필드 `user_first_paid_date` (통합) / `user_first_app_paid_date` (앱 한정) 가 존재. 이를 t=0 으로 사용하면 "분석 기간 내 첫 결제" 를 따로 계산할 필요 없이 **사용자의 진짜 첫 결제** 기준으로 코호트를 정렬할 수 있음. 그룹 A/B 에 따라 t=0 기준 필드가 달라짐 (A = 앱=통합이므로 동일, B = 앱 진입 시점 분석 시 `user_first_app_paid_date`).

### 1.3 신규 vs 기존 결제자 분리

재구매(리텐션) 동학은 **신규 결제자와 기존 결제자가 다름**. 두 그룹을 합치면 평균이 왜곡되므로 분리 측정:

| 구분 | 정의 (Phase 1 에서 데이터로 컷 확정) | 왜 분리하나 |
|---|---|---|
| **신규 결제자** | `user_first_app_paid_date` 가 분석 윈도우 내 (최근 유입) | 허니문 버스트·온보딩 효과. 추천 개입 여지 큼 |
| **기존 결제자** | `user_first_app_paid_date` 가 윈도우 이전 (기존 베이스) | 재활성화 동학. 푸시·리인게이지 중심 |

### 1.4 2회차 구매 — 동일일 vs 재방문 분리

"두 번째 구매" 는 카테고리 무관 **전체 SKU** 로 보되, **타이밍을 2가지로 분리** (현행 운영은 일단위 구분):

| 유형 | 정의 | 비즈니스 의미 | 주 적용 위치 |
|---|---|---|---|
| **동일일 2회차** | 첫 결제와 **같은 날** 추가 구매 | 인세션 업셀·장바구니 확장 | 결제 직후 추천, 번들 |
| **재방문 2회차** | 첫 결제 **익일(+1d) 이후** 재방문하여 구매 | 리텐션 재구매 — 본 분석의 핵심 | 푸시, 홈탭 리인게이지 |

→ 두 유형은 **다른 액션으로 연결**되므로 베이스라인·패턴·추천 매트릭스를 각각 산출.

## 2. EDA + Baseline (베이스라이닝)

세그먼트로 들어가기 **전에** 전체 평균을 먼저 본다. 모든 세그먼트별 결과는 이 베이스라인 대비 lift 로 해석함.

### 점검 항목

- 전체 1회 구매자 수, 두 번째 구매자 수 → **재구매율 기준선** (예: "전체 18%")
- 첫 결제 → 두 번째 결제까지 **시간 분포** (median, p25/p75) — H1 (시간 임계점) 의 1차 답
- 첫 결제 **카테고리별 raw 재구매율**
- 결측치·이상치 점검 — 생년월일·성별·관심 토픽 채움률, 이상 가격 결제

### 산출

이 단계에서 이미 "어디에 보석이 묻혀 있을지" 감이 옴. 다음 단계 세그먼트 우선순위의 근거가 됨.

## 3. Segmentation (세그먼트 분류)

### 두 가지 접근 — 병행이 표준

| 방식 | 정의 | 장단점 |
|---|---|---|
| **A. 규칙 기반 (a priori)** | 도메인 지식으로 미리 기준 정의 — 연령대 × 첫 결제 카테고리 × 가격대 | 해석·실행 쉬움, 마케팅·기획과 소통 잘 됨 |
| **B. 데이터 기반 (clustering)** | K-means, 계층적 군집, GMM 등으로 행동 패턴이 비슷한 사용자를 묶음 | 숨겨진 패턴 발견 가능, 해석 어려움 |

### 권장 순서: A → B

A 가 먼저인 이유:
1. 결과를 비즈니스 액션으로 연결하기 쉬움
2. B 의 클러스터를 사후 해석할 때 A 의 축이 라벨링에 쓰임
3. A 만으로도 충분히 의미 있는 결과가 나오는 경우가 많음 (특히 cross-sell)

### 헬로우봇에 흔히 쓰는 분류 축

| 축 | 헬로우봇에서의 형태 | 비고 |
|---|---|---|
| **인구통계** | 연령(5세 버킷), 성별 | `union_mart_user_key_actions.age_group_5yr` 컬럼 이미 있음 |
| **RFM** | Recency, Frequency, Monetary | 가장 검증된 프레임 |
| **첫 결제 특성** | 카테고리(연애/총운/결혼 등), 가격대(저/중/고), 단발 vs 패키지 | 본 분석의 **핵심 축** |
| **콘텐츠 소비 행태** | 이용 토픽 분포, 진입 빈도, 무료 vs 결제 비율 | 행동 시그널 |
| **라이프사이클 단계** | 신규(첫 결제 직후), 활성, 휴면, 이탈 가능성 | 시간축 |
| **자가 신고 시그널** | 관심 토픽, 생년월일 등 입력값 | 채움률 갭 주의 |

### 본 프로젝트의 세그먼트 정의 전략

**데이터 기반 의미 축 탐색을 우선**한다. S1~S6 라벨을 기계적으로 붙이는 게 아니라, **S1~S6 를 구성하는 차원들을 후보 축으로 놓고** 각 축이 실제로 재구매율 차이를 만드는지 검증한 뒤, 의미있는 축만으로 세그먼트를 구성한다.

후보 축 (S1~S6 구성 차원 + 결제 특성):

| 후보 축 | 출처 | 검증 방법 |
|---|---|---|
| 연령 5세 버킷 | `age_group_5yr` | 버킷별 재구매율 차이 (chi-square + effect size) |
| 성별 | 사용자 속성 | 성별 × 재구매율 |
| 첫 결제 토픽/카테고리 | 결제 SKU 메타 | 카테고리별 재구매율·transition |
| 첫 결제 가격대 (저/중/고) | 결제 금액 | 가격대별 재구매율 |
| 페르소나 (자가소비/선물 등) | S1~S6 라벨 룰 | 페르소나별 패턴 |

- **1차 — 단독 스크리닝**: 각 축 단독으로 재구매율 차이가 유의미한지 (effect size 포함) 확인
- **2차 — 조합**: 유의미한 축 2~3개를 곱해 세그먼트 구성 → S1~S6 와 유사하게 수렴할 수도, 다른 조합이 더 강할 수도
- **재활용**: S1~S6 라벨링 룰·기존 분포 (skills-by-segment-12m, segment-top3) 는 후보 축·비교 기준선으로 사용

> S1~S6 의 세부 구분 기준(연령 티어·페르소나·가격·토픽)이 초기 의미있는 기준이 될 가능성이 높음 (사용자 가설). 단, 이를 **검증 후 채택**하는 것이 데이터 기반 접근.

## 4. Profiling (세그먼트 프로파일링)

각 세그먼트가 "**무엇이 다른가**" 를 표로 정리. **이게 분석의 진짜 산출물**.

### 프로파일 표 예시

| 세그먼트 | 인원수 (%) | 평균 첫 결제가 | 재구매율 (30d/60d/90d) | 두 번째까지 median 일수 | TOP3 첫 카테고리 | TOP3 두 번째 카테고리 | 특이점 |
|---|---|---|---|---|---|---|---|
| S3 × 연애 × 저가 | 12,000 (28%) | ₩1,500 | 32% / 41% / 47% | 11일 | 연애·연애·연애 | 연애·총운 | 동일 카테고리 강함 |
| S4 × 결혼 × 고가 | 1,800 (4%) | ₩21,000 | 8% / 12% / 14% | 47일 | 결혼·연애·재물 | 결혼·재물 | 전환 느리지만 단가 큼 |
| ... | | | | | | | |

### 통계 검정 — 우연 vs 시그널 구분

- **카테고리형 변수** (세그먼트 × 재구매 여부): chi-square test
- **연속형 변수** (세그먼트 × 두 번째까지 일수): ANOVA / Kruskal-Wallis
- p-value 만 보지 말고 **effect size** (Cramer's V, eta-squared) 도 같이 확인

### 우선순위 셀 추출

전체 평균 재구매율 대비 ±10%p 이상 벗어난 셀을 우선 공략. 이게 "특성이 명확하게 드러나는 세그먼트".

| 우선순위 분류 | 기준 | 액션 방향 |
|---|---|---|
| **High Performer** | 재구매율 > 평균 + 10%p | 무엇이 잘 되고 있는지 학습 → 다른 셀에 적용 가능한가 |
| **Underperformer** | 재구매율 < 평균 - 10%p | 왜 안 되는지 진단 → 추천 SKU 가 안 맞는 것인가, 노출이 부족한 것인가 |
| **Volume Driver** | 인원수 비중 상위 | 작은 lift 도 절대 매출 영향 큼 |

## 5. Affinity / Next-Best-Offer (추천 콘텐츠 도출)

여기가 사용자 의도의 핵심. 표준 방법론 3개를 조합.

### 5.1. 어소시에이션 룰 (Association Rules / 장바구니 분석)

> "첫 결제로 A 를 산 사용자는 두 번째로 B 를 살 확률이 평균 대비 X배"

| 지표 | 정의 | 해석 |
|---|---|---|
| **Support** | A→B 페어가 전체 사용자 중 차지하는 비중 | 너무 작으면 통계적 의미 약함 |
| **Confidence** | A 산 사람 중 B 도 산 비율 | "추천 정확도" 와 유사 |
| **Lift** | Confidence ÷ B 의 전체 구매율 | **1보다 크면 의미 있는 연관** |

세그먼트 안에서 lift 가 큰 A→B 페어를 찾으면, 그게 곧 "그 세그먼트의 사용자가 A 를 샀을 때 추천할 B".

### 5.2. 시퀀스 분석 (Sequence Mining)

> "첫 결제 후 N일차에 어떤 카테고리가 많이 팔리는가"

- 시간 순서를 보존한 분석 — 어소시에이션보다 한 단계 깊음
- 첫 결제 → 7일차 푸시 / 30일차 푸시 등 **타이밍 의사결정**에 직접 연결
- 본 프로젝트에서는 30/60/90 일 윈도우별로 측정 → 푸시 타이밍 가설로 연결

### 5.3. 협업 필터링 (Collaborative Filtering) — 간이 버전

> "당신과 비슷한 사람들이 그 다음에 산 것"

- 본격 ML 모델까지 안 가도, **세그먼트 안에서 TOP K 구매 SKU** 만 뽑아도 같은 효과
- coop-integration 의 `segment-top3.md` 가 사실상 이 접근 — 같은 방법론을 1회 구매자 코호트에 적용

### 본 프로젝트 실용안

**5.1 + 5.3 결합**. 즉:

```
(Seg × 첫_결제_SKU) → 두 번째 결제 TOP K SKU + lift
```

위 표를 우선순위 셀에 대해 만들면, 그 자체가 추천 매트릭스가 된다.

## 6. Validation (검증)

도출된 추천이 진짜 효과가 있는지 검증.

| 방법 | 내용 |
|---|---|
| **Holdout 비교** | 데이터를 시간으로 분할 (예: 11M 학습 / 1M 검증). 학습 기간에서 도출한 추천이 검증 기간에서도 lift 가 유지되는지 |
| **A/B 테스트 설계 (사후)** | 분석 단계에서는 가설 도출까지. 실제 효과 검증은 운영 A/B — "추천 적용군 vs 미적용군 재구매율 비교" 설계까지 분석가가 제안 |

> 분석 단계에서 "이 추천이 실제로 먹힐 것"이라고 100% 단정할 수 없음. 분석 산출물은 **가설 + 우선순위** 까지, 효과 측정은 운영에서 A/B.

### 주의 사항

- **APP 채널 한정** 분석 (coop-integration 분석과 동일 범위). 웹/카카오톡 채널은 별도 후속
- **결제 이력 기준** — 무료 진입만 한 사용자는 본 분석 범위 밖
- **첫 결제 시점 정의** — 회원가입 후 첫 결제 vs 분석 기간 내 첫 결제. 기간 내 첫 결제로 통일 (12M 윈도우 일관성)
- **세그먼트 라벨 부여 시점** — 첫 결제 시점의 인구통계 vs 분석 시점. 첫 결제 시점으로 통일 (코호트 정합성)

### 데이터 권한·접근

- BigQuery 권한 — `/dev-data` 위임 시 사용자 권한 확인 필요
- 마트 컬럼 충분 — 신규 컬럼 추가 작업 없이 진행 가능
- 신규 이벤트 정의 불필요 — 결제·이벤트는 기존 카탈로그 사용
