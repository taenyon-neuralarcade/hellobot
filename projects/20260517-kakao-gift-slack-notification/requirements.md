# 요구사항 정의서 — 카카오 선물하기 운영 슬랙 알림

> 본 문서는 F1·F2 두 기능의 상세 요구사항을 정의합니다. 기술 설계(`architecture.md`) 와 분리하여, "무엇을" 만들지에 집중합니다.

## 0. 용어 정의

| 용어 | 정의 |
|------|------|
| **카카오 쿠폰** | 카카오 선물하기 채널에서 발급된 쿠폰. `coupon_prefix_rule` 의 `coupon_type = 'coop_marketing'` 매칭 (prefix `90`/`91` 등 — [검증 필요]) |
| **등록(=사용)** | 사용자가 헬로우봇 앱·웹에서 쿠폰 코드를 입력해 사용 처리되는 행위. `coop_marketing_coupon_usage.used_at` 기록 시점 |
| **카카오 채널 신규 구매자** | 첫 카카오 쿠폰 등록일 = 사용자 계정 생성일 인 사용자. [coop-integration data-measurement-plan §2.1](../20260324-coop-integration/data-measurement-plan.md) 정의 재사용 |
| **출시일** | 카카오 선물하기 1차 출시일. 현재 일정 2026-05-22 |

## 1. F1 — 쿠폰 등록 실시간 알림

### 1.1 발화 조건

| 항목 | 요구사항 |
|------|---------|
| 발화 시점 | `POST /api/coupon/register` 가 성공 응답을 반환하는 시점 (서버 사이드 hook) |
| 발화 대상 | `coupon_prefix_rule.coupon_type = 'coop_marketing'` 인 쿠폰만 |
| 미발화 케이스 | 일반 쿠폰·giftiel·테스트 사용자·등록 실패 응답 |
| 토글 상태 | ON 일 때만 발송. OFF 면 알림 skip (서버 로그는 남김) |

### 1.2 메시지 포맷

**필수 항목**:

- 시각: `YYYY-MM-DD HH:mm:ss KST`
- 상품명: `coop_marketing_coupon_product.name` (예: "하트 1만원 충전권")
- 카테고리: `하트 충전권` 또는 `스킬 교환권`
- 금액: KRW (정가 기준, 할인가 별도)
- 사용자 식별: `user_id` 마지막 4자리만 노출 (예: `user_id=****1234`) — 개인정보 보호
- 그날 누적 등록 건수: 등록 시점의 KST 자정 이후 카카오 쿠폰 등록 건수 (예: `오늘 23건째`)

**선택 항목 (1차 제외, 추후 검토)**:

- 발급 채널: 카카오/B2C/B2B (`coop_marketing_coupon.issued_channel`)
- 환불 가능 기간: `coupon.expired_at`
- 쿠폰 코드: 디버깅 용도 (보안 검토 후 결정)

**메시지 예시**:

```
🎁 [카카오 선물하기] 쿠폰 등록
시각: 2026-05-22 14:23:11 KST
상품: 하트 1만원 충전권 (하트 충전권)
금액: 10,000원
사용자: user_id=****1234
오늘 누적: 23건째
```

### 1.3 ON/OFF 토글

| 항목 | 요구사항 |
|------|---------|
| 구현 방식 | **환경변수 + 재시작** (D7 확정, 2026-05-19) |
| 환경변수 명 | `KAKAO_GIFT_SLACK_NOTIFY_ENABLED` (가칭, `/architect` 단계 최종) — boolean (`true`/`false`) |
| 토글 단위 | F1 알림 전체 ON/OFF (1개 환경변수) |
| 반영 즉시성 | 변경 시 ArgoCD sync 후 반영. 실측 수 분~10분. **즉시 OFF 아님** |
| 변경 절차 | `common-infra-eks-deploy` overlay 환경변수 변경 → PR 머지 → ArgoCD sync |
| 변경 권한 | 인프라/서버 개발자 (운영팀 직접 변경 불가) |
| 초기값 | **ON** (출시 직후 운영 가시성 우선) |
| 긴급 OFF 대안 | 슬랙 측 Webhook 비활성화 또는 채널 Mute (서버 변경 없음) — 운영 매뉴얼에 기재 |

### 1.4 에러·장애 처리

| 시나리오 | 처리 |
|---------|------|
| 슬랙 Webhook 실패 (5xx, timeout) | 서버 로그에 ERROR 레벨로 기록. 쿠폰 등록 트랜잭션은 정상 종료 (알림 실패가 쿠폰 등록에 영향 없음) |
| 토글 OFF 상태 | 알림 skip. 서버 로그에 DEBUG 레벨로 기록 (선택) |
| 메시지 포맷 오류 (예: 상품명 NULL) | 가능한 정보만으로 메시지 발송 + 누락 항목 표기 |

### 1.5 알림 폭증 대비

| 시나리오 | 1차 대응 | 후속 대응 |
|---------|---------|----------|
| 시간당 100건 초과 | 토글 OFF (수동) | 배치 요약 모드 (10분/30분 단위 합산) — 본 프로젝트 외 |
| 채널 사용자 알림 피로 | 별도 채널 분리 (D1 결정) | 메시지 thread 그룹화 — 후속 검토 |

## 2. F2 — 매일 오전 9시 전일자 통계 푸시

### 2.1 발화 조건

| 항목 | 요구사항 |
|------|---------|
| 발화 시각 | **ETL 완료 sensor 후 즉시 발화** (D9 확정, 2026-05-19). 실제 발화 시각은 09:00~10:00 가변 |
| 발화 데이터셋 | 전일(D-1) 00:00:00 ~ 23:59:59 KST |
| 발화 대상 채널 | `#chatops-hellobot-kakao-gift` (D1 확정) |
| 데이터 소스 | BigQuery `hellobot-f445c.server_rdb.coop_marketing_coupon_usage` 일배치 결과 |
| ETL 의존성 | `coop_marketing_coupon_usage` 일배치 ETL 의 완료 마커를 sensor 로 감지 후 발화. 의존 DAG 명·완료 마커 식별은 T-D1 에서 확정 |
| 운영팀 안내 | 발화 시각이 가변(09:00~10:00)임을 운영팀에 사전 공지 |

### 2.2 메시지 포맷

**섹션 A — 어제 단일 (D-1)**:

| 지표 | 정의 | 비고 |
|------|------|------|
| 등록 건수 (성공) | `status IN ('used')` 건수 | 카테고리별 분포 (하트/스킬) |
| 등록 건수 (실패) | 등록 시도 중 실패 응답 건수 | 서버 로그 또는 별도 이벤트 — [검증 필요] |
| 등록 매출 | 성공 등록 건의 정가 합계 (KRW) | 등록 시점 인식. 정산은 별도 |
| 신규 구매자 수 | 카카오 채널 첫 등록 사용자 중 헬로우봇 계정 신규 (가입일 = 등록일) | [data-measurement-plan §2.1](../20260324-coop-integration/data-measurement-plan.md) 정의 |
| 환불·취소 건수 | `status = 'canceled'` 건수 (어제 발생) | |
| 등록 실패율 | `실패 / (성공 + 실패)` | 임계 N% 초과 시 경고 표기 |

**섹션 B — 누적 (출시일 ~ D-1)**:

| 지표 | 정의 |
|------|------|
| 누적 등록 건수 | 출시일 이후 성공 등록 합계 |
| 누적 매출 | 출시일 이후 성공 등록 정가 합계 |
| 누적 신규 구매자 수 | 출시일 이후 카카오 채널 신규 구매자 (DISTINCT user_id) |

**섹션 C — 이상 신호 (조건부)**:

- 어제 등록 실패율 > 5% — 🚨 표기
- 어제 등록 건수 = 0 — ⚠️ 표기 (출시 후 첫 주 제외)
- 어제 환불 건수 > 등록 건수의 10% — ⚠️ 표기

**메시지 예시**:

```
📊 [카카오 선물하기] 일일 리포트 — 2026-05-23 (금) 자

▎ 어제 (5/23) 성과
• 등록: 47건 (성공 45 / 실패 2) — 하트 38 · 스킬 7
• 매출: 583,000원
• 신규 구매자: 18명
• 환불·취소: 1건
• 실패율: 4.3%

▎ 누적 (5/22~5/23)
• 등록: 89건
• 매출: 1,107,000원
• 신규 구매자: 34명

▎ ✅ 이상 신호 없음
```

### 2.3 에러·장애 처리

| 시나리오 | 처리 |
|---------|------|
| DAG 실패 (집계 쿼리 에러) | `SLACK_DATA_ERROR_CHANNEL` 로 자동 알림 (기존 `on_failure_callback` 활용) |
| 슬랙 Webhook 실패 | DAG task 재시도 (Airflow `retries=2`). 최종 실패 시 에러 채널 알림 |
| 일배치 ETL 미종료 | sensor 가 timeout(예: 11:00) 까지 대기 후 timeout 시 `SLACK_DATA_ERROR_CHANNEL` 에러 알림 — `/architect` 결정 |
| 데이터 0건 (출시 전 또는 사용 없음) | 메시지는 정상 발송 (모두 0 으로 표시). 출시 전엔 DAG 자체를 schedule OFF 유지 |

### 2.4 출시 시점 운영

| 시점 | 동작 |
|------|------|
| 출시 전 | DAG 비활성화 또는 skip (메시지 발송 안 함) |
| 출시일 (2026-05-22) | F1 만 가동. F2 는 다음날부터 발송할 수 있도록 schedule 활성화 |
| 출시 +1 일 (2026-05-23) | 첫 F2 메시지 발송 (어제 = 출시일 데이터) |

## 3. 비기능 요구사항

### 3.1 보안

- Slack Webhook URL 은 k8s Secret 으로 관리. 코드·문서 하드코딩 금지
- 사용자 ID 마스킹 (마지막 4자리만 노출)
- Admin 토글은 Admin 페이지 권한 내에서 변경 가능

### 3.2 성능

- F1: 쿠폰 등록 응답 시간에 영향 없음 (비동기 또는 응답 후 처리)
- F2: DAG 실행 시간 5분 이내 (BigQuery 집계 쿼리 + 슬랙 발송)

### 3.3 운영

- F1 토글 ON/OFF 변경은 운영팀이 Admin 페이지에서 직접 (개발팀 개입 불필요)
- F2 DAG 의 schedule 변경은 데이터팀이 Airflow UI 에서 직접

## 4. 검증 기준 (Acceptance Criteria)

### F1

- [ ] 카카오 쿠폰 등록 성공 시 슬랙 채널에 메시지 발송됨 (환경변수 ON 상태)
- [ ] 일반 쿠폰·giftiel 쿠폰 등록 시 본 알림은 발송되지 않음
- [ ] 환경변수 OFF + 재배포 후에는 알림 발송되지 않음
- [ ] 슬랙 Webhook 실패 시 쿠폰 등록 트랜잭션은 정상 종료됨
- [ ] 사용자 ID 가 마스킹된 형태로만 노출됨

### F2

- [ ] 매일 ETL 완료 후 (오전 중) 슬랙 메시지 발송됨
- [ ] 메시지 내 어제 등록 건수·매출·신규 구매자·환불 수치가 BigQuery 직접 쿼리 결과와 일치
- [ ] 누적 지표가 출시일 이후 누적 합계와 일치
- [ ] DAG 실패 또는 ETL sensor timeout 시 `SLACK_DATA_ERROR_CHANNEL` 로 에러 알림 발송됨
- [ ] 출시 전 기간에는 메시지 발송되지 않음 (또는 0 으로 표시)

## 5. 의존성·전제

- [coop-integration](../20260324-coop-integration/) 의 `POST /api/coupon/register` 엔드포인트와 `coupon_prefix_rule` 라우팅이 운영 환경에 배포되어 있어야 함
- BigQuery `coop_marketing_coupon_usage` 일배치 ETL 이 운영 중이고 KST 09:00 이전 종료
- 슬랙 워크스페이스에서 새 채널 생성·기존 채널 분기 정책 합의 완료 (D1)
- 신규 슬랙 Webhook URL 발급 및 k8s Secret 등록 (`/dev-infra` 협조)
