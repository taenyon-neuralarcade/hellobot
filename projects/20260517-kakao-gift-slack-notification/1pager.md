## **Problem**

카카오 선물하기 채널이 곧 출시(2026-05-22 1차)되지만, **운영 가시성이 부재**합니다.

- 사용자가 카카오 선물하기 쿠폰을 등록해도 운영팀이 실시간으로 인지할 수단이 없음 (기존 `#chatops_hellobot_coupon` 채널의 알림 범위·필터 미정 — [검증 필요])
- 일일 카카오 선물하기 성과(등록 건수·매출·신규 구매자 등)를 매일 별도로 BigQuery 쿼리로 확인해야 함 — 운영 루틴화·체크 누락 위험
- 출시 직후 1~2주는 운영팀이 **실시간 상황 인지**와 **일일 추세 인지** 둘 다 필요한 결정적 시점

_root cause_

1. 카카오 선물하기는 신규 채널이라 기존 운영 알림·리포트 체계에 통합되어 있지 않음
2. 헬로우봇 일반 쿠폰과 카카오 선물하기 쿠폰이 같은 DB 테이블(`coop_marketing_coupon_usage`)에 누적되지만, 운영팀 시선에서 카카오만 격리해 보여줄 채널 정책이 없음
3. 일일 성과 리포트는 데이터팀 Looker 대시보드에서 확인 가능하나, 운영팀의 daily 루틴에 push 되어야 의사결정 지연이 없음

## **Customer Job**

1. 운영팀(사업·CS·데이터)이 카카오 선물하기 채널의 **현재 흐름**과 **어제 성과**를 별도 액션 없이 슬랙으로 인지하고 싶다
2. 이상 신호(등록 실패 급증·특정 상품 매진·환불 급증 등)를 빠르게 감지하고 싶다

## **Solution / Feature**

운영팀 전용 슬랙 채널에 두 가지 알림을 발송합니다.

### F1. 카카오 선물하기 쿠폰 등록 실시간 알림

- 사용자가 카카오 선물하기에서 발급받은 쿠폰을 헬로우봇 앱·웹에서 등록(=사용)하는 시점에 슬랙 채널로 즉시 푸시
- 메시지 내용: 등록 시각, 상품명, 상품 카테고리(하트충전권/스킬교환권), 금액, 사용자 ID(마스킹), 그날 누적 등록 건수
- **ON/OFF 토글**: 카카오 채널이 확장되면 알림 빈도가 높아져 운영 채널 과부하 우려 → 즉시 OFF 할 수 있는 수단 필요 (Admin 페이지 토글 권장)
- 헬로우봇 일반 쿠폰·giftiel 등 카카오 외 발급처는 본 알림에서 **제외** (`coupon_prefix_rule` 의 카카오 라우팅만)

### F2. 매일 오전 9시 카카오 선물하기 전일자 통계 푸시

- 매일 KST 09:00 슬랙 채널에 어제(D-1) 카카오 선물하기 성과 요약 메시지 발송
- 핵심 지표 (어제 단일):
  - 등록 건수 (성공/실패) — 상품 카테고리별 분포
  - 등록 매출 (등록 시점 인식 기준, 정산은 별도)
  - 신규 구매자 수 (카카오 채널 첫 등록 + 헬로우봇 계정 신규)
  - 환불·취소 건수
  - 등록 실패율 (시도 대비)
- 누적 지표 (출시일~어제):
  - 누적 등록 건수
  - 누적 매출
  - 누적 신규 구매자 수
- 이상 신호 강조 (예: 어제 등록 실패율 > N%, 어제 등록 건수 = 0 등)

## **Success Metric**

input metric

1. 운영팀이 카카오 선물하기 채널의 **실시간 상태**를 별도 액션 없이 슬랙으로 인지 가능 (정성)
2. 일일 통계 푸시 후 운영팀이 BigQuery·Looker를 별도로 열어 보는 빈도 감소

output metric

1. 카카오 선물하기 출시 후 첫 2주간 운영 이슈(등록 실패·매진 등) 평균 인지 지연 시간 < 1시간
2. 매일 09:00~09:15 사이 슬랙 통계 메시지 발송률 100% (DAG 안정성)

## **Benchmark**

- 사내 기존: `hellobot_daily.py` DAG 가 매일 KST 11:00 슬랙 알림 발송 (전사 일일 KPI)
- 사내 기존: `notifyToSlack()` 유틸 + `#chatops_hellobot_coupon` 채널이 일부 쿠폰 이벤트를 알림 중 (범위·필터 [검증 필요])

## **Trade off**

- **인프라 리소스**: 신규 슬랙 채널 1개 + Webhook URL 시크릿 + Airflow DAG 1개 + Admin 토글 1건 (소형)
- **운영 부담**: 출시 직후 알림 폭증 시 채널 과부하 가능성 → ON/OFF 토글로 완화
- **데이터 정합**: 일일 통계는 BigQuery 기반 (`coop_marketing_coupon_usage` 일배치 ETL 종료 후 집계). 일배치 종료 시점이 09:00 이전이어야 함 — [검증 필요]
- **중복 알림**: 기존 `#chatops_hellobot_coupon` 의 알림 범위와 충돌 가능 → 신규 채널 분리 또는 기존 채널 분기 정책 결정 필요

## **Unhappy Path**

- **알림 폭증**: 카카오 노출 확대로 등록 건수가 시간당 100건 이상이 되면 슬랙 채널이 알림으로 도배되어 정작 중요한 신호를 놓침 → ON/OFF 토글 + 시간당 N건 초과 시 배치 요약 모드 전환(후속 검토)
- **DAG 장애**: 매일 09:00 푸시가 실패해 어제 통계가 누락되면 운영팀이 인지 못함 → 실패 시 데이터팀 에러 채널(`SLACK_DATA_ERROR_CHANNEL`)로 알림
- **시크릿 노출**: Slack Webhook URL 유출 시 외부 스팸 가능 → k8s Secret 관리, 코드 하드코딩 금지
- **데이터 정합 오류**: 일배치 ETL 지연 시 통계가 부정확 → DAG 의 sensor 또는 지연 시 알림 표시

## **Feedback loop**

1. 출시 후 1주: 운영팀과 알림 메시지 포맷·지표 항목 리뷰. 추가 필요 지표·제거 가능 지표 식별
2. 출시 후 2주: ON/OFF 토글 사용 빈도·시간당 알림량 모니터링. 배치 요약 모드 필요성 판단
3. 출시 후 1개월: 일일 통계 푸시가 운영 의사결정에 실제 활용되는지 정성 평가 (운영팀 인터뷰)
