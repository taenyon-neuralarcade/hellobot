## 1. 대상 범위

| 항목 | 포함 | 제외 |
|------|------|------|
| 리포 | `hellobot-server` (AdminJS 옵션 + 서비스 + 진단 SQL) | 그 외 모든 리포 |
| 파트 | 서버 단일 | iOS / Android / Web / Studio / Data — 영향 없음 |
| 운영 | 운영 DB 진단 + 핫픽스 코드 + 정합화 보정 | 일반 모니터링·알람 인프라 |
| 사용자 영향 | 운영자(직접) / 엔드유저(간접 — 발급 쿠폰 보호) | 신규 화면·UX 없음 |

## 3. 기능 요구사항 (FR)

### FR-1 동기화 정책 (CRUD ↔ CouponSpec/CouponCondition)

| ID | 요구사항 | 출처 스토리 | 우선순위 |
|----|---------|------------|---------|
| FR-1.1 | skill 상품의 `fixedMenuSeq` 가 변경될 때, 기존 `CouponCondition.skillSeqs` 를 in-place 로 갱신하지 **않는다**. 옛 `CouponSpec` 은 신규 발급 차단 상태로 동결(`issueEndDate=now()`)하고, 새 `CouponSpec` + `CouponCondition` 을 생성한 뒤 상품의 `couponSpecSeq` 를 새 spec 으로 교체한다. | US-01 | **P0** |
| FR-1.2 | 동일 `skillSeqs` 에 매칭되는 활성 `CouponSpec` 은 항상 단 1 개만 존재해야 한다. 동일 스킬에 새 상품을 등록·재등록할 때 옛 spec 은 자동으로 동결되어 매칭 후보에서 빠진다. | US-02 | **P0** |
| FR-1.3 | skill 상품 `delete` 시 연동 `CouponSpec` 은 신규 발급 차단 상태로 동결되고 (정확한 방식은 D-1 결정), 발급된 쿠폰은 만료 전까지 사용 가능 상태를 유지한다. | US-03 | P1 |
| FR-1.4-a | `productType` heart → skill 전환 처리는 D-2 정책에 따른다. 허용 시 새 `CouponSpec` + `CouponCondition` 생성 + `couponSpecSeq` 업데이트가 수반. 비허용 시 validation error 로 reject. **결함 D (ISS-071) 대응**. | US-04 | P1 |
| FR-1.4-b | `productType` skill → heart 전환 처리는 D-2 정책에 따른다. 허용 시 옛 `CouponSpec` 동결(`issueEndDate=now()`) + `couponSpecSeq` 의 처리 정책(보존 vs 클리어)을 D-2 가 포함. 비허용 시 validation error 로 reject. **결함 E (ISS-071) 대응 — orphan 잔존 차단**. 이미 발생한 orphan 의 정리는 FR-3.3 정합화 범위. | US-04 | P1 |
| FR-1.5 | 상품명 변경으로 `CouponSpec.name` UNIQUE 충돌이 발생하면 D-3 정책에 따라 처리. **D-3 의 단독 결정은 금지** — D-9 (CouponSpec.name 노출 경로 1줄 확인) + D-6 (발급 시점 스냅샷 정책) + collision-free 증명이 선결. 어떤 정책을 선택하든 부분 정합성(상품만 수정, spec 미동기) 상태로 commit 되지 않는다. | US-05 | P1 |

### FR-2 가시화 (운영자 인지)

| ID | 요구사항 | 출처 스토리 | 우선순위 |
|----|---------|------------|---------|
| FR-2.1 | skill 상품 등록·수정 시 `fixedMenuSeq` 가 비어 있으면 validation error 로 reject 한다. silent skip 으로 `couponSpecSeq=null` 상품을 만들지 않는다. | US-06 | P1 |
| FR-2.2 | 상품 INSERT/UPDATE 와 CouponSpec 동기화의 **UNIQUE 충돌·예외**는 AdminJS UI 상 명백한 error notice 로 표시되어야 하고, success 와 시각적으로 구분되어야 한다. **결함 F (ISS-071) 의 UNIQUE 충돌 silent catch 차단 책임은 본 FR**. | US-07 | P1 |
| FR-2.3 | 상품 INSERT/UPDATE 와 CouponSpec 동기화의 **트랜잭션 경계 + 부분 실패 시 보상 패턴**을 /architect 가 명세한다. "가능하면 단일 트랜잭션" 같은 임의 표현 금지. AdminJS 훅 모델(상품 commit 이후 afterEdit 실행)이 단일 트랜잭션을 막는다면, 보상 동작이 결정론적이어야 한다. **결함 H (ISS-071) 대응**. | US-07 | **P0** (P1 → 격상 — C-2) |
| FR-2.4 | FR-2.3 의 부분 실패가 발생한 경우, 시스템은 **다음 사용자 요청 시점에도 결정론적으로 같은 결과**(retry 가능 / 자동 정합화 / 운영자 수동 정정 중 하나) 에 도달해야 한다. "이전 실행의 잔재가 다음 실행에 silent 한 영향" 을 남기지 않는다. | US-07 (확장) | P1 |

### FR-4 데이터 보존 (불변 조건)

| ID | 요구사항 | 출처 스토리 | 우선순위 |
|----|---------|------------|---------|
| FR-4.1 | 이미 발급된 `Coupon` (특히 미사용분) 의 **적용 가능 대상(skillSeqs 등 CouponCondition 기반)** 이 운영자 측 CRUD 동작으로 인해 **retroactively 변경되지 않는다**. 새로운 적용 대상은 신규 발급에만 반영된다. | US-01 | **P0** |
| FR-4.2 | 이미 발급된 `Coupon` 은 만료 전까지 사용 가능 상태를 유지한다 (운영자가 상품을 삭제·비활성화해도). | US-01, US-03 | P1 |
| FR-4.3 | 이미 발급된 `Coupon` 의 **표시 필드 (`name`·할인율·`usableDays`·`minPurchasePrice` 등)** 의 발급 시점 값 보존 정책을 명시한다. 보존 방식(옛 spec row 의 필드 변경 금지 vs `Coupon` 행 자체에 snapshot 컬럼) 은 D-6 결정. 어느 방식이든 `CouponSpec.name` 의 운영자 임의 변경이 발급된 쿠폰의 표시 일관성을 깨면 안 된다. **C-5 의 핵심 — FR-4.1 의 보호 범위를 적용 대상에 한정하지 않음을 명시**. | US-01 (확장) | **P0** (D-6 결정 의존) |
