# REVIEW-5: 스코프·비즈니스 비판 리뷰

**리뷰어 렌즈**: 스코프·비즈니스 (스코프 적절성·D-1~D-5 비즈니스 적합성·핫픽스 vs 정식 솔루션·우선순위·이해관계자)
**검토 대상**: requirements.md / user-stories.md / 1pager.md (2026-05-21 작성)
**작성일**: 2026-05-21
**전체 평가**: **H (High)** — "핫픽스" 라벨에도 불구하고 실질 스코프는 어드민 옵션 재설계에 가깝고, 이해관계자 합의·발생 빈도 분석·비용/효과 비례성·정량화된 비즈니스 임팩트가 모두 누락되어 의사결정 근거가 약하다.

## 요약

본 핫픽스는 결함을 정확히 식별했으나, **(a) 9건 결함 일괄 처리(풀 스코프)** + **(b) `핫픽스` prefix** + **(c) 마감 미정** 의 조합이 모순적이다. 진정한 핫픽스라면 P0-A/P0-B 의 운영 발생 빈도와 정량 임팩트(예: 영향 받을 수 있는 발급 쿠폰 수, CS 문의 예상치)가 1pager 첫 줄에 와야 하는데 모두 빠져 있고, D-5 "풀" 권장의 근거가 "같은 영역을 두 번 건드리지 않기 위해" 라는 개발자 편의 한 줄에 그친다. 사용자(엔드유저) 영향은 강조되나 **CS/운영/재무(정산) 팀과의 합의 절차가 정의되지 않았고**, Success Metric 의 핵심 지표 "CS 문의 0 건" 은 noise/false negative 가 큰 후행 지표라서 회수성이 약하다. D-1~D-5 권장안 자체는 합리적이나 권장 근거가 거의 1줄이라 비판적 검증 없이 채택되면 부채를 만든다.

## 발견한 문제

### F-1: "핫픽스" 라벨 vs 실제 스코프 불일치 (Branding mismatch)

- **현상/위반 지점**: 1pager Solution / requirements §2 / D-5 "풀"
- **비즈니스 영향**: 9건 결함(P0 2 + P1 6 + P2 1) 중 7건을 한 번에 처리하면서 트랜잭션 경계 재설계(FR-2.3) + delete 액션 신설(US-03) + productType 전환 정책 도입(US-04) + 이름 정책 변경(D-3) 까지 포함된다. 이는 사실상 `CoopMarketingProduct.options.ts` 의 어드민 동기화 모듈 재설계이고, "핫픽스" 라는 단어가 주는 1~2 일짜리 봉합 작업이라는 인식과 분리(branch/PR/배포)·QA 부담·롤백 위험을 호도한다.
- **권장 액션**: 명칭을 "Skill Coupon Sync Stabilization (1st phase)" 등으로 재설정하거나, 진짜 핫픽스(P0-A 의 in-place 갱신 차단 1건만)와 후속 정합성 작업(P1) 을 분리. 적어도 1pager 에 "이 작업은 핫픽스 절차로 진행하지만 변경 규모는 어드민 옵션 재설계 수준이다" 를 명시.

### F-2: 비즈니스 임팩트의 정량화 부재 (Decision-grade evidence 누락)

- **현상/위반 지점**: 1pager Problem / Success Metric / TODO-026 (마감 "사용자 협의")
- **비즈니스 영향**: P0 라고 명시되어 있지만 **(a) 현재 운영 DB 의 위반 row 수**, **(b) 영향받을 수 있는 발급 쿠폰 수**, **(c) 운영자가 fixedMenuSeq 변경/재등록을 분기당 몇 번 수행하는지**, **(d) 만약 진단 결과 0 row 면 본 핫픽스의 시급성이 어떻게 변하는지** 가 전혀 없다. 의사결정자가 본 1pager 만 받으면 "왜 5/22 출시보다 우선이 아니고 마감조차 미정인가" 를 설명할 수 없다.
- **권장 액션**: T-1~T-4 진단 SQL 을 5/22 출시 직후 즉시 실행하고, **그 결과(row 수 + 운영자 작업 빈도 추정)** 를 입력으로 D-5(스코프) 와 일정을 결정. 진단 결과가 0 row 면 이 프로젝트의 priority 가 ⭐⭐⭐ 인지 자체가 재검토 대상.

### F-3: 비용/효과 비례성 — 저빈도 케이스에 P1 가중

- **현상/위반 지점**: FR-1.4 / US-04 / D-2 (productType heart↔skill 전환)
- **비즈니스 영향**: heart↔skill 전환은 운영 실무상 거의 발생하지 않는 케이스(서로 다른 결제·정산·표시 경로). 그럼에도 P1 로 분류되어 코드/테스트 부담을 키운다. 결함 ISS-071 매트릭스 D/E 가 P1 으로 분류된 근거 자체가 약하다 — 이론적 가능성과 실 발생 빈도가 구분되지 않았다.
- **권장 액션**: D-2 를 "validation error 로 reject" 로 결정 + **이걸 P1 핫픽스로 묶지 말고 1~2 라인 추가만으로 끝낸 뒤 P3(차후 정리) 로 강등**. 풀 스코프 6건 중 사실상 0.5건 분량.

### F-4: D-5 "풀" 권장 근거의 빈약함

- **현상/위반 지점**: requirements §8 D-5 "1 회 배포로 끝. P1 가 6 건이라 따로 누적시키면 같은 영역을 두 번 건드리게 됨"
- **비즈니스 영향**: 이 한 줄에는 (a) 단계적 배포의 위험 감소 효과, (b) P1 의 운영 발생 빈도, (c) 1차 출시 직후 시기의 가용 QA·배포 윈도우 부족, (d) 같은 영역 두 번 건드리는 비용 vs 큰 PR 한 번 머지의 충돌 비용 비교가 없다. **트레이드오프 분석이 결론에 비해 결여**.
- **권장 액션**: 최소 스코프(P0-A: in-place 갱신 차단만) + 다음 스프린트 P1 묶음의 두 단계로 분리하는 옵션을 명시적으로 평가. 둘 중 하나를 고르는 게 아니라 **단계적 배포 = "P0 즉시 + P1 1주 후"** 가 디폴트 권장이어야 한다.

### F-5: Non-Goals "I, P2" 의 정당성 미증명

- **현상/위반 지점**: 1pager Non-Goals / user-stories Non-User-Stories — "CouponSpec/CouponCondition 어드민 직접 수정 차단 (I, P2) — 별건"
- **비즈니스 영향**: 본 핫픽스가 동기화 정책을 강화해도, **운영자가 AdminJS 에서 CouponSpec/CouponCondition 을 직접 편집할 수 있는 한 정합성 보장이 무효화**된다. 즉 본 핫픽스의 FR-1.1~1.2 가 효력을 갖기 위한 **선결 조건**이 별건으로 분리된 상태. 핫픽스 배포 후에도 정합성 위반이 계속 발생할 수 있는데 Success Metric 은 "위반 0 건" 이라 모순.
- **권장 액션**: 최소한 **direct edit 시 경고 배너** 라도 본 핫픽스에 포함, 또는 Success Metric 에 "단, 운영자 SOP 가 direct edit 을 금지함" 을 전제로 명시. 이상적으로는 ISS-071 I 를 P2 가 아니라 본 핫픽스의 사전 조건으로 격상.

### F-6: 이해관계자 합의 절차 부재 (CS/운영/재무)

- **현상/위반 지점**: requirements.md / 1pager / user-stories 어디에도 CS·운영·재무팀 사인오프가 정의 안 됨
- **비즈니스 영향**: D-1(soft delete)·D-2(전환 비허용)·D-3(이름 suffix) 모두 **운영자 UX·CS 대응 스크립트·정산 데이터 흐름**에 영향. 코디네이터/PM/dev 만 합의하고 배포하면 운영자가 "왜 갑자기 transition 이 reject 되나" 류 항의가 발생할 수 있다.
- **권장 액션**: D-1~D-3 별로 stakeholder 컬럼 추가 (예: D-1=CS 팀 / D-2=운영팀 / D-3=마케팅·운영팀). 사용자 협의 단계에 "운영팀 사인오프" 를 명시.

### F-7: D-3 suffix 의 사용자 노출 부작용 미평가

- **현상/위반 지점**: D-3 "`${productName} 이용권 #${productCode}` 형태"
- **비즈니스 영향**: `CouponSpec.name` 은 운영 어드민 내부 식별자에 그치는지, **엔드유저가 보유 쿠폰 리스트에서 보는 표시명**인지 명세에 없다. 만약 후자라면 사용자에게 `점성술사 이용권 #SK-1284` 같은 문자열이 노출되어 브랜드/UX 품질 저하. 1pager Non-Goals 가 "엔드유저 표면 변경 없음" 이라 단언하는데 D-3 이 이 단언을 위반할 수 있다.
- **권장 액션**: `name` 컬럼의 사용 경로(스토어 표시 / 쿠폰함 표시 / 푸시 알림 본문 / 어드민 전용) 매핑을 /dev-server 가 1줄로 확인. 사용자 노출이면 suffix 가 아니라 **별도 internal_key 컬럼 신설** 또는 **atomic reject + 운영자에게 이름 변경 안내** 로 회피.

### F-8: D-1 soft delete 와 카카오 환불 시나리오의 정합성 미검토

- **현상/위반 지점**: D-1 "soft (issueEndDate)" / FR-4.2 / US-03
- **비즈니스 영향**: 카카오 선물하기는 **환불 시 발급 쿠폰을 회수**해야 한다 (TODO-022/ISS-056 의 사용취소 회로가 이미 그 영역). soft delete 로 spec 을 유지하는 정책이 환불 회수 시 "이미 사용 불가능한 옛 spec 의 쿠폰을 어떻게 회수하나" 같은 케이스와 충돌하지 않는지 확인 안 됐다. 또한 발급분 보존이 "표준" 이라는 1차 권장 근거 자체가 어디서 온 표준인지 명시 안 됨.
- **권장 액션**: TODO-022 의 사용취소 시나리오(/coupon 사용·환불 회로)와의 교차 점검을 /architect 의 선행 과제로 명시. "발급분 보존이 표준" 의 출처(예: 쿠폰 약관·결제 표준·내부 결정) 1줄 인용.

### F-9: D-4 "위반 발견 시 일회성 SQL" 의 결정 조기성

- **현상/위반 지점**: D-4 "점검 결과 = 0 row 면 불필요, 위반 발견 시 일회성 SQL (작은 규모 예상)"
- **비즈니스 영향**: 점검 SQL(T-1~T-4) 을 아직 실행하지도 않은 상태에서 "작은 규모 예상" 결론을 미리 적어 두면 막상 진단 결과 row N=100+ 일 때 일회성 SQL 적용 자체가 운영 위험이 됨. 정합화 작업의 위험은 운영 데이터 규모에 비례.
- **권장 액션**: D-4 결정 자체를 **진단 SQL 실행 후로 이연**. 1차 권장안 컬럼을 "진단 결과 의존 — 결정 보류" 로 수정.

### F-10: Success Metric 의 회수성 약함

- **현상/위반 지점**: §7 검증 기준 / 1pager Success Metric — "CS 문의 0 건, 14일"
- **비즈니스 영향**: "CS 문의 0 건" 은 (a) 사용자가 문의하지 않으면 0 으로 보이는 false negative, (b) CS 채널이 여럿이면 누락, (c) 14 일이 정합성 깨짐의 잠복기보다 짧을 수 있음(쿠폰 사용 기한이 30~90일). 핫픽스 효과 측정으로 부적합.
- **권장 액션**: 선행 지표(진단 SQL 0 row) 를 1차 지표로, 후행 지표(CS 문의) 를 보조 지표로 명시. 14일을 30일+ 로 늘리고, 측정 채널(어드민 헬프데스크/Zendesk/직통 챗) 정의.

### F-11: TODO-013 umbrella·TODO-023 와의 일정·배포 윈도우 경합

- **현상/위반 지점**: 1pager 일정 "1차 출시 안정화 + TODO-013 우선순위 협의 후" / TODO-013 잔여 ISS-043 / ISS-052 / TODO-023 compCode env 분리
- **비즈니스 영향**: 5/22 1차 출시 후 카카오 측 정산/인증 모니터링(TODO-023 영역) 과 본 핫픽스가 같은 hellobot-server 모듈을 건드리고, TODO-013 산하 잔여 항목(스킬이용권 인트로·자동복구 등) 도 같은 서버 PR 큐로 들어온다. **본 프로젝트가 큰 PR(어드민 재설계급) 이라 PR 큐의 다른 항목을 지연**시키거나 conflict 를 유발할 가능성.
- **권장 액션**: 본 핫픽스 PR 의 머지 윈도우를 TODO-023 (compCode env) 머지 이후로 직렬화. 동일 영역 워크트리 동시 활성이 발생하면 base rebase 비용 명시 (CLAUDE.md §동시 작업 충돌 회피).

### F-12: 운영 정책으로 회피 가능한 영역 미평가

- **현상/위반 지점**: 전체 (특히 US-02 동일 스킬 재등록 / US-04 productType 전환)
- **비즈니스 영향**: 어드민 super admin 은 소수 운영자다. "동일 스킬에 새 상품을 재등록할 때는 옛 상품을 먼저 비활성화 후 1시간 대기" 같은 **운영 SOP 한 줄로 P0-B 의 발생 빈도를 0 에 가깝게 누를 수 있는지** 가 평가되지 않았다. 코드 핫픽스의 정당성은 "SOP 로 회피 불가" 또는 "SOP 회피 비용 > 코드 비용" 일 때만 성립.
- **권장 액션**: 각 P0/P1 결함에 대해 "SOP 회피 가능성" 컬럼을 issues.md ISS-071 매트릭스에 추가. SOP 회피 가능한 항목은 단기 SOP + 장기 코드 의 두 단계로 분리.

### F-13: 장기 부채 — 어드민 옵션 파일의 분기 증식

- **현상/위반 지점**: tasks T-8 — `CoopMarketingProduct.options.ts` 의 5 개 케이스 분기를 한 파일에 누적
- **비즈니스 영향**: 향후 신규 상품 유형(시즌 상품·굿즈·구독 등) 도입 시 같은 파일의 분기가 더 늘어난다. 본 핫픽스가 **분기 모듈 재구성(예: 정책 strategy 분리) 없이 if-else 증식으로 끝나면**, 다음 상품 유형 추가가 본 핫픽스보다 더 큰 리스크를 동반.
- **권장 액션**: /architect 단계에서 정책 분리(ProductSyncStrategy 또는 동등) 를 명시적으로 평가. 최소한 결정 로그에 "본 핫픽스는 기존 구조를 유지하며 분기 추가만 함, 구조 개선은 별건" 을 남겨 부채를 가시화.

### F-14: 마감일 부재의 비즈니스 신호 약화

- **비즈니스 영향**: ⭐⭐⭐ + P0 라고 표시된 항목의 마감이 "사용자 협의" 면 의사결정자에게 우선순위가 잘못 전달된다. 위기 가시성 약함 — 다른 ⭐⭐⭐ 항목(비트윈 인프라 분리·카카오 출시) 에 밀려 영구 연기될 위험.
