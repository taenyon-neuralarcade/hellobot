# REVIEW-2: 엔드유저·CS 영향 비판 리뷰

**리뷰어 렌즈**: 엔드유저·CS 영향 (쿠폰 보유자 시나리오·표시 일관성·CS 절차)
**검토 대상**: requirements.md / user-stories.md / 1pager.md (2026-05-21 작성)
**작성일**: 2026-05-21
**전체 평가**: **H (High risk — 엔드유저 표시 정합성·CS 가시성·재발급 시 중복 보유 시나리오가 본 문서에서 모두 미정의)**

## 요약

요구사항은 "발급 시점 기준 보호"라는 추상 원칙(FR-4.1)을 천명했지만, 그 원칙이 사용자에게 보이는 표면(스킬 상세의 적용 가능 쿠폰 배지·쿠폰 리스트의 표시명/할인율/기간/적용 스킬·결제 화면의 자동 선택·푸시 알림 본문)에서 어떻게 구체화되는지는 어디에도 정의되어 있지 않다. retroactive 변경을 막아도 **운영자가 옛 상품을 비활성화/삭제/이름변경하면 보유 쿠폰의 "표기"는 자동으로 따라 바뀌거나(`CouponSpec.name` 갱신 경로 — F의 P1) 끊겨버린다(orphan)** 는 점이 사용자 측 일관성을 깨뜨린다. 또한 정합화(FR-3.3) 전에 이미 두 개 활성 spec 으로 쿠폰을 발급받은 사용자의 사후 처리, 카카오 환불/교환과의 정합성, CS 가 사용할 조회 도구 — 어느 것도 본 핫픽스 문서에 들어와 있지 않다.

## 발견한 문제

### F-1: "발급된 쿠폰의 적용 대상"만 보호하고 표시명·할인율·기간·minPurchasePrice 의 보존 정책은 미정의
- **현상/위반 지점**: FR-4.1 / FR-4.2 / US-01. FR-4.1 은 "적용 대상(skillSeqs)"만 retroactive 변경 금지를 명시. 옛 `CouponSpec` 을 `issueEndDate` 로 동결하더라도 동일 spec row 의 `name` / `discountRate`(또는 가격) / `usableDays` / `minPurchasePrice` 가 운영자에 의해 바뀔 수 있는지 침묵.
- **엔드유저 시나리오**: 사용자가 "체질테스트 100% 할인 쿠폰" 으로 발급받았는데, 운영자가 옛 spec 의 `name` 을 "체질테스트 50% 할인" 으로 바꾸면 — 적용 대상은 여전히 A 라 사용은 되지만 쿠폰 리스트·결제 화면에서 표기가 50% 로 바뀜. 실제 결제 시점에는 100% 가 적용되어 사용자는 "표기는 50인데 100 깎였네?" 또는 그 반대로 혼란.
- **CS 영향**: "내 쿠폰 이름이 갑자기 바뀌었어요" / "표시는 X 인데 실제 할인은 Y 더라" — 원인 추적 불가 (운영자가 어떤 필드를 언제 바꿨는지 이력 없음, 본 핫픽스 Non-Goals 에 변경 이력 UI 제외).
- **권장 액션**: FR-4 에 명시적 컬럼 분리 필요 — 발급 쿠폰의 "표시 스냅샷"이 보존되어야 할 필드(name, discountRate/금액, usableDays, minPurchasePrice) vs 변경 허용 필드를 enumerate. 권장: 발급분 보호 대상이 되는 옛 spec 의 `name` / 할인율 / 기간은 동결 후 운영자 수정 자체를 막거나, 발급 시점의 값을 `Coupon` row 에 스냅샷.

### F-2: `CouponSpec.name` suffix 정책(D-3)이 발급분 표기를 바꿔버린다
- **현상/위반 지점**: D-3 1차 권장안 "suffix" + US-05 acceptance "성공" + FR-4.1 의 "적용 대상" 한정. D-3 권장안은 "이미 발급된 spec 의 name 은 변경하지 않음" 단서를 달았지만, 본 핫픽스의 동기화 정책(FR-1.1)이 옛 spec 을 동결하고 새 spec 으로 교체하므로 — 같은 운영자 동작이 "fixedMenuSeq 변경"이 아니라 "productName 변경"인 경우는 FR-1.1 경로를 타지 않고 `afterEdit` 의 `name` 갱신 경로를 그대로 탄다. 그 결과 옛 발급 쿠폰의 표기가 새 이름으로 바뀐다.
- **엔드유저 시나리오**: 운영자가 상품명을 "체질테스트 100% 할인권" → "체질테스트 무료체험권" 으로 단순 리네이밍. 사용자 쿠폰함의 옛 쿠폰 이름이 일제히 "무료체험권"으로 바뀜. 사용자는 "안 산 쿠폰이 추가됐다" 또는 "기존 쿠폰이 사라졌다"로 오인.
- **CS 영향**: "쿠폰함 갱신 안 됨"·"내가 받은 쿠폰이 아닙니다" 문의. 운영자가 무심코 한 작은 텍스트 수정이 보유 쿠폰 전수 표기 변경으로 번짐.
- **권장 액션**: FR 에 "productName 변경 시 동기화 범위" 명시 — 옛 발급분의 표기는 발급 시점 name 으로 freeze (Coupon 또는 옛 CouponSpec row 에 보존), 새 발급분만 새 name. US-05 의 acceptance 에 "옛 발급 쿠폰의 표기는 변하지 않는다"를 추가.

### F-3: 동일 스킬 활성 spec 2 개 상태에서 이미 두 개의 쿠폰을 받은 사용자의 사후 처리 미정의
- **현상/위반 지점**: FR-3.3 / US-09 / Success Metric "쿠폰 중복 발급 0 건". 핫픽스 배포 전 운영 DB 에 이미 발생한 "동일 사용자, 동일 스킬, 활성 쿠폰 2 장" 상태를 발견했을 때 무엇을 어떻게 하는지가 비어 있음.
- **엔드유저 시나리오**: 사용자 X 가 두 채널(예: 동일 스킬에 대한 두 번의 카카오 선물 수신)로 쿠폰 2 장을 발급받은 상태. 운영자가 정합화 SQL 로 spec 한 개를 issueEndDate 동결해도 두 쿠폰은 그대로 남아 결제 시 자동선택 우선순위에 따라 한 쪽만 소모됨. 나머지는 만료 전까지 "사용 가능" 상태로 표시되지만, 실 결제에서는 항상 한 장만 자동 선택되어 사용자가 "한 장이 안 써진다"로 인지할 수 있음. (혹은 두 장이 동시에 다른 결제에 동작하면 "왜 한 번에 100% 가 두 번 적용되지" 류 분쟁.)
- **CS 영향**: 카카오 환불 요청 시 어느 쿠폰을 회수해야 하는지 불명확. 카카오 보낸 사람 → 받은 사람 → 쿠폰 1:1 매핑이 깨진 상태로 환불 처리.
- **권장 액션**: US-09 acceptance 에 "이미 중복 발급된 Coupon 의 처리 방침" 항목 추가 — (a) 그대로 둠(사용자 보호) (b) 운영자가 카카오 측 환불 정산과 연계하여 1 장 회수 (c) 한 장을 만료처리하고 사용자 안내 푸시. 본 결정을 D-X 로 사용자 협의 필요.

### F-4: 카카오 선물하기 측 환불·교환 ↔ 우리 spec 동결 흐름의 정합성 부재
- **현상/위반 지점**: 전 문서. 카카오 측에서 발신자가 선물을 환불하거나 수령자가 교환 요청 시 우리 쪽에서 어떤 Coupon/CouponSpec 상태 전이가 일어나야 하는지 명시 없음. Non-Goals 에도 "외부 정산은 별건"으로 배제되어 있지 않으나 들어와 있지도 않다.
- **엔드유저 시나리오**: 선물 발신자가 카카오에서 환불 → 카카오 측은 "선물 회수됨" 상태이지만 우리 DB 에는 Coupon row 가 active 로 남음. 사용자가 그 쿠폰을 사용 시도 → 결제 성공 → 카카오와 정산 불일치.
- **CS 영향**: "선물 환불 받았는데 쿠폰이 아직 사용 가능해요" vs "사용했더니 환불 처리가 막혀요" 같은 모순 문의. 본 핫픽스가 옛 spec 을 보존하는 정책을 강화할수록 이런 환불 ↔ 우리 쿠폰 불일치가 심화될 수 있음.
- **권장 액션**: 본 핫픽스 Non-Goals 에 "카카오 측 환불/교환 시 우리 쪽 Coupon 회수 정책은 별건"임을 명시하거나, FR 에 최소한 "외부 시스템 회수 요청에 대응할 수 있는 식별자(가령 카카오 주문번호 ↔ Coupon.seq)가 유지되어야 한다"를 추가. 부모 프로젝트 architecture.md 와 정합성 확인.

### F-5: 옛 상품 비활성화·삭제 후 "스킬 상세 / 쿠폰 리스트 / 결제 화면 / 검색·추천" 표시 일관성 미정의
- **현상/위반 지점**: FR-4.2 ("사용 가능 상태 유지")만 추상적으로 명시. 사용자가 볼 5 개 표면 — (1) 스킬 상세의 "쿠폰 적용 가능" 배지/카드, (2) 마이페이지 쿠폰 리스트, (3) 결제 시도 시 자동선택 후보, (4) 검색·추천 영역의 쿠폰 노출, (5) Braze 등 푸시 발송 시 본문 — 가 각각 어떤 데이터를 보는지 spec out 안 됨.
- **엔드유저 시나리오**: 운영자가 옛 상품을 delete (D-1 soft) 후, 사용자가 스킬 상세에 들어가면 적용 가능 쿠폰이 정상 노출되어야 함. 그러나 옛 spec 의 `name`/`discountRate` 가 운영자 의도로 바뀌었으면 표기가 어긋남(F-1, F-2). 또는 사용자 쿠폰 리스트에는 옛 이름으로 떠 있는데 스킬 상세에서는 새 이름으로 떠 있을 수도 있음 (옛 spec 보존 + 새 spec 신규 발급 화면에서 같이 노출되는 경우).
- **CS 영향**: "쿠폰함과 스킬 화면 표기가 달라요" 류 문의. 사용자가 어느 쪽을 신뢰해야 할지 모름.
- **권장 액션**: FR-4 에 "발급된 쿠폰의 표시 스냅샷 일관성" 절을 추가. 모든 표면에서 동일한 (발급 시점 기준) name·할인율·기간을 보여야 한다는 원칙 명시. /architect 가 어느 컬럼을 freeze 하고 어느 조인을 쓸지 결정해야 한다는 위임.

### F-6: 옛 쿠폰 사용 시점의 결제·발급 후속 백엔드 경로(스킬 0 하트 구매)가 옛 fixedMenuSeq 기준으로 작동함이 본 문서에서 보증되지 않음
- **현상/위반 지점**: FR-1.1 / FR-4.1 + 부모 architecture.md §처리 로직(쿠폰 → 스킬 구매 0 하트). 옛 `CouponCondition.skillSeqs=[A]` 가 보존되므로 `coupon.ts:161-188` / `:257` 의 매칭은 통과하지만, 옛 상품(`CoopMarketingProduct`) 자체가 delete 또는 비활성화된 후 "쿠폰 → 스킬 구매" 의 후속 단계가 외래 참조나 가격/할인율 산정에서 옛 상품 row 를 요구하지는 않는지 검증되지 않음.
- **엔드유저 시나리오**: 사용자가 옛 쿠폰으로 A 스킬 0 하트 구매 시도 → 매칭은 통과 → 그러나 결제 backend 가 product row 를 참조해 가격 환산하는 단계가 있다면 NULL 참조로 실패.
- **CS 영향**: "쿠폰이 보이는데 사용하려니 오류". 가장 흔하고 가장 빠른 폭주성 CS 패턴.
- **권장 액션**: NFR-5 또는 FR-4.2 에 "옛 상품이 비활성/삭제된 상태에서 쿠폰 사용 시 결제·발급 경로 end-to-end 정상 동작 확인"을 acceptance criterion 으로 명시. /qa-test-cases.md 에 명시적 테스트 시나리오 추가.

### F-7: 만료일·이용 가능 기간 변경 시 사용자 경험 미정의
- **현상/위반 지점**: 전 문서. CouponSpec 에는 `usableDays` (발급 후 N 일 사용 가능) 같은 만료 산식 필드가 존재. FR 은 적용 대상만 다루고 만료 정책 변경(예: 운영자가 `usableDays` 를 30 → 7 로 변경) 시 옛 발급분의 `expiresAt` 가 어떻게 되는지 침묵.
- **엔드유저 시나리오**: 사용자가 30 일짜리로 받은 쿠폰을 가지고 있는데 만료일이 단축되어 표시됨. 또는 그 반대로 사용자에게 유리하게 늘어남.
- **CS 영향**: "만료일이 짧아졌다"·"분명 N 일이라고 했는데 X 일밖에 안 남았다" — 보유 쿠폰 인벤토리 전체 영향.
- **권장 액션**: FR-4 에 "이미 발급된 Coupon 의 `expiresAt` 는 발급 시점에 결정된 값이며 후속 운영자 동작으로 단축되지 않는다"를 명시. (현행 coupon.ts:464, 743 의 `calculateCouponExpiresAt` 는 발급 시점 산정이라 안전해 보이나, 운영자가 옛 spec 의 `usableDays` 를 변경하고 그 다음 어떤 경로(예: 만료일 재계산 batch)가 이를 다시 적용할 가능성 점검 필요.)

### F-8: CS 가 사용할 조회/해결 도구가 정의되어 있지 않음
- **현상/위반 지점**: 전 문서. Success Metric 에 "CS 문의 0 건" 목표만 있고, 문의 발생 시 CS 가 어떤 화면/쿼리/툴로 (사용자 쿠폰 상태 / 옛 spec vs 새 spec / 카카오 주문번호) 를 확인할 수 있는지가 없음.
- **엔드유저 시나리오**: 사용자가 "왜 두 장이지 / 왜 다른 스킬에 적용되지" CS 문의 → CS 담당이 운영 DB·어드민에 직접 SQL 을 쳐야만 답변 가능.
- **CS 영향**: 핫픽스 후에도 문의 0 건이 아닌 경우, CS 의 1차 응대 SLA 가 운영 DB 접근에 묶임. CS 가 자체적으로 해결 못 함 → 모든 케이스 개발팀 에스컬레이션.
- **권장 액션**: FR-2 에 "CS 가 사용자 단위로 (보유 쿠폰 / 발급 시점 spec snapshot / 어떤 카카오 주문에서 왔는지) 를 확인할 수 있는 어드민 화면 또는 진단 SQL 3 종 (CS 용)"을 추가. 본 핫픽스 범위가 아니면 Non-Goals 에 명시하고 별 트랙 TODO 발행.

### F-9: 두 개 쿠폰을 보유한 사용자에 대한 사용자 안내·복구 절차 부재
- **현상/위반 지점**: US-02 + 정합화(US-09) + Success Metric. P0-B 의 사후 결과를 만난 사용자가 핫픽스 배포 시점에 두 장의 쿠폰을 보유 중일 때, 자동 회수·자동 안내·자동 만료 어느 것도 정의되어 있지 않음. "발급분 보존"이라는 원칙은 동일 사용자에 대한 1+1 의 정당성을 보장하지 못함 (한 사람이 한 선물에 대해 두 쿠폰 받는 것은 정합성 깨짐 그 자체).
- **엔드유저 시나리오**: 사용자는 두 쿠폰을 다 쓸 수 있다고 기대했는데 운영자가 한 장을 회수하면 손실 인식.
- **CS 영향**: "쿠폰 한 장이 사라졌어요" 또는 회수하지 않으면 "왜 두 장 다 쓸 수 있는지 / 사용자 측면에서 이게 정당한가" 의 후속 정책 논쟁.
- **권장 액션**: 결정 요청 D-X 추가 — "이미 발급된 중복 쿠폰에 대한 처리: 그대로 둠 / 회수 + 사용자 안내 / 한 장 만료 처리". 정책에 따라 사용자 안내 (in-app notice·푸시) 텍스트와 운영 절차를 함께 정의.

### F-10: 카카오 선물하기 측 푸시·앱 내 카드 표기 ↔ 우리 spec name 갱신의 정합성
- **현상/위반 지점**: F-2 의 확장. 카카오 선물 수신 시 발송되는 알림톡/푸시·앱 내 "선물 도착" 카드는 발신 시점의 상품명을 캡처해 보낸다고 가정. 그러나 사용자가 그 알림을 누르고 들어왔을 때 우리 쪽 화면이 보여주는 쿠폰 이름은 spec 의 현재 name. 운영자가 이름을 바꾼 후 들어온 사용자는 "카카오는 A 인데 헬로우봇은 B" 의 불일치를 본다.
- **엔드유저 시나리오**: 알림톡 "체질테스트 무료체험권" → 앱 진입 → 쿠폰함 "체질테스트 50% 할인권" (운영자가 같은 spec name 을 바꾼 경우).
- **CS 영향**: 채널톡/CS 에 "받은 선물이 다른데?" 류 문의 + 카카오톡 캡처 첨부 → 우리 측이 변명 불가.
- **권장 액션**: F-2 권장 액션과 동일하게 발급 시점 name 의 freeze 가 강제되면 본 케이스도 해결. 별도 FR 로 "외부 채널 표기 ↔ 우리 쪽 표기 일관성"을 명시하거나, /architect 단계에서 freeze 컬럼 결정을 받아들이는 형태.

### F-11: 정합화 SQL 적용 중 라이브 트래픽과의 경쟁
- **현상/위반 지점**: FR-3.3 / US-09. 정합화는 운영 DB 에 직접 SQL 을 친다 (D-4). 그 시점에 새 쿠폰 발급/사용 트래픽이 흐를 수 있는데, "발급 진행 중 spec 동결" 같은 경쟁 조건이 사용자에게 어떤 모습으로 보일지가 spec out 되어 있지 않음.
- **엔드유저 시나리오**: 사용자가 카카오에서 막 선물 받은 시점에 우리가 그 spec 의 `issueEndDate=now()` 를 적용 → 발급 실패 → 카카오에서는 발송 완료 / 우리에서는 미발급.
- **CS 영향**: "선물이 도착했다고 했는데 쿠폰함에 없다" — 가장 명확한 사용자 불만 패턴.
- **권장 액션**: FR-3.3 acceptance 에 "정합화는 한산 시간대(예: 03~05 KST) 에 실행" + "발급 실패 시 retry/recovery 절차"를 명시. 또는 정합화 적용 직전에 카카오 측에 일시 정지 협조를 받을지 여부 결정.

### F-12: 사용자가 카카오에서 받은 쿠폰을 등록하기 전(발급 전) 운영자가 옛 상품을 비활성화한 경우
- **현상/위반 지점**: FR-1.1·1.3. 운영자가 상품을 비활성화하면 옛 spec 은 `issueEndDate=now()` 로 동결 → 신규 발급 차단. 그런데 카카오 선물 수신 ↔ 헬로우봇 측 쿠폰 등록은 사용자가 앱을 켜는 시점까지 시차가 있을 수 있음 (carry-over). 등록 시도 시 신규 발급 차단 상태라 실패.
- **엔드유저 시나리오**: 사용자가 카카오에서 선물 수신 → 며칠 후 헬로우봇 앱에서 등록 시도 → "사용 불가" 또는 에러. "이미 받은 선물인데 왜?"
- **CS 영향**: "카카오에서 분명 받았는데 헬로우봇에서 사용 불가" — 카카오 ↔ 헬로우봇 어느 쪽 책임인지 불분명, 양사 CS 핑퐁.
- **권장 액션**: FR 에 "신규 발급 차단(`issueEndDate`) 의 의미는 '카카오 측 신규 선물 등록 차단'이지 '사용자가 이미 받은 미등록 선물의 헬로우봇 측 발급 차단'은 아니다"를 분리. 또는 운영자에게 "비활성화 전 미등록 쿠폰 잔량 안내" UI 를 요구사항으로 추가.

## 인정할 만한 부분

- FR-4.1 의 "retroactive 변경 금지" 원칙 자체는 사용자 보호의 코어 의도를 정확히 짚었다. 본 리뷰에서 지적한 누락 항목들은 이 원칙을 **표시 일관성·외부 시스템 정합성·기존 위반 데이터 사후 처리**까지 확장하는 작업이지, 방향성 자체를 반박하지 않는다.
