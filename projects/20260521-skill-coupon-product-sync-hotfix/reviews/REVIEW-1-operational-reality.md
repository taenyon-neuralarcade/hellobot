# REVIEW-1: 운영 현실성 비판 리뷰

**리뷰어 렌즈**: 운영 현실성 (운영자 멘탈 모델·실수 가능성·SOP 누락·UI 표현력)
**검토 대상**: requirements.md / user-stories.md / 1pager.md (2026-05-21 작성)
**작성일**: 2026-05-21
**전체 평가**: **위험도 H** — 정합성 정책(특히 D-3 suffix)이 운영자 멘탈 모델과 정면 충돌하고, 본 핫픽스 후 어떻게 일해야 하는지에 대한 운영 SOP가 통째로 누락됨. 코드만 배포되면 운영자가 "수정 버튼"을 그대로 in-place 수정으로 믿고 사용해 동일 결함 패턴(중복 spec·이름 충돌 사고)을 재발시킬 가능성이 매우 높음.

## 요약

핫픽스 코드 자체의 의도(옛 spec 동결 + 새 spec 생성)는 **운영자의 일반적 멘탈 모델 — "수정 = in-place 변경" — 과 정반대**이지만 본 요구사항에는 이 갭을 메우는 운영자 교육·UI 표현·SOP 가 전혀 정의되지 않았다. 또한 진단 SQL 의 실행 주체·주기·결과 대응 절차, suffix 정책 시 productCode 작명 규칙, 복수 운영자 동시 수정 / 운영자 실수 복구 / "잠깐 비활성화 후 재활성화" 같은 정상 운영 흐름에 대한 정의가 모두 비어 있다. 가장 큰 두 문제는 **F-1(in-place 수정 멘탈 모델 충돌)** 과 **F-3(suffix 정책 운영 규칙 부재)**.

## 발견한 문제

### F-1: "수정" 의 의미가 운영자 멘탈 모델과 정반대인데 UI·가이드로 보정되지 않음

- **현상/위반 지점**: FR-1.1 (옛 spec 동결 + 새 spec 생성). user-stories US-01.
- **운영 시 발생할 시나리오**: 운영자가 어드민에서 상품 A 의 `fixedMenuSeq` 를 B 로 "수정" 한다. 그의 멘탈 모델은 "오타 고친 것처럼 in-place 로 바뀐다" 이다. 실제 시스템은 옛 CouponSpec 을 동결하고 새 spec 을 만들어 교체한다. 운영자는 "수정 성공" notice 만 본다.
  - 이후 운영자가 "어, B 가 아니라 C 였네" 하고 다시 수정한다 → 또 새 spec 이 생긴다. 동결된 spec 이 2 개 누적.
  - 운영자는 "잘못 눌렀으니 되돌리자" 하고 B → A 로 복원 수정 → 또 새 spec 이 생긴다. **원래 A 활성 spec 으로 복원되지 않음**. 발급된 쿠폰의 적용 대상은 의도와 다르게 분기.
- **영향**: 운영자가 동일 상품을 N 번 수정할 때마다 동결 spec 이 N 개씩 누적. 운영자 본인은 "수정한 거니까 한 개일 것" 으로 인식. 추후 진단 SQL 에서 orphan/동결 spec 이 대량 잡혀도 그게 정상인지 비정상인지 판별 불가. CS 대응 시 "이 쿠폰이 어느 시점 spec 인지" 추적 어려움.
- **권장 액션**: 다음을 요구사항에 정의해야 한다:
  - 운영자에게 "수정 시 옛 spec 은 동결되고 새 spec 이 생성된다" 는 사실을 **수정 액션 시점에 confirm 다이얼로그·경고 배너로 노출**할지 여부 (= AdminJS UI 의 어디에 어떻게 표시).
  - 운영자가 실수로 잘못 수정했을 때 **롤백·복원 절차** (어떤 동작이 옛 활성 spec 을 되살리는가 / 되살릴 수 없다면 그 사실을 운영자에게 어떻게 알리는가).
  - "이 변경은 신규 발급에만 반영된다, 기존 발급분에는 적용되지 않는다" 라는 문장을 어떤 형태로든 어드민 UI 에 명시할지.

### F-2: 본 핫픽스 후 운영자 SOP / 가이드 문서가 산출물에 없음

- **현상/위반 지점**: tasks.md / requirements.md 어디에도 "운영자에게 변경된 동작을 어떻게 알릴지" 가 과업으로 정의되지 않음. 1pager Non-Goals 에도 SOP 가 명시적으로 제외되지 않음 (즉, 깜빡한 영역).
- **운영 시 발생할 시나리오**: 핫픽스 배포 직후 운영자가 동일 동작을 그대로 한다 → "어 이거 왜 spec 이 두 개야?" → 운영자가 직접 spec 을 어드민에서 지우려고 시도 → ISS-071 P2-I (어드민 CouponSpec 직접 수정) 와 정면 충돌. 본 핫픽스가 Non-Goal 로 정의한 영역을 운영자가 어쩔 수 없이 건드리게 됨.
- **영향**: 코드는 고쳐졌는데 운영자 행동은 그대로라 결과적으로 같은 결함을 다른 경로로 재현. 진단 SQL 이 0 row 가 되더라도 운영자 행동으로 다시 위반 데이터 생산.
- **권장 액션**: 다음을 요구사항·과업으로 추가 정의해야 한다:
  - 핫픽스 배포 전·후 운영자에게 전달할 **공지문 1 페이지** (변경된 동작 / 옛 동작과의 차이 / 하지 말아야 할 행위 / 추천 흐름).
  - 본 핫픽스 후에도 운영자가 CouponSpec 을 직접 수정하면 위반이 발생함을 운영자에게 알리는 절차 (현재 P2 로 미루는 결정과 운영자 행동 가이드는 별개 문제).
  - 운영자 교육·확인 책임자가 누구인지 (PM / CS Lead / 운영 admin owner).

### F-3: D-3 suffix 정책 — productCode 작명 규칙·운영자 권한이 정의되지 않음

- **현상/위반 지점**: requirements.md D-3 권장안 `${productName} 이용권 #${productCode}`. 1차 권장으로 채택된 것처럼 보이지만 productCode 가 무엇이고 누가 어떻게 정하는지가 비어 있음.
- **운영 시 발생할 시나리오**:
  - 운영자가 새 상품을 등록할 때 productCode 를 직접 입력하는지, 시스템이 자동 생성하는지 정의 없음.
  - 운영자가 입력한다면 — 두 운영자가 같은 productCode 를 우연히 입력하면 또 다른 UNIQUE 충돌. 작명 규칙(prefix·길이·중복 검사·예약어) 없음.
  - 자동 생성이라면 — 운영자가 어드민 화면에서 productCode 를 어떻게 인지하나, 변경 가능한가, CS 가 "이 쿠폰은 어떤 코드의 어떤 버전인가" 를 추적 가능한가.
  - 운영자가 동일 상품을 2 회 수정 → suffix #ABC, #ABC 두 번 — 같은 productCode 면 UNIQUE 또 충돌. (productCode 가 상품 단위라면 동일 상품의 수정마다 spec 이 생기는 FR-1.1 과 정면 충돌).
- **영향**: D-3 의 1차 권장안 자체가 **운영적으로 미완성**. 이 상태로 /architect 단계로 넘어가면 아키텍트가 임의 규칙을 정하게 됨 → 운영자가 모르는 규칙으로 spec.name 이 결정 → 운영자 멘탈 모델 또 깨짐.
- **권장 액션**: D-3 채택 전 다음을 정의해야 한다:
  - productCode 의 소유자 (운영자 입력 vs 시스템 자동) 와 변경 가능성.
  - 동일 productCode 의 spec 이 여러 번 생성될 때 spec.name UNIQUE 를 어떻게 회피하는지 (예: `#{productCode}-{seq}` / `#{productCode}-{createdAt}`).
  - 운영자가 spec.name 을 어드민 화면에서 어떻게 보는지 (productName 으로만 보면 N 개 중 무엇이 활성인지 식별 불가).
  - 한국어/일본어 양 도메인에서 동일 규칙 적용 가능한지 (jp 어드민도 본 핫픽스 영향 받는지가 본문에 없음 — 별 항목 F-9 참조).

### F-4: 복수 운영자 동시 수정·감사 로그 가정 부재

- **현상/위반 지점**: requirements.md 전체에서 "운영자 1 명" 만 암묵 가정. NFR / FR 모두 race condition 을 NFR-1 한 줄로만 다룸.
- **운영 시 발생할 시나리오**: 운영자 X 가 상품 A 를 B 로 수정하는 사이에 운영자 Y 가 같은 상품을 다시 A 로 되돌리는 수정을 한다. 본 정책에서는 옛 spec 동결 + 새 spec 생성이 두 번 — 동결 spec 2 개 + 활성 spec 1 개 (어느 운영자가 마지막에 commit 했느냐에 따라). 둘 다 "수정 성공" notice 를 받음. 누구의 의도가 반영됐는지 어드민에서 알 길이 없음.
- **영향**: 정합성은 깨지지 않을 수 있어도(활성 spec 은 1 개) 운영자 의도 추적이 불가능. CS 폭탄·내부 인수인계 사고의 원인.
- **권장 액션**: 다음을 정의해야 한다:
  - AdminJS 의 record-level 동시성 제어 (optimistic lock 컬럼 / updatedAt 비교 거부) 를 본 핫픽스 범위에 포함할지 여부.
  - 운영자별 변경 이력이 어디에 남는지 (현재 audit log 가 있는지 본 문서에서 확인 불가). 없다면 **본 핫픽스 후속 별 트랙으로 명시 등록**.
  - "누가·언제·무엇을 수정했는지" 를 CS 가 추적해야 하는 케이스가 본 핫픽스 후 늘어남을 명시 (옛 spec 동결 기록은 추적 가능해야 함).

### F-5: "잠깐 비활성화 → 점검 → 재활성화" 정상 흐름이 본 정책에서 어떻게 보호되는지 미정의

- **현상/위반 지점**: FR-1.3 (delete = soft = issueEndDate). user-stories US-02 의 "비활성화" 가 무엇을 의미하는지 정의되지 않음.
- **운영 시 발생할 시나리오**: 운영자가 "스킬 점검 동안 상품 발급 잠깐 막기" 의도로 상품을 비활성화(혹은 D-1 의 soft delete) → 점검 후 다시 활성화하려고 동일 스킬에 새 상품 등록 → FR-1.2 에 의해 옛 spec 도 동결되고 새 spec 생성. 운영자가 의도한 것은 "잠깐 멈춤 → 재개" 였는데 시스템은 "별개의 새 상품" 으로 본다. 결과: 동결 spec + 새 spec 2 개, 옛 발급 쿠폰의 spec 과 새 발급 쿠폰의 spec 이 분리 → CS 가 "왜 같은 스킬인데 2 개 쿠폰을 받았다" 같은 문의에 시달림.
- **영향**: 일상적인 운영 흐름이 핫픽스로 **막혀버리거나, 가능하긴 한데 의도와 어긋남**. 운영자가 우회 패턴 (예: 스킬을 살짝 다른 값으로 바꿔놓기) 을 발명할 수 있음.
- **권장 액션**: 다음을 정의해야 한다:
  - "활성/비활성 토글" 이 어드민에 있는지, 있다면 본 핫픽스에서 어떻게 동작해야 하는지 (활성 상태에서만 신규 발급 / 비활성 시 발급 차단 / 재활성 시 새 spec 만들지 않음).
  - "재등록" 과 "재활성화" 를 운영자가 구분할 방법 (UI 라벨·확인 다이얼로그).
  - 점검 시나리오에 대한 권장 운영 SOP (어떤 동작을 해야 하는가).

### F-6: 운영자 실수 복구 (잘못 수정 / 잘못 삭제) 시나리오 부재

- **현상/위반 지점**: 요구사항·스토리 어디에도 "실수 복구" 항목 없음.
- **운영 시 발생할 시나리오**:
  - 운영자가 fixedMenuSeq 를 잘못 입력 (B 가 아니라 C). 옛 spec 동결 + C 매칭 새 spec 생성. 운영자가 알아채고 다시 B 로 수정 → C 매칭 spec 동결 + 새 B spec 생성. 그 사이 발급된 쿠폰은 C 에 묶임.
  - 더 큰 문제: 운영자가 "되돌리고 싶다 — 원래 활성 spec 으로 복원" 을 어드민에서 할 수 없음. FR-1.1 정책상 새 spec 만 만들어짐.
  - 운영자가 잘못 삭제했을 때 — soft 라 발급분은 보존되지만 신규 발급 재개 방법이 정의 없음 (다시 등록 = 또 새 spec, 옛 동결 spec 은 영영 동결).
- **영향**: 운영자가 실수했을 때 정확하게 이전 상태로 되돌릴 수 없다는 사실이 본 핫픽스의 명시 트레이드오프인지, 단순히 누락된 항목인지 불분명.
- **권장 액션**: 다음 중 결정 필요:
  - "옛 동결 spec 을 재활성화" 어드민 액션을 본 핫픽스 범위에 포함할지 (포함 시 새 FR 필요, 미포함 시 운영자에게 명시).
  - 운영자가 실수했음을 알아챘을 때의 권장 절차 (CS 통해 발급 쿠폰 처리 / 신규 발급 어떻게 보장).
  - 어드민 UI 에서 "이전 활성 spec" 을 운영자가 볼 수 있는지 (보지 못한다면 복원이 불가능함을 명시).

### F-7: AdminJS UI 가 실제로 본 요구사항을 표현할 수 있는지 검증 부재

- **현상/위반 지점**: FR-2.2 (부분 실패 → 명백한 error notice, success 와 시각적으로 구분). FR-2.3 (단일 트랜잭션 권장). AdminJS 의 UI 표현력 한계가 검증된 흔적 없음.
- **운영 시 발생할 시나리오**: AdminJS 의 기본 notice 는 toast 형태 (몇 초 후 사라짐) 인 경우가 많고 success / error 색 구분만 있을 뿐 운영자가 "성공인지 실패인지" 를 사후 확인하기 어렵다. 본 요구사항이 요구하는 "후속 액션 잘못 취하지 않도록" 의 강도를 toast 로는 못 맞춤. 운영자가 다음 화면으로 넘어간 뒤에는 실패 사실 자체가 화면에 남지 않음.
- **영향**: FR-2.2 가 코드상 구현됐다고 해도 UI 가 받쳐주지 않으면 결과적으로 운영자가 success 로 착각하는 ISS-071 P1-F·H 결함이 그대로 재현 가능.
- **권장 액션**:
  - AdminJS 가 어떤 형태의 error 표현을 지원하는지 (toast / banner / modal / redirect-to-error-page) 를 /architect 전에 확인.
  - "error 가 발생하면 화면 전환 없이 같은 폼에 머무른다" 같은 구체 동작을 FR 로 명시.
  - 운영자가 사후에 "내가 한 마지막 동작이 실패했었는지" 를 확인할 수 있는 경로 (history view / audit log) 가 본 핫픽스에 포함되는지 결정.

### F-8: 진단 SQL 의 실행 주체·주기·결과 대응이 단발성 점검으로만 정의됨

- **현상/위반 지점**: FR-3.1~3.3, NFR-4 ("수동 또는 cron — P1 단계에서 결정"). user-stories US-08 / US-09.
- **운영 시 발생할 시나리오**:
  - 진단 SQL 이 한 번 0 row 가 나왔다고 해서 영구히 0 row 가 보장되지 않음. 본 핫픽스 후에도 운영자가 ISS-071 P2-I (어드민 직접 수정) 같은 경로로 위반을 다시 만들 수 있음.
  - 정기 점검 책임자가 누구인지 정의 없음 (PM? CS? /dev-server? 데이터?). 1 주 후엔 아무도 진단을 안 하게 될 가능성 매우 높음.
  - 위반 발견 시 누가 어떤 권한으로 보정하는지도 D-4 의 "일회성 SQL" 권장안 뿐 — 일회성을 누가 작성·승인·실행하는지 절차 없음.
- **영향**: 1pager Success Metric "배포 후 0 row 유지" 가 실제로 측정되지 않을 위험. 위반이 누적되어도 알아챌 메커니즘 없음.
- **권장 액션**:
  - 진단 SQL 의 **정기 실행 주체와 주기** (예: 데이터 팀 weekly / CS Lead daily / cron job 결과 Slack 알림) 를 NFR-4 의 결정 항목으로 명시.
  - 위반 발견 시 alert 대응 SOP — 누가 보고 → 누가 보정안 작성 → 누가 승인 → 누가 적용 → 누가 결과 확인.
  - 본 핫픽스 배포 후 위반이 재발하면 그 자체가 ISS-071 의 신규 변형이 있다는 신호 — 이를 감지·티켓화하는 절차.

### F-9: 한국어/일본어 양 도메인 운영자 차이가 검토되지 않음

- **현상/위반 지점**: 영향 범위 매트릭스 (요구사항 §2) 가 리포만 다루고 도메인(한국·일본)을 구분하지 않음. AdminJS·CoopMarketingProduct 가 일본 운영자에게도 동일하게 노출되는지 본 문서에서 확인 불가.
- **운영 시 발생할 시나리오**: 일본 운영자가 동일 어드민을 사용한다면 본 핫픽스의 모든 변경(특히 suffix·notice·validation) 이 일본어 환경에서도 동일하게 동작해야 함. 일본 운영자에게 한국어 notice 가 노출되거나, 일본어 라벨이 누락되어 운영자가 다른 결과를 만들 위험.
- **영향**: 다국어 도메인 (CLAUDE.md §다국어 — jp.hellobot.co) 의 운영자가 본 핫픽스의 사각지대.
- **권장 액션**:
  - 본 핫픽스 영향 영역의 **한국어/일본어 어드민 노출 여부**를 영향 범위 매트릭스에 명시.
  - 다국어 어드민이 노출된다면 notice / error 메시지의 i18n 처리 범위.
  - 일본 운영자에게도 동일 SOP·공지 적용할지.

### F-10: "운영자가 본 핫픽스 후 어떻게 행동을 바꿔야 하는지" 가 검증 기준에 없음

- **현상/위반 지점**: 요구사항 §7 검증 기준 — 모두 "코드 / 데이터 / UI 동작" 의 정합성 검증만. **운영자 행동·인지 변경** 의 검증 항목 0 건.
- **운영 시 발생할 시나리오**: 핫픽스가 코드·데이터 측면에서 100% 통과해도 운영자가 행동을 바꾸지 않으면 1pager Success Metric 의 "운영자 보고 0 건" 은 달성 불가능. 운영자가 새 동작을 이해했는지 측정할 방법이 없음.
- **영향**: Success Metric 이 "CS 문의 0 건 / 운영자 보고 0 건" 으로 잡혀 있는데 검증 기준에는 이를 측정·확인할 단계가 없음. 즉, 메트릭이 자동으로 실패할 가능성을 만듦.
- **권장 액션**:
  - 운영자에 대한 사전 공지·교육 완료 여부 (책임자 sign-off) 를 검증 기준에 추가.
  - 배포 후 **2 주차 운영자 인터뷰 또는 어드민 동작 로그 샘플 검토** 를 검증 단계로 추가.
  - "운영자 보고 0 건" 메트릭의 측정 방법 (어디서 보고를 수집하는지) 정의.

### F-11: 진단 SQL 의 보정(D-4 일회성 SQL) 적용 시 발급된 쿠폰 영향이 운영자에게 어떻게 통지되는지 부재

- **현상/위반 지점**: FR-3.3 / US-09 / D-4 — 보정 SQL 의 작성·승인·적용 절차만 다루고, 보정 결과로 발급된 쿠폰의 적용 대상·사용 가능 여부가 어떻게 변하는지 운영자·CS 가 어떻게 인지하는지 미정의.
- **운영 시 발생할 시나리오**: 동일 skillSeqs 매칭 활성 spec 2 개 케이스를 보정하면서 한 쪽을 동결한다. 그 쪽 spec 으로 발급된 미사용 쿠폰은 어떻게 되는가 — US-01 보존 원칙에 따라 사용 가능 유지지만, 운영자·CS 는 "이 쿠폰들은 어느 시점에 발급된 것이고 어느 spec 에 묶여 있는지" 를 추적해야 함. 그 데이터를 누가 정리해서 어디에 남기는지 없음.
- **영향**: 보정 후 CS 가 사용자 문의를 받을 때 "이 사용자의 쿠폰은 보정 영향 받음 / 아님" 을 판별 불가.
- **권장 액션**:
  - 보정 시 **영향받은 쿠폰 리스트 (사용자 ID + 쿠폰 ID + 옛 spec / 새 spec) 를 산출·보관** 하는 단계를 FR-3.3 에 추가.
  - 그 리스트를 CS 에 사전 공유할지 결정.
  - 사용자에게 통지가 필요한 케이스가 있는지 검토 (예: spec 동결로 사용 정책이 바뀐 경우 — 본 정책은 바뀌지 않는다고 했지만 검증 필요).

### F-12: tasks.md 의 T-10 QA 시나리오가 운영자 시각이 아니라 개발자 시각으로 정의됨

- **현상/위반 지점**: 요구사항 §7 의 "QA 시나리오 (tasks.md T-10)" 가 "UNIQUE 충돌·fixedMenuSeq 누락·부분 실패 케이스 재현" 같은 결함 재현 시나리오만 다룸.
- **운영 시 발생할 시나리오**: 위 QA 가 통과해도 운영자가 일상적으로 수행하는 "정상 흐름" 시나리오 (잠깐 비활성화 → 재활성화 / 오타 수정 / 다국어 환경 등) 가 통과하는지는 검증되지 않음. 결함 재현만 검증하는 QA 는 "결함은 안 나지만 정상 흐름이 부수적으로 망가지는" 케이스를 놓침.
- **영향**: F-5·F-6 의 시나리오가 QA 에서 잡히지 않을 가능성 매우 높음.
- **권장 액션**:
  - QA 시나리오에 **운영자 정상 흐름** 시나리오 명시 추가 (정상 등록 / 정상 수정 / 정상 삭제 / 잠깐 비활성화 후 재활성화 / 동일 상품 2 회 연속 수정 / 실수 후 복원 시도).
  - 운영자 1 명이 작성한 시나리오를 QA 단계에서 검토.

## 인정할 만한 부분 (간단히)

- 운영자·엔드유저·CS 세 페르소나를 명시적으로 분리하고 US-01 (발급 쿠폰 보호) 을 P0 로 둔 점.
- FR-1.1 의 "in-place 갱신하지 **않는다**" 라는 부정 명세를 명시적으로 박은 점 — 옛 결함 패턴이 재발하지 않도록 정책 의도가 분명함.
- D-1~D-5 결정 보류 항목과 1차 권장안을 분리 표기하여 /architect 단계 전에 사용자 합의 필요한 항목을 명확히 한 점.
