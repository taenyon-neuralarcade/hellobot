# REVIEW-3: 요구사항 명확성·완결성 비판 리뷰

**리뷰어 렌즈**: 요구사항 명확성·완결성 (모호·누락·충돌·검증 가능성·매핑 누락)
**검토 대상**: requirements.md / user-stories.md / 1pager.md (2026-05-21 작성)
**작성일**: 2026-05-21
**전체 평가**: **위험도 H** — FR ↔ ISS-071 매핑에 실질적 누락(E, H, I)이 있고, FR-1.5 ↔ FR-4.1 의 표면적 충돌이 D-3 권장안에서 해소되지 않으며, "동결", "atomic", "단일 활성 spec", "부분 정합성" 등 핵심 용어가 정의 없이 사용된다. /architect 진입 전 보강 필수.

## 요약

요구사항 문서는 구조(FR/NFR/C/의존)와 사용 스토리 매핑 골격은 잡혀 있지만, **결함 9 건 → FR 14 건 매핑의 누락·약화·임의 통합**, **FR 간 정합성 미해소**, **검증 가능성이 떨어지는 용어·NFR**, **D-1~D-5 권장안의 근거·대안 비교 부재**가 다수 발견된다. ISS-071 결함 E(skill→heart) 와 H(부분 실패) 는 FR 본문에서 사실상 약화되었고, F(UNIQUE 충돌 catch + 성공 표시) 는 FR-2.2 / FR-1.5 양쪽에 분산되어 어디서 책임지는지 불분명하다. 동시성·격리·중복 클릭 시나리오는 본문에 한 줄도 없다.

## 발견한 문제

### F-1: FR-1.5 ↔ FR-4.1 의 표면적 충돌 — D-3 권장안(suffix)이 이를 해소하지 않음

- **현상/위반 지점**:
  - FR-1.5 (req §3) "상품명 변경으로 `CouponSpec.name` UNIQUE 충돌이 발생하면 D-3 정책에 따라 처리(atomic reject 또는 suffix 정책)"
  - FR-4.1 (req §3) "이미 발급된 `Coupon` ... 의 적용 가능 대상이 운영자 측 CRUD 동작으로 인해 retroactively 변경되지 않는다"
  - D-3 권장안 (req §8) "**suffix** (`${productName} 이용권 #${productCode}` 형태) ... 단, 이미 발급된 spec 의 name 은 변경하지 않음(US-01 정합)"
  - US-02 AC (us §1순위) "동일 상품명이라 `CouponSpec.name` UNIQUE 충돌이 잠재하더라도 등록은 성공해야 한다(이름 정책으로 회피)"
- **문제 유형**: 충돌 + 정의 미흡
- **근거**:
  1. FR-1.5 는 "상품명 변경 시 충돌" 시나리오인데, D-3 권장안의 단서 "이미 발급된 spec 의 name 은 변경하지 않음"이 이 FR 의 핵심 동작인지 예외인지 불명. 옛 spec 이 동결만 되고 name 은 그대로면 새 spec 만 suffix 를 받는 정책인가? 옛 spec 의 name 은 그대로면 UNIQUE 가 정말 회피되는가는 자명하지 않다 (옛 spec 이 이미 `X 이용권` 점유 → 새 상품도 `X 이용권` 으로 바꾸려 함 → 충돌. suffix `#productCode` 가 new product 측이라면 회피 가능하나 명시 안 됨).
  2. US-02 의 "이름 정책으로 회피" 는 P0-B(재등록) 시나리오인데, FR-1.5 는 "상품명 변경"(P1-F) 으로 정의됨. 두 시나리오에서 suffix 가 같은 규칙으로 동작하는지 명시 부재.
  3. FR-4.1 의 "retroactively 변경되지 않음"은 발급된 쿠폰의 적용 대상(condition) 의 불변이지, spec.name 의 불변은 아니다. 그러나 D-3 단서는 "이미 발급된 spec 의 name 은 변경하지 않음"을 US-01 정합이라 적시 — US-01 은 적용 대상(skillSeqs) 의 불변을 다루는데, 여기서 name 까지 끌어들이는 근거가 약하다.
- **권장 액션**: FR-1.5 본문에 "옛 활성 spec 은 동결되며 그 spec.name 은 보존된다. 새 spec 생성 시 충돌 위험이 있는 경우 신규 spec.name 에 suffix 를 부착한다 (예: ...)" 로 명시. 또한 "옛 spec.name 의 보존 이유" 를 FR-4 (불변 조건) 로 끌어올리고 (FR-4.3) "발급된 CouponSpec.name 은 retroactive 변경되지 않는다" 를 추가. US-01 정합이 아니라 별 조건.

### F-2: 결함 E (skill→heart 전환) 가 FR-1.4 안에 임의 통합되어 약화됨

- **현상/위반 지점**:
  - ISS-071 결함 E "productType skill→heart 수정 → CouponSpec/Condition orphan, 이미 발급된 쿠폰 사용 가능, productCode 매핑 단절" (심각도 P1)
  - FR-1.4 (req §3) "`productType` 의 heart↔skill 전환은 명시적 정책에 따른다(D-2 결정). 허용 시 적절한 spec 생성/동결이 수반되어야 하고, 비허용 시 validation error 로 reject."
  - US-04 AC — heart→skill / skill→heart 두 케이스를 별도 AC 로 다루나, "OR 이행 자체를 막아도 됨" 으로 사실상 양쪽 모두 reject 라는 D-2 권장안 하나로 수렴
- **문제 유형**: 매핑 누락(약화) + 모호
- **근거**: 결함 E 와 D 는 방향이 반대(D=heart→skill silent skip, E=skill→heart orphan) 이며 둘이 만드는 데이터 상태도 다른데, FR-1.4 한 줄이 둘을 묶고 D-2 가 "비허용" 으로 가버리면 **이미 D 와 E 가 발생해 운영 DB 에 들어가 있는 데이터**(좀비/orphan) 의 처리가 어디에서 다뤄지는지 불명확하다. 결함 E 의 "productCode 매핑 단절" 도 FR 어디에도 명시 없음.
- **권장 액션**: FR-1.4 를 FR-1.4-a(heart→skill) / FR-1.4-b(skill→heart) 로 분리하거나, FR-1.4 본문에 "D-2 가 비허용이면 이미 발생한 좀비/orphan 의 처리는 FR-3.3 정합화 범위" 라고 명시. productCode 매핑 단절도 진단 SQL §3 (orphan) 에 잡히는지 명시.

### F-3: 결함 H (부분 실패가 success 로 표시) 가 FR-2.2/FR-2.3 에 분산되어 책임 경계 불명

- **현상/위반 지점**:
  - ISS-071 결함 H "afterNew/afterEdit 트랜잭션 부분 실패 → 상품 INSERT/UPDATE 와 CouponSpec 트랜잭션 분리, 후자 실패 시 상품은 commit, notice 만 출력" (심각도 P1)
  - FR-2.2 (req §3) "부분 실패는 AdminJS UI 상 명백한 error notice 로 표시되어야 하고, success 와 시각적으로 구분되어야 한다."
  - FR-2.3 (req §3) "가능하면 ... 단일 트랜잭션으로 묶는다. 분리해야 한다면 실패 시 명확한 보상/통보 로직을 정의한다."
- **문제 유형**: 모호 + 비검증 가능 (책임 경계 불명)
- **근거**:
  1. FR-2.3 "가능하면" 은 무엇이 가능 / 불가능 조건인지 정의 없음. AdminJS 훅 컨텍스트(`beforeSave/afterEdit`)에서 트랜잭션 통합이 가능한지 여부는 /architect 단계의 결정 사항인데, 요구사항 자체가 "가능하면" 으로 빠져나가면 어떤 결과가 적합인지 검증 불가.
  2. FR-2.2 의 "명백한 error notice" 의 "명백" 의 객관적 기준 부재 (색상? 배치? 텍스트? 페이지 차단?). 검증 가능한 AC 없음.
  3. FR-2.3 의 "보상/통보 로직" 이 어디에서 정의되는지 본 문서 안에 없음.
- **권장 액션**: FR-2.3 을 "단일 트랜잭션으로 묶는다 (불가 시 /architect 가 그 이유를 architecture.md 에 명시하고 보상 로직을 본 문서 FR-2.4 로 환류)" 로 강화. FR-2.2 에 "구체 기준: success 와 다른 색상 배너 + '저장 실패' 라는 표제어 + 후속 액션 제안 (재시도 / 운영 통보)" 등 객관 기준 추가.

### F-4: 결함 F (UNIQUE 충돌 catch + 성공 표시) 가 FR-1.5 와 FR-2.2 양쪽에 분산되어 어느 쪽이 SSOT 인지 불명

- **현상/위반 지점**:
  - ISS-071 결함 F: "`CouponSpec.name` UNIQUE 충돌 시 catch → notice 만 띄우고 상품만 변경. 운영자는 success 로 착각"
  - FR-1.5 = UNIQUE 충돌 처리 정책
  - FR-2.2 = 부분 실패 시 error notice
- **문제 유형**: 매핑 중복 + 책임 경계 불명
- **근거**: 결함 F 의 동작은 (1) UNIQUE 충돌 처리 (2) 부분 commit 가시화 두 측면. 어느 쪽 FR 이 결함 F 의 SSOT 인지 명시 안 됨. D-3 권장안(suffix)이 선택되면 충돌이 사라지므로 (2) 만 남는데, atomic reject 가 선택되면 둘 다 적용. FR 간 의존 관계가 의존 그래프 §6 에 누락됨.
- **권장 액션**: FR-1.5 와 FR-2.2 에 cross-reference 추가. "결함 F 는 1차로 FR-1.5 가 차단 → 차단 실패 시 FR-2.2 가 가시화" 의 직렬 관계 명시. 의존 그래프 §6 에 FR-1.5 → FR-2.2 화살표 추가.

### F-5: 결함 I (CouponSpec/Condition 직접 수정) 의 Non-Goals 처리 — 본 핫픽스 후 진단 SQL 에서 잡힐 위험 명시 안 됨

- **현상/위반 지점**:
  - 1pager Non-Goals "P2-I 본 핫픽스 후 별건"
  - US §Non-User-Stories "CouponSpec/CouponCondition 어드민 직접 수정 차단 — 본 핫픽스 범위 외"
  - FR-3.1 진단 SQL §3 "orphan CouponSpec"
- **문제 유형**: 모호 (스코프 vs 진단 SQL 의 불일치)
- **근거**: 결함 I 를 본 핫픽스 범위 외로 두면, **본 핫픽스 배포 후에도 운영자가 CouponSpec 을 직접 수정하여 다시 정합성이 깨질 수 있다**. 그런데 §7 검증 기준 "진단 SQL 재실행 결과 = 0 row" 는 핫픽스 직후만 보증할 뿐, 운영자 직접 수정 후의 0-row 는 보증 불가. 검증 기준의 시간적 범위가 정의 안 됨.
- **권장 액션**: §7 검증 기준에 "측정 시점: 핫픽스 배포 직후 + 정합화 직후. 이후 운영자 직접 수정으로 인한 위반은 별 트랙(ISS-071-I) 의 범위" 명시. NFR-4 (관찰성) 의 "정기 재실행" 주기를 정의 (예: 주 1 회).

### F-6: "동결(freeze)", "단일 활성 spec", "atomic", "부분 정합성" — 핵심 용어가 본 문서 안에 정의 없음

- **현상/위반 지점**:
  - "동결" — FR-1.1, FR-1.3, US-02, US-03, US-04 다수 등장. "신규 발급 차단 상태" 라는 동의어가 한 번씩 같이 나오지만 별도 정의 섹션 없음.
  - "단일 활성 spec" — FR-1.2 "동일 skillSeqs 에 매칭되는 활성 CouponSpec 은 항상 단 1 개만 존재". "활성" 의 정의가 `issueEndDate > now()` 인지 다른 boolean 컬럼인지 본 문서에 없음 (코드 기반 추정 가능하나 요구사항이 코드 의존).
  - "atomic" — FR-1.5, US-05 AC "atomic 하게 reject" — DB 트랜잭션 격리 수준인지 단순 all-or-nothing 인지 모호.
  - "부분 정합성" — FR-1.5, US-05 — 정의 없음.
- **문제 유형**: 정의 미흡
- **근거**: /architect 단계에서 이 용어들을 그대로 받아 설계하면 해석 차이로 인한 후속 재작업 발생. "활성" 의 정의가 컬럼 1 개로 명시되지 않으면 진단 SQL §1 의 조건절도 흔들린다 (실제 진단 SQL 작성자가 어떤 컬럼을 활성 판정으로 쓸지 결정해야 함).
- **권장 액션**: requirements.md 상단에 §0 "용어 정의" 신설:
  - **활성 CouponSpec**: `issueEndDate IS NULL OR issueEndDate > now()` (또는 실제 컬럼 합의)
  - **동결(freeze)**: `issueEndDate = now()` 로 갱신하여 비활성 전이, 발급된 쿠폰의 사용성과는 독립.
  - **atomic 처리**: 상품 row, CouponSpec, CouponCondition 변경이 단일 DB 트랜잭션 내에서 all-or-nothing.
  - **부분 정합성 (위반)**: 상품 row 와 spec/condition 의 의미적 연결이 한쪽만 commit 된 상태.

### F-7: NFR-5 "성능 영향 미미" — 측정 가능한 합격 기준 부재

- **현상/위반 지점**: NFR-5 "AdminJS 훅에서 추가되는 트랜잭션·쿼리는 운영자 1 명의 CRUD 요청 단위. 엔드유저 쿠폰 발급 경로(`/api/coupon/register`) 의 핫패스에는 영향 없음. — 자명"
- **문제 유형**: 비검증 가능
- **근거**: "자명" 으로 검증 면제됐는데, 진단 SQL §3 (orphan CouponSpec 패턴 매칭) 은 전체 CouponSpec 스캔 가능성이 있고, 동기화 로직이 트랜잭션 확장으로 lock 시간 늘면 동시 운영자 작업에 영향. "엔드유저 핫패스 영향 없음" 만 단정한 상태로 측정 기준 없음.
- **권장 액션**: NFR-5 를 "엔드유저 핫패스(`/api/coupon/register`) p99 latency 가 핫픽스 전후 차이 < N ms (배포 직후 1 일 관측)" + "진단 SQL 실행 시간 < N sec (운영 DB)" 로 객관화. 핫패스 우회는 "AdminJS 훅과 엔드유저 API 가 동일 connection pool 을 공유하지 않는다" 같은 구조적 근거 명시.

### F-8: 동시성·격리 수준·중복 제출에 대한 요구사항이 본문 어디에도 없음

- **현상/위반 지점**: req §3~5 어디에도 동시성·격리·중복 클릭·운영자 2 명 동시 편집 시나리오 없음. NFR-1 "race condition 이 없다" 가 한 줄 언급되나, 무엇을 어떻게 보장하라는 요구사항 부재.
- **문제 유형**: 누락
- **근거**: FR-1.2 "동일 skillSeqs 에 활성 spec 단 1 개" 는 동시 INSERT 두 건이 race 로 통과하면 그대로 위반. UNIQUE 제약 회피 정책(D-3 suffix) 의 결과로 동일 spec.name 이 race 로 통과할 가능성도 있다.
- **권장 액션**: FR-1.6 (또는 NFR-6) 추가:
  - "skill 상품 INSERT/UPDATE 와 연동 spec 생성/동결은 동일 트랜잭션에서 적절한 격리 수준(READ COMMITTED 이상 + 명시적 lock 또는 UNIQUE 제약 의존) 으로 race 가 차단된다"
  - "AdminJS 훅의 동일 운영자 중복 클릭은 가드된다 (이중 spec 생성 방지)"
- 단, 동시성의 실제 위험·설계는 REVIEW-4 영역이므로 여기서는 "요구사항 명시 누락" 까지만.

### F-9: D-1~D-5 권장안의 근거·대안 비교 부재 (트레이드오프 미명시)

- **현상/위반 지점**: req §8 표 컬럼이 `ID | 결정 사항 | 1차 권장안` 3 개뿐. 대안 옵션의 장단점·리스크·비용·운영 부담의 비교 표 없음.
- **문제 유형**: 누락 (의사결정 정보 부족)
- **근거**:
  - D-1 (soft vs hard delete): 권장안 "soft" 의 근거가 "발급분 보존 표준" 한 줄. hard 의 단점이 무엇이고, soft 를 선택했을 때 orphan-like 데이터(연동 상품 없는 spec) 의 진단 SQL §3 의 처리 정책이 어떻게 바뀌는지 명시 없음.
  - D-2 (productType 전환 비허용): 권장안 근거 "단순·안전". 비허용으로 인한 운영자 워크어라운드(신상품 등록 → 옛 상품 삭제) 의 데이터 영향이 D-1 (soft delete) 와 결합하여 어떤 결과를 만드는지 분석 없음.
  - D-3 (suffix): F-1 에서 지적했듯 "이미 발급된 spec 의 name 은 변경하지 않음" 의 함의가 분석 안 됨. atomic reject 의 운영자 UX 단점도 정량 비교 없음.
  - D-4: "점검 결과 = 0 row 면 불필요" 는 합리적이나, 점검 결과 > 0 일 때 "일회성 SQL" 권장의 위험(롤백, 운영 시간, 트랜잭션 범위) 분석 없음.
  - D-5 (풀 스코프): "같은 영역을 두 번 건드리게 됨" 한 줄. 1차 출시 안정화 일정과의 충돌 가능성, P1 6 건 동시 배포의 리뷰 부담 등 비교 없음.
- **권장 액션**: §8 표를 `ID | 결정 사항 | 옵션 A | 옵션 B | 트레이드오프 | 1차 권장 + 근거` 형태로 확장. 각 D 항목에 최소 2 옵션의 영향 비교 1~2 줄.

### F-10: 검증 기준 §7 — 측정 도구·주기·책임자 미정의

- **근거**:

  - "CS 문의 0 건 / 14 일 추적" — CS 시스템(Channel Talk? Zendesk?) 의 어떤 필터로 추적? "쿠폰이 다른 스킬에 적용" 같은 자연어 매칭은 어떻게?

### F-14: US 9 ↔ FR 14 매핑 일관성 — 일부 FR 이 단일 US 만 출처로 표기되나 실제로는 다중 US

### F-15: US-04 AC 의 "OR" — 검증 불가능한 양자택일

- **현상/위반 지점**: US-04 AC heart→skill "T ... 정상 사용 가능. OR 이행 자체를 막아도 됨 (validation error 로 reject). 사용자 결정 필요." / skill→heart "couponSpecSeq 는 그대로 유지(이력 추적용) 또는 null 로 clear (정책 결정 필요)."
- **문제 유형**: 비검증 가능
- **근거**: AC 에 "OR" / "또는" 으로 두 결과가 모두 합격이라고 적시되면 QA 가 어떤 행동을 테스트하고 어떤 결과를 합격으로 판정할지 정할 수 없음. D-2 의 결정 후 AC 를 재작성해야 하는데, 그때까지는 AC 가 비검증 가능.
- **권장 액션**: US-04 AC 를 "D-2 결정 후 본 AC 가 확정된다. 현재는 placeholder." 로 명시하거나, D-2 결정을 /architect 진입 전에 강제. 마찬가지로 skill→heart 의 couponSpecSeq 처리도 별도 D 항목(D-2-a) 으로 분리.

### F-16: NFR-1 "race condition 이 없다" — NFR 이 검증 불가능한 절대 부재 단언

- **현상/위반 지점**: NFR-1 "결정론적 동작 ... silent skip · race condition · 부분 실패 invisibility 가 없다"
- **문제 유형**: 비검증 가능
- **근거**: "race condition 이 없다" 는 부재 단언(absence claim) 으로 일반적으로 검증 불가. 어떤 시나리오의 race 를 어떤 보장으로 막는지 정의 안 되면 합격 판정 불가.
- **권장 액션**: NFR-1 을 시나리오별로 분해: "(a) 동시 INSERT 두 건이 동일 skillSeqs 활성 spec 2 개를 만들지 않는다 — UNIQUE 제약 또는 SELECT FOR UPDATE 로 보장. (b) 운영자 중복 클릭이 spec 2 개를 만들지 않는다 — 멱등성 또는 클라이언트 가드. (c) 트랜잭션 부분 commit 이 운영자에게 success 로 보이지 않는다 — FR-2.2 와 연결."

1. **D-1~D-5 권장안 확정 또는 트레이드오프 분석 보강** — /architect 진입 전 결정 필요. 특히 D-2(productType 전환 비허용) 가 비허용이면 결함 D/E 의 정합화 경로(FR-3.3) 가 어떻게 정의되는지 동시 결정.

3. **결함 E 의 "productCode 매핑 단절" 을 FR 본문에 별도 명시할지** — 진단 SQL §3 (orphan) 만으로 검출 가능한가, 별 SQL 이 필요한가.
