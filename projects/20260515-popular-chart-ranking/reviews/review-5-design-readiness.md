## 설계 가능한 부분 (지금 바로)

architect가 6/5 시점 문서만으로 **막힘 없이** 그릴 수 있는 것:

### 1. 통합 필터 정의 레이어 스키마 골격 — 🟢 전면 가능
- FR-F1이 평가 대수를 **닫힌 형태**로 준다: `section_filter(skill) = Π match(axis_i, values_i)` (축 간 AND, 축 내부 OR). 이건 스키마의 형태(섹션 1개 = 술어 N개, 술어 1개 = `{axis, op(IN/=), values[]}`)를 그대로 결정한다.
- 축 도메인도 닫혀 있다 — FR-F2가 5축(topic·intents·temporal·content_type·targets)을 **확정 enum**으로 고정. 새 축 추가는 명시적 비목표(readme §7).
- 복합 조건(FR-F4: `intents ∩ targets`)도 "축 간 AND"의 한 사례로 동일 스키마에 떨어진다 — 별도 모델링 불요.
- → **결론**: 스키마 골격(엔티티/컬럼/술어 표현형)은 지금 완성 가능. DF-S4-3(운영편집 vs 코드고정)이 미정이어도 막히지 않는다. **이유는 아래 "막히지 않는 이유" §A 참조.**

### 2. 랭킹 마트 + popularity_score 산출 설계 — 🟢 전면 가능
- 공식이 canonical 로 확정(data-measurement-plan §5): 6항목·가중치·7일 윈도우·정규화 자리(norm). 산출물 스키마까지 제안됨 `(date, menu_seq, popularity_score, rank, signals_json)`.
- 6시그널 전부 BQ 원천·입도·판정이 매핑됨(§4). 평점은 긍정평가비율로 재정의 완료(FR-R5). 키 정합(menu_seq STRING ↔ fixed_menu.seq INT, CAST)도 확정.
- α(가중치 6 vs 5)·조회정의가 미확정이지만 — **둘 다 마트 스키마를 바꾸지 않는다**(상수·분모 정의일 뿐, 컬럼 구조 불변). architect는 norm 방식 선택(min-max vs percentile)을 설계 결정으로 잡으면 된다.

### 3. 적재(reverse-ETL) 데이터 계약 — 🟢 전면 가능
- 적재 *결과물*의 형태가 고정됨: `섹션 → [정렬된 menu_seq, rank, score]`(FR-B2, S4 §8). 적재 *대상 테이블* 신설은 서버 과업으로 명시(tasks.md 서버).
- NFR-2(배치 실패 폴백)·NFR-3(삭제·비노출 스킬 제외 정합)이 적재 계약의 무결성 조건을 이미 준다.
- → 적재 테이블 DDL과 reverse-ETL 인터페이스(무엇을 넣고 무엇을 읽나)는 지금 그릴 수 있다. **적재 "주체"(서버 vs MWAA vs CronJob)가 미정이어도 데이터 계약은 불변** — §B 참조.

### 4. 랭커 추상화 인터페이스 — 🟢 전면 가능
- FR-R1이 시그니처를 준다: 입력=후보 풀+컨텍스트, 출력=정렬 목록. 2종 구현(AsIs/PopularityScore v1)도 확정.
- AsIs 의미(재정렬 없음=서버 입력 순서 유지)도 FR-R6에서 닫힘. 다중 랭커는 "구조만 비파괴, 운영은 범위 밖"으로 경계 명확(FR-R7).

### 5. variant 주입점의 **위치·계약 골격** — 🟢 가능 (단 §c 한정)
- FR-AB1/AB2가 구조 요구를 1개로 압축: "2종 목록 산출 + 어느 것을 서빙할지 외부 신호로 주입받는 지점 분리". 이 *seam 의 위치*(서빙 경로상 어디서 A/B 목록을 가르나)는 지금 설계 가능.

## 설계가 막히는 부분 + 무엇 때문에

### 🟠 [Major] api-spec.md — 섹션 12종 ↔ 기존 `featuredSkillsTabs` 실매핑
**무엇 때문에 막히나**: S1 §5·§9가 정직하게 남긴 **운영 DB 의존 미해결분**.
- S1 §5: "신규 인기"·"1:1 상담" 섹션의 As-Is 대응이 **불확실(DB 의존)**. `targetSection` 화이트리스트 12종은 알지만, 기획이 말한 12개 *논리 섹션*(사주/재회/솔로 애정운…)이 그중 어느 `targetSection`+`targetSectionTag` 조합으로 떨어지는지는 **운영 DB `TodayTagSkillsTag`/`FixedMenuTag` 덤프 없이 확정 불가**(S1 §9-2,3).
- 즉 readme §6의 "12종 섹션"과 S1의 "12종 `targetSection` 화이트리스트"는 **이름이 12로 같을 뿐 같은 집합이라는 보장이 문서에 없다**(예: 논리 섹션 다수가 `recommendedTagSkills` 하나에 `targetSectionTag` 만 달리해 매핑될 수 있음). 이 매핑이 **N:1인지 1:1인지**가 api-spec의 요청 파라미터·응답 형태를 가른다.

**막히는 범위(정직하게)**: api-spec **전체가 막히는 게 아니다.**
- 서빙 API의 *골격*(기존 `GET /api/home/featured-skills-tab/:tabSeq` 재사용, variant 주입점 경유, 2종 목록 서빙)은 S1 §1·§7 레퍼런스로 **지금 그릴 수 있다**.
- 막히는 건 **"섹션 식별자 → 자동화 결과 조회 키" 매핑 필드 1개 면**. 이게 N:1이면 `(targetSection, targetSectionTag)` 복합키로 적재/조회해야 하고, 1:1이면 단순 섹션키로 충분 — **DDL과 조회 시그니처가 갈린다.**

**왜 Major(설계 일부 차단)이고 Blocker가 아닌가**: architect는 "복합키(`targetSection`+`targetSectionTag`) 가정"으로 **더 일반적인 쪽을 디폴트로 잡아 설계를 진행**할 수 있다(단순키는 복합키의 특수사례). 따라서 *진행은 가능*하되 **하나의 큰 가정을 강요**받는다. 실측이 단순키로 나오면 over-engineering, 복합키로 나오면 정합 — 디폴트를 일반형으로 잡으면 손해가 비대칭적으로 작다.

### 🟡 [Minor] FR-B3/B4 적재 주체·타이밍 — status.md ↔ requirements.md 불일치가 설계자에게 혼선
- **status.md "확정 사항"** 은 `적재=서버 위임(컨벤션), D+1 일배치` 를 **확정**으로 적었다.
- **FR-B3** 은 같은 사안을 `S5에서 확정(SHOULD)` 으로, **FR-B4** 는 타이밍을 `S5에서 조정(MUST 타이밍 확인)` 으로 적었다.
- → architect 입장에서 "이미 정해진 걸 다시 정하라는 건지, 미정인지" 신호가 충돌. **단 이건 설계를 막지 않는다** — 적재 주체/타이밍은 *데이터 계약을 바꾸지 않는* 배포 토폴로지 결정이고, S5가 정하는 게 정상 범위(인프라 △). status.md 쪽을 "방향(잠정)"으로 낮추면 해소. **설계 중 architect가 결정 가능.**

### 🟡 [Minor] variant 주입 *신호의 형태·기본값* — 명시적으로 후속 이연
- (c) 항목: 주입점 *위치/seam*은 설계 가능하나, **입력 신호의 자료형·기본값**은 S2가 의도적으로 후속 단계로 넘김("신호의 출처·구현은 후속 단계 책임", FR-AB4).
- → architect는 **"기본값 = A(현행), 주입 인터페이스는 불투명 토큰 1개를 받는 자리만 노출"** 로 **계약의 최소 골격**을 박을 수 있다. 버킷팅 미구현 상태에서 주입점이 *동작*해야 한다는 수용기준(requirements §I-3)이 이미 "기본값 통과" 동작을 함의 → **설계 가능, 막히지 않음.** Minor인 이유: 기본값(A 디폴트)을 문서가 명시 안 해 architect가 1줄 가정을 채워야 함.

## 막히지 않는 이유 (architect가 의심할 핵심 3개 — 명시)

**§A. DF-S4-3(운영편집 vs 코드고정)이 미정인데 필터 스키마를 왜 그릴 수 있나?**
술어 *구조*(FR-F1의 곱집합 형태)와 술어 *저장 표면*(테이블 row냐 코드 상수냐)은 직교한다. 같은 `{axis, op, values[]}` 구조를 운영 DB row 로도, config 상수로도 표현 가능 — **둘은 같은 논리 스키마의 두 물리화**다. architect는 논리 스키마를 확정하고 물리화(운영편집형 우선, DF-S4-3 디폴트)를 설계 결정으로 잡으면 된다. → **막히지 않음.**

**§B. 적재 주체 미정인데 파이프라인을 왜 설계할 수 있나?**
적재 *데이터 계약*(`섹션→[menu_seq,rank,score]` 테이블, 무결성 NFR-2/3)은 적재 *주체*와 무관하게 고정이다. 서버가 넣든 MWAA가 넣든 *테이블 형태와 읽기 인터페이스는 같다*. 주체 선택은 "누가 트리거하나"의 배포 토폴로지(인프라 △)일 뿐 — S5가 정하는 게 정상이며, **기존 컨벤션(refresh_recent_purchased_skills=서버계산+MWAA트리거)이 디폴트로 이미 제시됨**. → 데이터 계약·DDL은 지금, 주체는 설계 중 결정.

**§C. 가설(태그 값)이 실측 전인데 스키마·배치·API를 어디까지?**
- 스키마: `values[]` 컬럼만 있으면 값은 *데이터* → **끝까지 설계 가능.**
- 배치 골격: 필터 적용 로직(`axis ∋ values` 평가)은 값과 무관 → **끝까지 가능.**
- API: 조회 시그니처는 값과 무관 → **가능** (단 섹션↔컨테이너 *키형*은 위 Major로 별개 종속).
- **막히는 지점**: 값 *바인딩 자체*(어느 섹션에 어떤 distinct 값이 들어가나)와 T1 후보 풀 검증(섹션이 비는가). 이건 *설계 산출물이 아니라 구현 전 데이터 확정* → S5를 막지 않고 **구현 착수를 막는다**(tasks.md 의존관계가 이미 이 순서를 명시).

## 선결 결정 Top 3 (우선순위 + 이유)

### 1순위 🟠 — 섹션 12종 ↔ `featuredSkillsTabs`(`targetSection`/`targetSectionTag`) 실매핑이 **N:1인지 1:1인지**
- **왜 이게 설계를 늦추나**: api-spec.md 의 조회키 시그니처와 적재 테이블의 키형(단순 섹션키 vs `(targetSection, targetSectionTag)` 복합키)을 직접 가른다. 이게 미정이면 architect는 *복합키 일반형*을 가정으로 강요받고, 실측 결과에 따라 DDL/조회 시그니처를 되돌려야 할 위험이 유일하게 남는다.
- **어떻게 빨리 푸나**: 코드 탐색 불필요 — **운영 DB `HomeSectionFeaturedSkillsTab` + `TodayTagSkillsTag` 의 현재 row 덤프 1회**(S1 §9-2,3 가 이미 지목). /dev-data 인증 복구와 무관(서비스 PG). → **api-spec 착수 *전* 또는 *병행 초기*에 1회 확인이면 충분.** (s1-current-state §7 이 정확한 테이블/서비스 경로 제공)

### 2순위 🟡 — status.md ↔ FR-B3 의 "적재 주체 확정 여부" 표기 정합
- **왜 이게 설계를 늦추나**: status.md 는 '서버 위임 확정', FR-B3 는 'S5에서 확정'. architect 가 SSOT 를 어느 쪽으로 읽느냐에 따라 "이미 결정됨/내가 결정" 이 갈려 **배치 아키텍처 절(section)의 권한 경계가 흐려진다.** 설계를 *막진* 않지만 *재작업/혼선* 리스크.
- **어떻게 빨리 푸나**: 코디네이터가 **status.md '확정 사항'의 배치 행을 '잠정 방향(S5 확정)'으로 1줄 강등** — FR-B3/B4(SHOULD·S5확정)와 정렬. (architect 가 설계 중 직접 확정해도 무방하므로 *문서 정합*만의 문제)

### 3순위 🟡 — Phase A(β)가 **1차 범위를 바꾸는가**의 1줄 확인
- **왜 이게 설계를 늦추나**: 현 스펙은 Phase B/C(쿼리·테이블·API·UI·부스트·튜닝)에 대응하고 골격은 그것만으로 완결된다. 그러나 Phase A 의도가 "전체 스킬 대상 파이프라인 선(先)구축" 같은 *범위 전제*면 마트 대상 모수(母數) 설계 전제가 바뀔 수 있다 — **스키마는 아니어도 배치 *대상 범위* 가정**에 영향.
- **어떻게 빨리 푸나**: 기획에게 "Phase A = (a) 단순 파이프라인 0단계인가 (b) 1차 모수 정의인가" **택1 질문 1개.** (a)면 설계 무관, (b)면 마트 WHERE 모수만 조정 → 어느 쪽이든 골격 불변, 확인만 빠르면 됨.

> Top 3 외 나머지(α·조회정의·DF-S4-2·태그 값 바인딩·DF-S4-4 임계)는 **전원 "구현 전 확정"** — S5 설계 골격을 막지 않으며, /dev-data 실측 + 기획 합의로 *구현 착수 전*에 닫으면 된다. tasks.md 의존관계가 이미 이 순서(실측→값 바인딩→T1)를 명시하고 있어 **누락 위험 낮음.**

## 종합

이 5개 계약 문서는 **S5 설계를 *전면 차단*하는 미결정이 없다** — 이것이 핵심 판정이다. 9개 오픈 결정이 모두 *값·상수·저장표면·배포토폴로지* 층에 정확히 격리되어 있어, **구조 설계(필터 술어 스키마·랭킹 마트·적재 데이터 계약·랭커 인터페이스·variant seam)는 지금 끝까지 그릴 수 있다.** 기획 6스텝의 S1~S4가 "구조를 가르는 결정(개념모델·랭킹공식·필터대수·하이브리드 태그소스)"을 닫고, "값/운영형식"을 의도적으로 후속/병행으로 분리한 설계가 **architect 입력으로서 잘 작동한다.**

단 **정직한 부분 판정**의 근거는 1개 면이다: **api-spec.md 의 섹션↔컨테이너 실매핑(N:1 여부)** 은 S1 이 스스로 "DB 의존 미해결"로 남긴 지점이고, 운영 DB row 덤프 1회 없이는 조회키·적재키형이 *일반형 가정*에 묶인다. 이는 설계 *시작*을 막지 않지만(복합키 디폴트로 진행 가능) 설계 *완결*을 그 1회 확인에 종속시킨다 — 그래서 "전면"이 아니라 "부분".

**권고 진행 순서**:
1. **architecture.md 전면 착수** — 필터 스키마·랭킹 마트·적재 계약·랭커 추상화·variant seam. 차단 없음. (DF-S4-3 운영편집형·적재주체 서버위임을 디폴트 설계결정으로 명기)
2. **api-spec.md 골격 착수 + Top 1 실측 1회**(운영 DB 매핑 덤프)로 조회/적재 키형 확정 → 골격을 실값으로 봉인.
3. **status.md 배치행 강등(2순위) + Phase A 1줄 확인(3순위)** 은 코디네이터/기획이 병행 — 설계 흐름과 독립.
4. α·조회정의·태그 값 바인딩·임계는 **구현 착수 게이트**로 미뤄도 안전(설계 산출물 불변).

**최종**: S5 = **부분 시작 가능.** architecture.md 전면 / api-spec.md 골격(+실매핑 1회로 완결). Blocker 0, Major 1(섹션↔컨테이너 실매핑), 나머지 Minor.
