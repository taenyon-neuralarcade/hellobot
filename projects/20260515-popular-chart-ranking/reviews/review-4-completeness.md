## 🔴 Blocker (이게 없으면 S5 설계를 시작할 수 없음)

### GAP-01 · 기획 섹션 12종 ↔ `targetSection` 화이트리스트 12종 매핑 전무 (신규 섹션 타입 필요 여부 미결)
- **무엇이 빠졌나**: readme §6 / s4 §2 의 "인기스킬 섹션 12종"은 **콘텐츠/사업 섹션**(실시간 인기·신규 인기·사주·타로·1:1 상담·솔로 애정운·짝사랑/썸·커플 궁합·재물운·커플 결혼 궁합·재회·솔로 결혼운)이다. 그런데 PR #2414 칩이 가리킬 수 있는 `targetSection`은 **닫힌 화이트리스트 12종**(s1 §5: `recentPurchasedSkills` / `recommendedTagSkills` / `freeTagSkills` / `popularTagSkills` / `appFree` / `todayFree` / `tomorrowFree` / `categorySkills` / `personalRecommendedSkill` / `frequentlyUsedSkills` / `skillHistory` / `premiumSkillHistory`)이며, 이는 **섹션 타입 enum**이지 "사주/재물운" 같은 콘텐츠 섹션이 아니다. 두 "12"는 **의미가 완전히 다른데** 어느 문서에도 대응표가 없다.
- **왜 필요한가**: s1 §5 의 12종 중 콘텐츠 섹션(사주/재회/솔로 애정운 등)에 대응되는 건 사실상 `recommendedTagSkills`/`freeTagSkills`/`popularTagSkills` **3종의 태그 섹션 하나**로 수렴하고(s1 §5 표: 9개 콘텐츠 섹션 → 이 3개 태그 섹션), 나머지 화이트리스트(appFree/todayFree/categorySkills/skillHistory…)는 이 프로젝트 12종과 **무관**하다. 즉 "기획 12 섹션"을 PR #2414 그릇에 얹으려면 ① 기존 태그 섹션 타입을 재사용하면서 `targetSectionTag`로 12종을 구분하는가, ② 아니면 신규 `targetSection` 타입(예: `popularChartSection`)을 화이트리스트에 추가하는가 — **이 한 줄이 갈리지 않으면 적재 테이블의 키(섹션 식별자가 무엇인가)도, API 응답 형태도, 칩 row 구성도 설계 불가**.
- **어디서 터지나**: **S5 최초 5분.** architect가 `architecture.md`에서 "적재 대상 테이블 PK = ?" 를 적는 순간 막힌다. FR-B2 는 `섹션 → [정렬된 menu_seq, rank, score]` 라 하지만 "섹션"의 **식별자 도메인**(화이트리스트 enum? `targetSectionTag` 문자열? 신규 섹션 키?)이 없다. tasks.md 서버 첫 줄("통합 필터 정의 레이어 — 섹션=필터 술어 집합")도 섹션 식별 단위가 없어 스키마를 못 짠다. dev-data 의 reverse-ETL 적재 키도 동일하게 막힌다.
- **제안**: **readme.md 에 §6.1 "섹션 매핑표" 신설** — 기획 12 섹션 각각에 대해 `(targetSection 타입, targetSectionTag 값, showRanking, 신규타입 필요 여부)` 4열. s1 §5 표의 "기획 예시 ↔ As-Is 대응"을 **확정 매핑**으로 승격. 신규 `targetSection` 타입 추가가 필요하면 그 자체가 클라이언트 영향(화이트리스트 검증)·어드민 영향을 낳으므로 readme §8 영향표도 갱신. **이건 기획 결정이라 S5 전에 코디네이터가 메워야 함**(architect 가 임의로 못 정함).

### GAP-02 · 노출 슬롯 수(상위 N) 정의 부재 — 합격기준·랭커 출력·빈 섹션 판정이 전부 미정의 위에 서 있음
- **무엇이 빠졌나**: 섹션당 **몇 개를 노출하는가(N)** 가 어느 primary 문서에도 없다. data-measurement-plan §6 T1 합격기준은 `섹션당 후보 ≥ 노출 슬롯×3`(s4 §5 도 "슬롯×3")인데 **슬롯 수가 미정의** → 합격선이 계산 불가능한 상수. s2 §개념모델은 "정렬·**상위 N**"이라 쓰지만 N을 끝까지 안 정한다.
- **왜 필요한가**: N 은 ① T1 합격기준의 분모(슬롯×3), ② 랭커 출력 길이(상위 N 절단), ③ "빈/과소 섹션" 판정 기준(후보 < N 이면 과소), ④ reverse-ETL 적재 행 수(섹션당 상위 N 만 적재 vs 전체 적재 후 서빙에서 절단), ⑤ A/B 비교 시 동일 슬롯 수 보장 — **다섯 군데가 N 에 종속**. 섹션마다 N 이 다를 수도 있다(실시간 인기 10개 vs 사주 6개). 단일 상수인지 섹션별 파라미터인지조차 안 정해짐.
- **어디서 터지나**: **dev-data T1 쿼리 작성 시 즉시** — "슬롯×3" 의 슬롯에 넣을 숫자가 없어 합격/불합격을 못 낸다(Acceptance 1번 검증 불가). **dev-server 적재 스키마**에서 "섹션당 상위 N 적재" 의 N 이 없어 "전부 적재" 로 잘못 굳으면 후속 재작업. **NFR-2 빈 섹션 금지**(GAP-04)도 "슬롯 채움 기준 = N" 이 없으면 판정 불가.
- **제안**: **data-measurement-plan §2 핵심정의에 "노출 슬롯(N)" 행 추가** — 단일 상수인지 섹션별인지, 기본값(예: PR #2414 `capFeaturedSkillsTabData` 의 현행 cap 값을 s1 §7 레퍼런스에서 확인해 채택), 클라이언트 표시 한계와의 관계 명시. requirements.md 에 `FR-API5`(섹션 노출 슬롯 수 = N, 섹션별 오버라이드 가능) 신설. **PR #2414 `capFeaturedSkillsTabData` 현행 cap 을 dev-server 가 1줄 확인하면 즉시 채울 수 있음** — 이 확인을 S5 입력으로 명시.

### GAP-03 · 빈/과소 섹션 런타임 동작 미정의 (NFR-2는 "배치 실패 폴백"만 다룸)
- **무엇이 빠졌나**: 후보 풀이 노출 슬롯보다 **적을 때**(또는 0일 때) 서빙이 어떻게 동작하는지 규칙이 없다. NFR-2 는 "**배치 실패** 시 직전 결과 폴백(빈 섹션 금지)" 만 규정 — 이건 *배치가 죽었을 때*다. **배치가 정상인데 필터 결과가 비거나 슬롯 미달인 정상 케이스**(T1 이 사전 탐지하려는 바로 그 "빈/과소 섹션")의 런타임 처리는 공백. 숨김? AsIs 로 폴백? 다른 섹션 후보로 채움? 슬롯 미달인 채 노출? — 미정.
- **왜 필요한가**: s4 §3 의 1차 가설 필터는 값 바인딩이 실측 전이라(FR-F7) **일부 G3 섹션(재회·솔로 결혼운 등)은 후보가 슬롯 미달일 개연성이 높다**(좁은 의도∩대상 복합 조건). T1 이 "과소 섹션을 사전 탐지" 한다고 했으면, 탐지 후 **그 섹션을 프로덕션에서 어떻게 노출할지**가 반드시 따라와야 하는데 그 다음 동작이 비어 있다.
- **어디서 터지나**: **dev-server 서빙 로직 구현 중** — `fetchSkillListBySectionType` 결과가 빈 배열/슬롯 미달일 때 분기를 코더가 **임의로** 정하게 되고(숨김 vs 노출), QA 단계에서 "재회 섹션이 빈칸으로 떴다 / 갑자기 사라졌다" 로 재작업. NFR-2 만 보고 구현하면 *정상 배치 + 빈 결과* 케이스가 폴백 분기를 안 타서 빈 섹션이 그대로 노출됨(NFR-2 의 "빈 섹션 금지" 의도와 모순).
- **제안**: **requirements.md NFR-2 를 두 케이스로 분리** — (a) 배치 실패 → 직전 결과 폴백, (b) **정상 배치·후보 부족** → {섹션 숨김 | AsIs 폴백 | 슬롯 미달 노출 허용} 중 택1을 명시. 이는 **기획/사업 결정**(빈 사주 섹션을 숨길 것인가)이므로 readme §11 오픈 결정사항에 행 추가하고 S5 전 확정. data-measurement-plan T1 도 "과소 섹션 탐지 → 처리정책 X 적용" 으로 연결.

### GAP-04 · A 랭커(AsIs) 산출·캡처 방법 미정의 — "2종 동시 산출" 의 A쪽 파이프라인이 공백
- **무엇이 빠졌나**: FR-AB1·Acceptance 2 는 "동일 후보 풀에 A·B **2종 목록을 동시 산출**" 을 MUST 로 건다. B(PopularityScore)는 BQ 일배치로 떠서 적재된다(FR-B1/B2, s4 §8). 그런데 **A(AsIs=현행 서버 순서)를 어떻게 "산출/적재" 하는가**가 어디에도 없다. FR-R6 은 A 의 *정의*("현행 서버 순서 유지")만 말하고 *산출 경로*는 침묵.
- **왜 필요한가**: A 의 현행 순서는 s1 §3/§4 기준 **런타임에 생성**된다 — `recentPurchasedSkills`는 Redis 배치+SQL+pinned, 태그 섹션은 `getFixedMenusByTagName`(priority ASC)로 **서버가 그때 계산**. 즉 A 는 "BQ 에 미리 떠 있는 목록"이 아니다. 그런데 A·B 를 *동일 후보 풀*에서 비교하려면 둘이 **같은 기준선**(같은 후보·같은 시점·같은 노출가능 필터)을 공유해야 한다. A 를 ① 런타임 현행 경로로 그냥 두고 B 만 적재해 variant 분기에서 갈라치는가, ② 아니면 A 도 B 와 같은 배치 파이프라인에서 "현행 순서" 를 **캡처해 적재**하는가 — 이 선택이 적재 테이블 스키마(variant 컬럼 유무)·variant 주입점 위치를 바꾼다.
- **어디서 터지나**: **S5 적재 스키마 + variant 주입점 설계 시.** architect 가 "적재 테이블에 ranker_id/variant 컬럼을 둘 것인가(A,B 둘 다 적재)" vs "B 만 적재하고 A 는 런타임 현행 경로" 를 **A 산출 방법을 모르면 결정 불가**. 잘못 고르면 dev-server 가 variant 분기를 한쪽 가정으로 구현한 뒤 측정 단계에서 "A·B 가 같은 후보 풀이 아니다(A 는 pinned 포함, B 는 미포함)" 로 비교 자체가 무효 → 측정 재설계 + 구현 재작업.
- **제안**: **requirements.md 에 FR-AB5 신설** — "A(AsIs) 산출 경로 명시: {런타임 현행 경로 유지 | 배치 캡처 적재} 택1, 그리고 A·B 가 공유하는 후보 풀·노출가능 필터·시점 기준선 정의". readme §11 에 오픈 결정 행 추가(A 캡처 방식). s4 §8 이 "A=현행 서버 순서, B만 배치 산출" 이라 *한쪽으로 기울었지만*, 그 경우 **A·B 후보 풀 동일성 보장 방법**(pinned·minPrice·priority 차이를 어떻게 정렬하나)이 빠져 있으므로 이를 명문화.

## 🟠 Major (S5 일부는 가능하나 구현 중 재작업 위험)

### GAP-05 · 중복 노출·dedup 런타임 규칙 부재 (T2는 "측정" 만, "처리" 없음)
- **무엇이 빠졌나**: 한 스킬이 여러 섹션 필터에 동시 적합할 때(예: 어떤 사주 스킬이 `topic=사주` ∩ `intents∋궁합` 둘 다) **여러 섹션에 중복 노출되는가, dedup 하는가** 규칙이 없다. "중복" 은 T2 정합(요구 FR-V2, dmp §6)에서 **측정 지표("섹션 간 중복도")로만** 등장하고, **노출 시 중복 처리 규칙**은 전무.
- **왜 필요한가**: G2(주제)와 G3(의도)는 축이 직교해 **교집합이 상당**할 수 있다(연애 사주 스킬 = 사주 섹션 + 솔로 애정운 섹션). 같은 홈탭에 같은 스킬이 2~3번 뜨면 UX 저하 + A/B CTR 측정 왜곡(같은 스킬 중복 노출이 클릭 분산).
- **어디서 터지나**: **dev-data 섹션별 목록 산출 시** — dedup 을 파이프라인에서 할지(전역 1회 노출), 서빙에서 할지, 안 할지(중복 허용)를 코더가 임의 결정 → QA 에서 "홈탭에 같은 스킬 3번" 발견 시 재작업. T2 가 중복을 *측정* 만 하고 *기준치 초과 시 행동* 이 없어 측정-행동 루프가 끊김.
- **제안**: requirements.md FR-F 에 `FR-F8`(섹션 간 중복 노출 정책: {허용 | 섹션 우선순위 기반 전역 dedup | 섹션 내 무관}) 신설. 허용이 default 여도 **명시적 결정**으로 박아 둘 것.

### GAP-06 · 신규 스킬 cold-start 노출 정책 부재 (신규부스트 0.05 만으로 불충분)
- **무엇이 빠졌나**: 7일 윈도우 시그널(구매·조회·전환·매출)이 **데이터 부족한 신규 스킬**의 점수·노출 처리가 없다. dmp §2 는 "신규 스킬은 데이터 일수 보정"(분모를 실데이터 일수로) 1줄, FR-R3 는 신규부스트 0.05 가산뿐. 그런데 신규부스트 0.05 는 **6항목 중 가중치 최소** — 데이터 0인 신규 스킬은 나머지 0.95 가 거의 0 정규화라 **항상 바닥 랭크**가 된다. "신규 인기" 섹션(G1)의 존재 목적과 정면 충돌.
- **왜 필요한가**: readme §3 목표 + KPI "신규 진입율"(발견성)은 신규 스킬 노출이 핵심인데, 점수 공식이 신규 스킬을 구조적으로 하위에 깐다. "신규 인기" 섹션을 점수순으로 뽑으면 **신규인데 데이터 없어서 안 뜨는** 역설.
- **어디서 터지나**: **dev-data 랭킹 산출 후 검증 시** — "신규 인기 섹션 상위에 정작 신규 스킬이 없다" 가 T1/T4 에서 드러나 점수 공식 또는 신규 섹션 랭커를 **별도 설계**해야 함(재작업). G1 "신규 인기" 의 ranker 가 PopularityScore 인지, 등록일 정렬인지조차 s4 §3 에 "ranker=짧은 윈도우 인기" 로 모호.
- **제안**: data-measurement-plan §5 에 cold-start 정책 명시 — (a) 신규 섹션은 PopularityScore 가 아닌 **등록일/초기 속도 랭커** 별도 사용, 또는 (b) 데이터 부족 스킬에 prior/스무딩(베이지안 평균) 적용. readme §11 오픈 결정으로 승격.

### GAP-07 · 운영자 오버라이드·pinned 와 자동 랭킹의 공존 규칙 부재
- **무엇이 빠졌나**: 현행 `recentPurchasedSkills`는 `pinnedSkillSeqs` 로 상위 고정(s1 §3, 현재 `[]` 지만 기능 존재)이고 FR-R6 도 A 의 정의에 "pinned 상위 고정" 을 포함한다. 그런데 **B(자동 랭킹) 적용 시 pinned 가 살아있는가, 무시되는가**가 없다. readme §7 은 "운영자 오버라이드" 를 **범위 밖**으로 빼지만, pinned 는 *신규 오버라이드 기능*이 아니라 *현행 동작*이라 "범위 밖" 으로 치우면 **현행 기능을 조용히 깨는** 것.
- **왜 필요한가**: 운영자가 특정 스킬을 사주 섹션 최상단에 강제 노출 중인데 B 가 그걸 무시하고 점수순으로 덮으면 운영 사고. A→B 전환이 **현행 pinned 운영을 회귀(regression)** 시키는지 판정 필요.
- **어디서 터지나**: **dev-server B 랭커 결합 시 + 운영 전환 시점.** pinned 처리 분기를 안 넣으면 운영자가 "고정해둔 스킬이 사라졌다" 신고 → 핫픽스. FR-R6 이 A 에만 pinned 를 묶어둬서 B 경로에서 pinned 소실이 *의도된 동작*인지 *누락*인지 불명.
- **제안**: requirements.md FR-R 에 `FR-R8`(B 랭커의 pinned/고정 처리: {pinned 를 상위 고정 후 나머지만 점수정렬 | pinned 무시}) 명시. "운영자 오버라이드는 범위 밖" 과 "현행 pinned 는 유지" 를 readme §7 에서 분리 서술.

### GAP-08 · 측정 이벤트 "조건부" 함정 — Output KPI·A/B 비교가 측정 불가로 귀결될 위험
- **무엇이 빠졌나**: Output KPI(전환율·신규진입율·구매상품수, readme §4/dmp §3)와 A/B Δ 판정을 **실제로 측정하려면 섹션 단위 + ranker variant 단위 노출/클릭 이벤트가 필수**인데, tasks.md 클라이언트 과업은 "**(조건부)** 측정용 신규 이벤트 … event-spec 도입 시" 로만, dmp §0 도 event-spec 을 "**(필요 시)**" 로 둔다. 즉 측정의 전제(이벤트)가 *조건부*인데 KPI 는 *MUST 판정 기준*이라 모순.
- **왜 필요한가**: 섹션별 CTR·전환을 **ranker variant(A/B) 로 쪼개** 비교하려면 노출/클릭 이벤트에 `section_id` + `ranker_id(variant)` 파라미터가 박혀야 한다. 기존 이벤트에 이 dimension 이 없으면(s1 은 PR #2414 가 이벤트를 추가했다는 언급 없음) **사후에 A/B 를 비교할 데이터 자체가 안 쌓임** → 프로젝트 목표 ③(효과 검증)이 측정 불능.
- **어디서 터지나**: **클라이언트 구현을 "조건부" 라 스킵 → 배포 → A/B 단계에서 "비교할 이벤트가 없다"** 발견. 이벤트는 소급 불가(과거 노출에 variant 태그를 못 붙임)라 **재배포 + 데이터 재수집 기간(주 단위) 손실**. "조건부" 가 사실상 "필수" 인데 약하게 적혀 측정 리스크가 은폐됨.
- **제안**: tasks.md·readme §8 에서 측정 이벤트를 **조건부 → 필수(MUST)로 승격**하되 *발화 시점*만 A/B 단계와 정렬. data-measurement-plan §0 에 "event-spec 은 A/B 측정의 전제이며 늦어도 B 적재와 동시 배포" 명시. 최소한 readme §11 오픈 결정에 "측정 이벤트 dimension(section_id, ranker_id) 도입 시점" 행 추가.

### GAP-09 · 필터 규칙 데이터화(FR-F6) 시 어드민 편집 UI 필요 여부 미결
- **무엇이 빠졌나**: DF-S4-3 이 필터 정의를 "**운영 편집형(데이터화)**" 로 방향 잡았고 FR-F6 도 "코드 변경 없이 추가·수정" 을 요구한다. 그런데 운영자가 그 규칙을 **무엇으로 편집하는가**(어드민 화면? 직접 DB? 구글시트?)가 없다. s1 §5 의 비교 대상 `TodayTagSkillsTag` 는 어드민 화면이 있는데(s1 §2/§7 어드민 레퍼런스), 신규 필터 규칙 테이블의 편집 수단은 공백.
- **왜 필요한가**: "데이터화 = 운영자 편집 가능" 의 *운영자 편집* 은 어드민 CRUD 화면을 함의한다. 이게 범위에 있는지 없는지에 따라 hellobot-server 어드민(`src/admin/...`) 작업이 추가되고, 영향표(readme §8)·tasks 가 달라진다.
- **어디서 터지나**: **S6 범위 확정 / dev-server 착수 시** — 어드민 화면이 "당연히 필요" 로 뒤늦게 추가되면 서버 과업·일정 증가. 반대로 1차는 DB 직접 입력/시드로 가고 어드민은 후속이라면 그 결정을 명시해야 측정 가능한 범위가 됨.
- **제안**: readme §7 범위에 "필터 규칙 어드민 편집 UI: {1차 포함 | 후속(1차는 시드/DB)}" 명시. 후속이면 FR-F6 의 "운영자 수정 가능" 은 *데이터 구조상 가능* 까지로 한정하고 *UI 는 후속* 으로 분리. tasks.md 서버에 (조건부) 어드민 과업 행 추가.

### GAP-10 · i18n / 일본(jp) 적용 범위 미정의
- **무엇이 빠졌나**: 섹션·태그·랭킹이 **KR 전용인가, jp.hellobot 에도 적용인가**가 전무. readme §7/dmp §1 은 "**국가별 분기 제외**" 라 쓰는데, 이는 "유저 국가에 따라 다르게 노출하지 않는다" 일 뿐 **jp 서비스에 이 자동화가 도는가/안 도는가** 와는 다른 질문이다. CLAUDE.md 워크스페이스 기준 서비스는 KR+JP 이중 운영(jp.hellobot.co)인데 이 프로젝트 문서는 jp 를 한 번도 언급 안 함.
- **왜 필요한가**: 임시 태그(`taenyon_temp_skill_tag_info_v2`)·BQ 마트·`chatbot_content_type` 이 **KR 데이터 기준인지, jp 스킬도 커버하는지** 불명. jp 에도 동일 섹션 12종이 있는지, jp 칩이 PR #2414 를 쓰는지 모름. jp 가 범위면 태그 커버리지·필터 값 바인딩이 2배가 되고, 범위 밖이면 **jp 홈탭에서 이 섹션들이 어떻게 동작하는지**(빈 채로? 기존대로?)를 정의해야 한다.
- **어디서 터지나**: **dev-data 마트 산출 시(jp menu_seq 포함 여부)** + **dev-server 서빙 분기(jp 요청 시 자동화 적용?)**. 늦게 발견되면 "jp 홈탭 인기 섹션이 KR 스킬을 노출" 같은 사고 또는 jp 전체 미적용으로 범위 축소 재논의.
- **제안**: readme §7 범위에 "적용 서비스: {KR 전용(1차) | KR+JP}" 명시 + jp 가 범위 밖이면 "jp 홈탭 인기 섹션은 현행 AsIs 유지" 를 NFR 로 박기. data-measurement-plan §1 범위에 언어/서비스 축 추가. "국가별 분기 제외" ≠ "jp 미적용" 을 명확히 분리.

## 🟡 Minor (S5 비블로킹 — 설계 중·후에 채워도 됨)

### GAP-11 · 캐싱·무효화 전략 부재 (reverse-ETL 신규 적재의 freshness 경로)
- **무엇이 빠졌나**: NFR-1 은 "사전계산·적재 후 서빙 시 조회만"(런타임 점수계산 금지)을 요구하지만, **적재 결과를 캐싱하는지/TTL/일배치 갱신 시 캐시 무효화**가 없다. 현행 `recentPurchasedSkills`는 Redis(`recent-purchased-skills:data:latest`)+miss 폴백 구조(s1 §3)인데, 신규 섹션 목록도 Redis 캐싱인지 PG 직조회인지, 일배치 적재 후 캐시 갱신을 어떻게 트리거하는지 미정.
- **왜 필요한가**: NFR-2 신선도(D+1 최신)와 NFR-1 성능이 충돌할 때 캐시 무효화 타이밍이 freshness 를 좌우. 적재는 됐는데 캐시가 안 깨지면 묵은 목록 노출.
- **어디서 터지나**: S5 후반 또는 dev-server 구현 중 — 설계 블로킹은 아니나 누락 시 "갱신했는데 안 바뀐다" 버그. **제안**: architecture.md(S5) 에서 캐싱 레이어·무효화 트리거를 현행 Redis 컨벤션과 정합되게 설계하도록 §11/NFR 에 한 줄 가드 추가.

### GAP-12 · 롤백·점진 배포(canary) 절차 부재
- **무엇이 빠졌나**: 자동 랭킹(B)을 켰다가 문제 시 **현행(A)로 되돌리는 롤백 절차·점진 노출**이 없다. variant 주입점은 "A/B 비교용" 으로만 서술(FR-AB2)되고 **안전장치(킬스위치)** 관점이 없다.
- **왜 필요한가**: A/B 버킷팅은 범위 밖(FR-AB4)이라도, 최소한 "전체를 A 로 되돌리는 플래그"는 운영 안전상 필요. variant 주입점이 그 역할을 겸할 수 있으나 명시 안 됨.
- **어디서 터지나**: 운영 배포 시 — 설계 블로킹 아님. **제안**: variant 주입점 정의에 "기본값 = A(현행), B 는 명시 주입 시에만" 을 더해 자연 롤백(주입 제거 = A 복귀) 보장. requirements FR-AB2 에 한 줄.

### GAP-13 · rankings 이력 데이터 보존 정책 부재
- **무엇이 빠졌나**: 일배치 산출물 스키마 `(date, menu_seq, popularity_score, rank, signals_json)`(dmp §5)의 **보존 기간/이력 누적 여부**가 없다. 매일 덮어쓰는지(latest 만), 날짜별 누적인지 미정. T4 사후 분석·A/B 회고는 **과거 랭킹 스냅샷**을 요구.
- **왜 필요한가**: 적재 테이블이 "최신만" 이면 사후에 "어제 사주 섹션이 뭐였나" 를 못 본다 → T4 행동정합·A/B 비교의 사후 검증 불가.
- **어디서 터지나**: 측정 단계(사후) — 구현 블로킹 아님. **제안**: data-measurement-plan §5 산출물 스키마에 "이력 보존: 날짜 파티션 누적, 보존 N개월" 한 줄. BQ 산출물은 자연 누적이나 **서비스 DB 적재본의 보존**은 별도 결정.

### GAP-14 · NFR-3 "노출가능 스킬 정합" 과 배치 시점 간 정합성 갭
- **무엇이 빠졌나**: NFR-3 은 적재 menu_seq 가 "삭제·비노출 스킬 제외" 를 요구하나, **일배치 적재 시점(D+1 새벽/11시)과 서빙 시점 사이에 스킬이 비노출 전환되면** 묵은 목록이 비노출 스킬을 노출한다. 배치-서빙 시점차의 정합 보정(서빙 시 노출가능 재확인)이 명시 안 됨.
- **왜 필요한가**: 운영자가 스킬을 내렸는데 적재본엔 남아 다음 배치까지 노출되는 갭.
- **어디서 터지나**: 운영 중 — 블로킹 아님. **제안**: NFR-3 에 "서빙 시점 노출가능 재확인(적재본 ∩ 현재 노출가능)" 보강. (s1 §3 의 현행 `recentPurchasedSkills` 가 이미 SQL 에서 minPrice/visible 재확인하는 패턴 참고.)

### GAP-15 · "실시간 인기" 섹션의 ranker·윈도우 정의 미결
- **무엇이 빠졌나**: s4 §3 이 "실시간 인기 = 필터≈없음, ranker=**짧은 윈도우 인기**" 라 했으나, B 랭커는 **7일 고정 윈도우**(FR-R4)다. "실시간" 이 7일이면 실시간이 아니고, 별도 짧은 윈도우 랭커면 **랭커가 2종 초과**(범위 위반). 미해소.
- **왜 필요한가**: "실시간 인기" 가 12종 중 하나인데 그 랭커 정의가 B(7일)와 모순. 이 섹션을 어떤 ranker 로 채우는지 불명.
- **어디서 터지나**: dev-data 산출 시 — "실시간 인기" 를 7일 점수로 채우면 기획 의도(실시간)와 불일치. **제안**: readme §11 또는 data-measurement-plan 에 "실시간 인기 섹션 윈도우: {B 의 7일 그대로 | 별도 단기 윈도우}" 결정. 별도면 FR-R 범위(랭커 2종) 재서술 필요.
