### B2 · 측정계획 자기모순 — §5 공식은 `norm(긍정평가비율)`인데 §9 SQL 템플릿은 `0.15*eval`(norm 미적용)
- **위치**: data-measurement-plan §5 line 65 vs §9 SQL line 116 (둘 다 같은 문서 내).
- **문제**: §5 canonical 공식은 6항목 **전부 `norm()`으로 감싼다**: `norm(긍정평가비율) × 0.15`. 그런데 같은 문서 §9 대표 쿼리 템플릿은 `0.35*norm(buy7)+0.1*norm(view7)+0.2*norm(cvr)+0.15*eval+0.15*norm(rev7)+0.05*is_new` — **`eval`(긍정평가비율)만 `norm()` 없이 raw로 들어간다**. 전환율(cvr)도 §5는 `norm(전환율)`인데 §9는 `norm(cvr)`로 일치하므로, eval 누락은 단순 패턴이 아니라 **명백한 비대칭 모순**. 긍정평가비율은 이미 0~1(비-💩 비율)이라 norm 생략이 "의도된 설계"일 수 있으나, **§5는 명시적으로 `norm(긍정평가비율)`이라고 적었다** → 두 정의가 산출값을 다르게 만든다(min-max norm 은 0~1 입력도 재스케일하여 분포를 바꿈).
- **근거(대조 인용)**: §5 line 65 `+ norm(긍정평가비율)        × 0.15` ↔ §9 line 116 `...+0.15*eval+...`. 같은 항목, 한 곳은 norm 적용, 한 곳은 미적용.
- **다음 단계 영향**: /dev-data 가 §9 템플릿을 구현 출발점으로 삼으면 §5 계약과 다른 스코어를 산출 → 측정 SSOT(이 문서가 자칭 "데이터 측정 SSOT", line 4)가 내부에서 두 값을 정의하는 상태. 랭킹 순위가 미세하게 달라짐.
- **제안 수정/질문**: 긍정평가비율에 norm 을 적용할지 한쪽으로 통일하라. 권장 = **긍정평가비율은 이미 0~1 비율이므로 norm 미적용**으로 §5 를 `긍정평가비율 × 0.15`(norm 제거)로 수정하고 §9 와 맞추거나, 반대로 §9 를 `0.15*norm(eval)`로 고쳐라. 어느 쪽이든 §5↔§9 가 글자 단위로 같아야 함. (참고: 이 norm 미적용 결정은 readme §5·requirements FR-R3 와도 동기화 필요 — 둘 다 "각 항목 0~1 정규화"라고 일괄 표현하므로 긍정평가비율 예외를 명시해야 함.)

### M1 · 측정계획 §9 SQL 윈도우가 'BETWEEN DATE_SUB(-7 DAY) AND CURRENT_DATE' = **8일**이고 AVG() 사용 → '7일 합 ÷ 실데이터 일수' 정의와 불일치
- **위치**: data-measurement-plan §2 line 28 / requirements FR-R4 line 28 (정의) vs data-measurement-plan §9 line 108~112 (SQL).
- **문제**: 정의는 **"최근 7일 합 ÷ 실데이터 일수(=7일 평균)"**(FR-R4, §2 핵심정의 둘 다 동일). 그런데 §9 SQL 은 `WHERE date BETWEEN DATE_SUB(CURRENT_DATE, INTERVAL 7 DAY) AND CURRENT_DATE` — BigQuery `BETWEEN` 은 **양 끝 포함**이라 이 범위는 D-7 ~ D-0 = **8개 캘린더일**을 스캔한다. 게다가 집계가 `AVG(purchase_cnt)`로, 이는 **"합 ÷ 실데이터 일수"가 아니라 "합 ÷ (그룹 내 행 수)"** = 캘린더-데이별 평균에 가까움. "실데이터 일수 보정"(신규 스킬은 데이터 일수 보정, §2 line 28)이라는 정의를 AVG 가 우연히 만족할 수도 있으나(NULL 행이 GROUP 에서 빠지면), **8일 윈도우는 7일과 명백히 다르다**.
- **근거(대조 인용)**: §2 line 28 `7일 윈도우 | 최근 7일 합 ÷ 실데이터 일수(=7일 평균)` ↔ §9 line 112 `WHERE date BETWEEN DATE_SUB(CURRENT_DATE('Asia/Seoul'), INTERVAL 7 DAY) AND CURRENT_DATE('Asia/Seoul')`(8일). is_new 의 DATE_DIFF `<= 30` 은 30일과 정합(별개로 OK).
- **다음 단계 영향**: 템플릿이 "대표 — 실측 후 검증"이라 의사 SQL 이지만, /dev-data 가 그대로 베끼면 윈도우가 1일 더 길어진다. 7일 vs 8일은 신규/저빈도 스킬의 정규화 분모를 흔들어 랭킹에 영향.
- **제안 수정/질문**: §9 WHERE 절을 `date >= DATE_SUB(CURRENT_DATE, INTERVAL 6 DAY) AND date <= CURRENT_DATE`(=7일) 또는 `date > DATE_SUB(..., 7 DAY)`로 고치고, AVG 가 "÷ 실데이터 일수"를 의도한 것인지 주석으로 못박아라. 의사 SQL 임을 강조해 "실측 시 윈도우 7일 재확인" 캐비엇 추가.

- **문제**: readme/s3 는 "5종 가용 + 평점은 재정의 필요(6번째는 해소)"로 서술. measurement §4 표는 긍정평가비율 행까지 **6행 전부 `✅`**로 마킹(`✅ (비율 재정의 적용)`). 모순은 아님(재정의로 6번째도 가용해진 상태) 이나, readme 의 "5종"과 measurement 의 "6/6 ✅"가 카운트 인상을 다르게 줌. s3 line 81 `5시그널 BQ 가용 + 평점 재정의로 6번째 해소`가 정확한 표현.
- **제안**: readme D3 를 `6종 중 5종 직접 가용 + 평점은 긍정평가비율로 재정의해 6종 모두 충족`처럼 measurement 와 맞춰 다듬기.

### m4 · acceptance(requirements §I-4)는 'T1, 가능하면 T3'인데 FR-V1(MUST)·FR-V3(SHOULD)·measurement T1~T4 와 등급 매핑이 느슨
- **위치**: requirements §I line 95 (`T1, 가능하면 T3`) vs FR-V1~V5 등급 (V1 MUST, V2/V3 SHOULD, V4 MUST).
