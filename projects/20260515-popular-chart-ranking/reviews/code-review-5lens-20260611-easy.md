## ① 랭킹 켜면 스킬 카드가 깨집니다 — 챗봇 번호 체계 불일치 (ISS-001, 블로커)

- **무슨 문제**: 홈 칩의 스킬 카드에는 "이 스킬이 속한 챗봇 번호"가 실리고, 앱은 그 번호로 챗봇 정보 사전(이름·프로필 등)을 찾아 카드를 그립니다. 그런데 챗봇 번호에는 **내부 원본 번호와 화면용 변환 번호** 두 체계가 있습니다. 기존 코드는 전부 변환 번호를 쓰는데, 새 랭킹 코드만 원본 번호를 그대로 내보냅니다.
- **그대로 두면**: 실험을 켜는 순간 랭킹 군(C-A) 사용자 화면에서 사전 조회가 빗나가 **카드가 깨지거나, 번호가 우연히 겹치면 엉뚱한 챗봇이 표시**됩니다.
- **고치는 비용**: 변환 한 줄 추가 (기존 코드와 똑같은 방식으로).
- **판단**: 결정 불필요 — 무조건 고쳐야 함. (코드 직접 재확인 완료된 확정 버그)

## ② 랭킹이 서버에 영영 안 들어옵니다 — 날짜 라벨 약속 어긋남 (ISS-002, 블로커 · 🔴 결정 1)

- **무슨 문제**: 랭킹 데이터에는 날짜 라벨이 붙습니다. 배치는 "**데이터 기준일 = 어제**" 라벨을 붙입니다(어제까지의 7일치로 계산하니까 — 우유의 '제조일'이 아니라 '원유 기준일'인 셈). 그런데 설계 문서에는 배달부(매일 12시에 BQ→서버로 옮길 CronJob, 아직 미구현)에게 "**'오늘' 라벨 데이터만 옮겨라, 없으면 실패 알림**"이라고 적혀 있습니다.
- **그대로 두면**: 배달부는 매일 '오늘' 라벨을 찾는데 창고엔 항상 '어제' 라벨만 있음 → **매일 거짓 알람 + 랭킹이 서버로 한 번도 안 들어감** → C-A 사용자는 영영 기존 목록만 봅니다(폴백 동작).
- **선택지**:
  - **A안 (권고)** — 문서를 고친다: 라벨 정의를 "데이터 기준일(어제)"로 명시하고 배달부 규칙을 "어제 라벨 확인"으로 변경. **코드 무변경**, 분석 시 "6/11에 서빙된 랭킹 = 6/10 라벨" 한 줄만 기억하면 됨.
  - **B안** — SQL을 고쳐 라벨을 "만든 날(오늘)"로 변경. 직관적이지만 재실행 안전장치(같은 날 다시 돌려도 중복 안 생기게 하는 부분)까지 같이 손대야 해서 접면이 더 큼.
- **판단**: A/B 중 결정 필요. **권고 A.**

## ③ 웹 사용자를 실험에 포함할까요 — 노출 체크가 앱 기준뿐 (ISS-006, 메이저 · 🔴 결정 2)

- **무슨 문제**: 스킬에는 "앱에서 보임/숨김"과 "웹에서 보임/숨김" 설정이 따로 있는데, 새 랭킹 코드는 **앱 기준만** 봅니다. 칩 UI는 이미 웹에도 나가고 있습니다.
- **그대로 두면**: "웹에서만 보이는 스킬"이 랭킹 후보에서 아예 빠집니다. 웹의 C-A 사용자는 그 스킬을 못 보는데 C-M(수동) 사용자는 봄 → **C-M vs C-A 비교에 '랭킹 방식 차이' 말고 '후보 풀 차이'가 섞여 실험 해석이 흐려집니다.** 부가로 매일 적재 때 "버려진 스킬 있음" 경고가 반복돼 알람이 양치기 소년이 됩니다. (다행히 잘못 노출될 위험은 없음 — 화면 직전 최종 필터가 한 번 더 거릅니다. 문제는 "빠지는 것"뿐.)
- **선택지**:
  - **A안 (권고)** — 웹 포함 확정: "앱 또는 웹 어느 한쪽이라도 보이면 후보 유지"로 완화. 칩이 이미 웹에 나가는 현실과 정합.
  - **B안** — 1차 실험은 앱만: 웹 사용자를 랭킹 게이트에서 제외하는 코드 추가. 실험은 더 깔끔해지나 수정 범위가 늘고, 웹 트래픽만큼 표본이 줄어듦.
- **판단**: A/B 중 결정 필요. **권고 A.**

## ④ 불량 1건에 그날 적재 전체가 멈춥니다 (ISS-004, 메이저)

- **무슨 문제**: 적재 API는 "이상한 섹션 그룹은 건너뛰고(어제 것 유지) 나머지는 반영"하는 설계인데, 검사에 구멍이 하나 있습니다 — **한 그룹 안에 같은 스킬이 두 번 들어오는 경우**를 안 거릅니다. 이러면 DB의 중복 방지 장치에 걸리는데, 전체를 한 묶음으로 저장하다 걸려서 **멀쩡한 그룹까지 전부 되돌려지고 500 에러**가 납니다.
- **비유**: 상자 하나에 불량품이 있으면 그 상자만 빼면 되는데, 지금은 트럭 전체를 회차시키는 구조.
- **고치는 비용**: "그룹 내 스킬 중복이면 그 그룹만 건너뛰기" 검사 추가 — 기존 검사(순위 중복)와 똑같은 패턴 몇 줄.
- **판단**: 결정 불필요.

## ⑤ 그룹 이름이 충돌할 수 있습니다 — "수원시+장" vs "수원+시장" (ISS-003, 메이저)

- **무슨 문제**: 적재 데이터를 그룹으로 묶을 때 내부 키를 "섹션이름+태그"를 **구분자 없이 이어붙여** 만듭니다. "수원시"+"장"과 "수원"+"시장"이 같은 글자가 되듯, 서로 다른 두 그룹이 같은 키가 될 수 있습니다.
- **그대로 두면**: 충돌 시 남의 그룹 데이터가 엉뚱한 섹션 랭킹에 섞여 들어가거나, 멀쩡한 그룹이 통째로 거부됩니다. 실제 발생 확률은 낮지만(호출자가 내부 배치뿐) 조용한 데이터 오염 경로라 막아두는 게 정석입니다.
- **고치는 비용**: 구분자 추가 1줄.
- **판단**: 결정 불필요.

## ⑥ 구글시트 오타 한 줄에 무방비입니다 (ISS-007, 메이저)

- **무슨 문제**: 랭킹 재료 중 태그 정보는 **사람이 편집하는 구글시트**입니다. 시트에 같은 스킬이 실수로 두 줄 들어가면, 계산 과정에서 그 스킬이 두 번 불어나 랭킹 30위 안에 **같은 스킬이 두 번** 등장할 수 있습니다 → 서버가 거부(④번과 만나면 그날 전체 실패).
- **비유**: 재료 목록에 "설탕"이 두 번 적혀 있으면 진짜 두 번 넣는 기계 — 계량 전에 목록 중복부터 지워야 합니다.
- **고치는 비용**: SQL에 중복 제거 2줄 (사람 실수에 대한 안전벨트).
- **판단**: 결정 불필요.

## ⑦ 에러 신호등이 반대로 켜집니다 — 재시도 폭주 위험 (ISS-005, 메이저)

- **무슨 문제**: API 에러 코드는 호출자에게 주는 신호입니다 — **400 = "네 요청이 잘못됐어, 다시 보내도 똑같아" / 500 = "서버가 잠깐 아파, 재시도해 봐"**. 지금은 프레임워크와 전역 에러 처리의 궁합 문제로, 요청 형식 오류 대부분이 500으로 나갑니다.
- **그대로 두면**: 배달부(CronJob)는 보통 "500이면 재시도" 정책이라, **영영 안 고쳐질 잘못된 요청을 계속 재시도 + 알람 반복**. 고장난 자판기에 동전을 계속 넣는 꼴이 됩니다.
- **고치는 비용**: 검증을 한 곳에 모아 400으로 응답하게 정리 + 같은 부류의 자잘한 구멍 3개("2026-99-99" 같은 달력에 없는 날짜 등) 동시 처리.
- **판단**: 결정 불필요.

## ⑧ 다음 작업자를 위한 설명서에 함정이 3개 있습니다 (ISS-008, 메이저)

- **무슨 문제**: 배달부(CronJob)는 아직 안 만들었고, 만들 사람은 설계 문서만 보고 만들게 됩니다. 그 문서에 함정 3개 — ① "아직 칩에 연결 안 된 섹션은 빼고 보내라"는 규칙이 문서엔 없고 SQL 주석에만 있음(모르고 다 보내면 **전체 거부**) ② 응답 포장 형태가 실제와 다르게 적혀 있어 성공/실패 판독이 조용히 빗나감 ③ BQ에서 숫자를 뽑으면 문자열로 나와 변환이 필요.
- **고치는 비용**: 문서 3곳 보강 (코드 무변경).
- **판단**: 결정 불필요.

## 천천히 처리해도 되는 것

**문서 구버전 9곳 (ISS-009)** — 설계 문서가 코드보다 옛날입니다(예: 사주/타로 분류 기준이 실측 후 다른 컬럼으로 바뀐 것, 게이트 검사 순서). 전부 "코드가 정답"으로 확인 끝 — 문서만 갱신하면 됩니다. 단 ②⑧번 결정과 묶어 한 번에 개정 예정.

**자잘한 개선 12개 (ISS-010)** — 캐시 한 줄, 빠진 로그 등. 머지를 막을 건 없고, 수정 라운드 때 싼 것만 같이 태웁니다.

**확인 질문 7개 (ISS-011)** — 급하지 않지만 답이 필요한 것들:

1. **인기 집계 범위**: 구매수·매출에 **웹 구매와 패키지 구매**도 포함돼 있습니다. 기존 '지금 인기'는 앱 구매만 셌는데, 이대로 갈까요?
2. **조회수 하루 지연**: 조회 데이터 일부가 하루 늦게 도착해 계산 시점엔 약간 덜 잡힙니다(순위 영향 제한적) — 수용?
3. **태그 시트 타이밍**: 시트 동기화와 랭킹 계산이 동시에 돌 수 있어, 드물게 하루 전 태그로 계산될 수 있음 — 수용?
4. **배치 실패의 파급**: 랭킹 배치가 실패하면 같은 파이프라인의 일일 리포트 발송도 같이 멈춥니다(기존 관례와 동일) — 의도대로 둘까요?
5. **무의미 조합**: '지금 인기' 칩에 태그를 붙여 보내는 식의 무의미한 데이터도 조용히 저장됩니다 — 거절로 바꿀까요?
6. **배정 기록 정밀도**: 실험 배정 로그에서 "핵클 호출 실패로 수동군 처리된 사용자"를 구분할 필드가 필요한지 — 분석 설계와 함께 확인.
7. **운영 순서 고정**: 반영 순서는 반드시 "배포 → DB 마이그레이션 → 스위치 on"입니다. 순서를 어기고 스위치를 먼저 켜면 홈 화면 에러 — 런북에 못 박아둘 필요.
