## 🔴 Blocker (설계/구현이 갈라짐)

### B1 · 노출 슬롯 수(상위 N)가 전 문서에서 미정의 — 랭킹의 출력 길이가 부유

- **위치**: data-measurement-plan.md §6 T1 `섹션당 후보 ≥ 노출 슬롯×3` (line 78) / requirements.md FR-V1 (line 65) / s2 §개념모델 `정렬·상위 N`(line 23) / readme.md §5 `Ranker(정렬 전략)`(line 45).
- **모호한 표현**: "노출 슬롯", "상위 N" — `N`의 값도, 출처(섹션마다 다른가? 칩 UI가 정하나? config 상수인가?)도 어디에도 없다. s2가 "상위 N"이라 적었지만 readme/requirements/data-plan 어디에도 N이 이월되지 않았다.
- **해석A vs 해석B**:
  - **개발자 A (데이터)**: "랭킹 결과는 섹션당 전체 후보를 score DESC로 적재, cap은 서버가 서빙 시 자른다" → 적재 테이블에 섹션당 50+ row.
  - **개발자 B (데이터)**: "노출 슬롯 = 고정 10개로 보고 rank ≤ 10만 적재" → 섹션당 10 row. 그리고 **T1 합격선 `후보 ≥ 슬롯×3`을 A는 슬롯=알 수 없음이라 판정 불가, B는 30으로 판정** → 같은 데이터로 한쪽은 PASS, 한쪽은 측정 불가.
  - **서버 C**: `featuredSkillsTabs`가 칩별로 노출 개수를 이미 갖고 있다고 가정하고 N을 무시 → 데이터가 적재한 cap과 서버 cap이 충돌.
- **다음 단계 영향**: T1(FR-V1)이 정량 판정 불가 → "후보 풀 비어있지 않음"(수용기준 1)을 통과시킬 객관 기준 부재. 적재 테이블 row 수·서빙 cap·`capFeaturedSkillsTabData`(s1 §7 참조) 동작이 전부 N에 종속.
- **제안**: **섹션별 노출 슬롯 수**를 명문화 — (a) 전 섹션 공통 상수인지 섹션별인지, (b) 칩 UI(`featuredSkillsTabs`)에 이미 정의된 값을 SSOT로 쓰는지 신규 정의하는지, (c) 적재는 전체 후보인지 cap된 상위 N인지. 이게 정해져야 T1의 `슬롯×3`이 수치가 된다.

### B2 · norm() 정규화 방식 미정 — 같은 점수가 방식 따라 다른 순위

- **위치**: data-measurement-plan.md §5 `norm() = 0~1 정규화(방식은 min-max/percentile 중 S5/구현에서 확정)`(line 70) / §9 SQL 주석 `norm() = min-max/percentile (구현 확정)`(line 117) / readme.md §5 `각 항목 0~1 정규화`(line 48) / requirements.md FR-R3 `(각 0~1 정규화)`(line 27).
- **모호한 표현**: "0~1 정규화" — min-max인지 percentile(rank 기반)인지, **정규화 모집단이 무엇인지**(전체 스킬? 섹션 내 후보만? 당일만? 7일?), 분포 꼬리(상위 1개 outlier)를 어떻게 처리하는지 전부 미정.
- **해석A vs 해석B**:
  - **개발자 A**: `min-max(전체 스킬 풀)` → 구매수 1위 스킬이 buy7 norm=1.0을 독식, 나머지는 0에 깔림 → popularity_score가 사실상 구매수 단일 지표가 됨(0.35 가중치가 의미를 잃음).
  - **개발자 B**: `percentile_rank(섹션 내 후보)` → 같은 스킬이 "재물운 섹션에선 norm 0.9, 실시간 섹션에선 0.3"으로 **섹션마다 다른 점수** → 동일 스킬의 절대 인기가 섹션별로 뒤집힘.
  - 두 구현 모두 FR-R3 "0~1 정규화"를 만족하지만 **최종 rank 목록이 완전히 다르다.** 이게 곧 B 랭커의 정체성이므로 A/B 실험 결과 자체가 norm 선택에 좌우된다.
- **다음 단계 영향**: B 랭커의 핵심 출력. T4 행동정합(섹션→구매 전환)·골든셋 precision(T2)이 전부 norm 선택에 종속. "동일 후보 풀에 2종 목록 산출"(수용기준 2)에서 B 목록이 비결정적.
- **제안**: norm()을 (a) 함수(min-max vs percentile/rank vs z-score+clip), (b) **모집단 스코프**(전체 노출가능 스킬 vs 섹션 후보 vs 당일), (c) outlier 처리(상한 클립·log 변환 여부)까지 명문화. "S5/구현에서 확정"으로 미루면 데이터 엔지니어가 임의 선택.

### B3 · 'variant 주입점'이 파라미터/헤더/유저속성 중 무엇인지 정의 없음

- **위치**: requirements.md FR-AB2 `외부 신호로 주입받는 variant 주입점이 깔끔히 분리`(line 38) / FR-API3 `variant(A/B)에 따라 2종 목록을 경우에 따라 다르게 서빙`(line 58) / readme.md §7 `variant 주입점 분리`(line 71) / s2 §A/B `그 신호의 출처·구현은 후속 단계 책임`(line 44).
- **모호한 표현**: "외부 신호", "주입점" — 신호의 **형태**(요청 쿼리파라미터? HTTP 헤더? 유저 엔티티 속성? 핵클 키? 미들웨어 컨텍스트?)도, **주입 위치**(API 경계? 서비스 레이어? 적재 시점에 이미 A/B 둘 다 적재?)도 미정. s2가 "출처·구현은 후속"이라 명시적으로 비워뒀다.
- **해석A vs 해석B**:
  - **개발자 A (서버)**: 주입점 = `getFeaturedSkillsTabData(userId, variant)` 서비스 시그니처에 `variant` 인자 추가, 호출부는 일단 항상 'A' 하드코딩. → 적재 테이블은 A/B 컬럼 분리, 런타임에 select.
  - **개발자 B (서버)**: 주입점 = 컨트롤러에서 헤더/쿼리 `?ranker=B` 파싱 → DTO에 실어 전달. → API 시그니처·DTO가 바뀜, 클라이언트가 헤더를 보내야 함(FR-API4 "클라이언트 동작 유지"와 충돌).
  - **개발자 C (데이터)**: "주입점은 서버 책임"이라 보고 적재 테이블에 variant 차원을 안 넣음 → 서버가 B를 서빙하려는 순간 데이터가 없음.
- **다음 단계 영향**: 수용기준 3("variant 주입점을 경유해 A/B 목록 서빙, 버킷팅 미구현 상태에서 주입점만 동작")이 **무엇이 'PASS'인지 정의 불가** — 주입점의 인터페이스 모양이 없으면 QA가 검증 케이스를 못 쓴다. 서버↔데이터 적재 스키마 계약(variant 차원 유무)이 여기서 갈림.
- **제안**: S5에서 (a) 신호 형태(권장: 서비스 레이어 파라미터, 클라이언트 무변경 유지 — FR-API4와 정합), (b) **2종 목록을 적재 시점에 둘 다 적재할지 / 런타임 분기할지**, (c) 주입점의 함수 시그니처 또는 인터페이스를 명문화. "후속"으로 미루더라도 **인터페이스 모양**은 이번에 못 박아야 수용기준 3을 검증 가능.

### B4 · '섹션' vs '칩(targetSection)' vs '기획 섹션명 12종'의 매핑이 어디에도 없음

- **위치**: readme.md §6 표 "인기스킬 섹션 12종"(사주·타로·재물운 …, line 55-63) ↔ s1 §5 칩 `targetSection` 화이트리스트 12종(`recentPurchasedSkills`, `recommendedTagSkills`, `freeTagSkills`, `popularTagSkills`, `categorySkills` …, line 63) ↔ s1 §1 `HomeSectionFeaturedSkillsTab.targetSection`/`targetSectionTag`(line 22). requirements.md FR-F3 `12개 섹션 각각에 대해 필터 규칙을 정의`(line 15).
- **모호한 표현**: "섹션"이라는 단어가 **세 가지 다른 것**을 가리킨다 — ① 기획 섹션명 12종(사주/재물운/재회 …, 의미 라벨), ② 칩의 `targetSection` enum 12종(`recommendedTagSkills`/`freeTagSkills` …, 코드 섹션 타입), ③ `targetSectionTag`(태그 필터). **①과 ②는 둘 다 "12종"이지만 1:1 대응이 아니다** — ②는 같은 `recommendedTagSkills` 타입에 `targetSectionTag`만 바꿔 여러 기획 섹션(사주/타로/재물운)을 표현할 수 있다(s1 §5 line 60: 태그 섹션 9개가 `recommendedTagSkills`/`freeTagSkills`/`popularTagSkills` 셋에 몰려 있음).
- **해석A vs 해석B**:
  - **개발자 A**: "FR-F3의 12 섹션 = 칩 12개" → `HomeSectionFeaturedSkillsTab` row 12개에 각각 필터식 1개를 1:1로 박는다.
  - **개발자 B**: "12 섹션 = 기획 의미 라벨, 실제 칩은 `targetSection`+`targetSectionTag` 조합" → 필터식을 `(targetSection, targetSectionTag)` 키에 건다. **A의 적재 키(칩 seq)와 B의 적재 키(타입+태그)가 달라 서버↔데이터 조인이 깨진다.**
  - 한쪽은 "재물운"을 독립 섹션 타입으로, 다른 쪽은 `recommendedTagSkills`+`targetSectionTag='재물'`로 구현 → **같은 화면을 다른 데이터 모델로 산출.**
- **다음 단계 영향**: FR-B2 적재 산출물 `섹션 → [menu_seq,rank,score]`의 **"섹션" 키 타입이 미정**(칩 seq? 기획 섹션명? targetSection enum? (targetSection,tag) 튜플?). 이 키가 안 정해지면 reverse-ETL 스키마·서버 조회 키·`featuredSkillsTabs` 결합이 전부 어긋난다. FR-F7 "값 바인딩"도 이 매핑 위에서만 의미.
- **제안**: **3계층 매핑표를 명문화** — `기획 섹션명(12) ↔ targetSection enum ↔ targetSectionTag ↔ 필터식 ↔ 적재 키`. 특히 적재 산출물의 PK가 "칩 seq"인지 "(targetSection, targetSectionTag)"인지 "논리 섹션 ID"인지 못 박을 것. 이게 S5의 1순위.

### B5 · 적절성 합격 기준(T1~T3)이 전부 '초안/미합의' — 검증 통과 판정 불가

- **위치**: data-measurement-plan.md §6 표 — `섹션당 후보 ≥ 노출 슬롯×3, 미태깅률 ≤ 10%`(line 78) / `precision ≥ 0.8 (초안)`(line 79) / `충돌율 ≤ 15% (초안)`(line 80), 컬럼 헤더 자체가 `합격 기준(초안 — DF-S4-4, 기획 합의 전)`. requirements.md FR-V5 `합격 기준은 data-measurement-plan.md에 정의하고 기획과 합의한다`(line 69) — **합의 미완**. readme.md §11 `DF-S4-4 | 적절성 합격 기준 정량 임계 | data-measurement-plan 초안`(line 115).
- **모호한 표현**: 모든 임계가 "(초안)"·"기획 합의 전". 수치는 적혀 있으나 **계약이 아니라 placeholder**. 게다가 T1의 `슬롯×3`은 B1(슬롯 미정)에 종속, T2 골든셋은 "사람 라벨 스킬×섹션"인데 **골든셋이 존재하지 않고 누가 언제 만드는지 미정**, T3 "충돌율"의 분모(어떤 교차쌍을 충돌로 카운트하는지)가 미정.
- **해석A vs 해석B**:
  - **개발자 A**: "초안 수치를 그대로 합격선으로 적용" → precision 0.79면 FAIL 판정, 출시 보류.
  - **개발자 B**: "(초안)이니 합의 전엔 판정 보류, 측정만 리포트" → 같은 0.79를 "참고치"로 통과시킴. **동일 결과로 출시 가/불가가 갈림.**
  - T2: A는 자기가 50개 라벨링한 골든셋으로 precision 산출, B는 다른 사람이 200개 라벨링 → **골든셋 정의·크기·라벨러가 없어 precision 자체가 재현 불가.**
- **다음 단계 영향**: 수용기준 4("적절성 사전검증 결과가 합격 기준을 충족")가 **충족 여부를 판정할 기준이 없음** → QA·데이터가 "PASS"를 선언할 근거 부재. /architect가 검증 게이트를 설계 못 함.
- **제안**: S5/기획 합의 전 최소한 — (a) 각 임계를 "합의됨/잠정" 명시 + 합의 책임자·기한, (b) **골든셋의 정의**(라벨 단위·최소 크기·라벨러·셋 위치), (c) T3 충돌율의 정확한 분자/분모(어떤 (topic, content_type) 쌍을 "충돌"로 세는가)를 명문화. 임계가 placeholder인 채로는 수용기준 4를 삭제하거나 "측정만"으로 강등해야 함.

### B6 · '신규 인기' 섹션 윈도우 N vs 신규부스트 30일 — 같은가 다른가가 미해결

- **위치**: s4 §3 `신규 인기 | 등록일(menu_create_at_date) ≤ N일 ∩ 인기 | S3 신규부스트와 동일 원천 — N 정합 필요`(line 44, **s4가 스스로 "N 정합 필요"라 플래그**). readme.md §6 `G1 시간성 | temporal / 등록일 | 실시간 인기 · 신규 인기`(line 59) — N 언급 없음. requirements.md에는 "신규 인기" 섹션의 필터 N이 **아예 없음**(FR-F3는 "12 섹션 정의"만, FR-R5는 신규부스트 30일만). data-measurement-plan.md도 신규부스트 30일만(line 31).
- **모호한 표현**: "신규 인기" 섹션의 후보 자격선 `N일`과, 랭킹 점수의 "신규부스트 30일"이 **같은 30일인지 다른 값인지** 계약 문서 어디에도 결론이 없다. s4가 "정합 필요"라 적은 것이 그대로 미해결로 상위 문서에 승격됐다.
- **해석A vs 해석B**:
  - **개발자 A**: "신규 인기 섹션 = 신규부스트와 동일 30일" → 등록 30일 이내만 후보. 그런데 이 섹션은 **전원이 신규부스트 +0.05를 받으므로** 부스트가 변별력 0(전원 동일 가산) → 사실상 일반 인기 정렬과 같아짐.
  - **개발자 B**: "신규 인기 = 7일(더 빡센 신규)" → 후보 풀이 30일 A의 1/4. **두 구현의 '신규 인기' 섹션 멤버가 전혀 다름.**
  - 게다가 "∩ 인기"의 "인기"가 popularity_score인지 구매수인지 미정(B7 참조) → 신규 섹션의 정렬 기준도 갈림.

### M4 · '실시간 인기'의 '실시간'과 일배치(D+1)의 충돌 — 윈도우 정의 미정

- **위치**: s4 §3 `실시간 인기 | … ranker=짧은 윈도우 인기 | "실시간"은 랭커 윈도우 문제일 수 있음 → S3/S5`(line 43) ↔ FR-R4 `점수는 7일 윈도우로 산출하며 일배치로 갱신`(line 28) ↔ NFR-2 `노출 목록은 일배치(D+1) 기준`(line 76) ↔ readme.md §5 `7일 윈도우`(line 48).
- **모호한 표현**: "실시간 인기" 섹션이 "짧은 윈도우"를 쓴다고 s4가 적었지만, 그 윈도우 길이도(1일? 1시간? 7일?), **단일 popularity_score(7일 고정)와 어떻게 다른지도** 미정. 그리고 "실시간"인데 일배치 D+1 → 최대 ~24h+ 지연 → 명칭과 신선도가 모순.
- **해석A vs 해석B**:
  - **개발자 A**: "실시간 인기 = 다른 모든 섹션과 동일한 7일 popularity_score" → "실시간"은 라벨일 뿐, 신규/사주 섹션과 윈도우 동일. 그럼 s4의 "짧은 윈도우"는 무시됨.
  - **개발자 B**: "실시간 = 1일 윈도우 별도 랭킹" → popularity_score 공식을 1일로 재계산하는 **두 번째 점수 파이프라인** 필요 → 적재 스키마에 윈도우 차원 추가, FR-R4(7일 고정)와 충돌.
  - 두 구현의 "실시간 인기" 섹션 순위가 완전히 다르고, B는 FR-R4를 위반.
- **다음 단계 영향**: FR-R4가 "7일 단일 윈도우"를 MUST로 박았으므로 B 구현은 요구사항 위반인데, s4는 "짧은 윈도우"를 시사 → 구현자가 어느 쪽이 계약인지 모름. 적재 스키마(윈도우 차원 유무)가 갈림.
- **제안**: "실시간 인기" 섹션이 (a) 공통 7일 점수를 그대로 쓰는 라벨인지, (b) 별도 단기 윈도우를 갖는지 명시. (b)면 FR-R4를 "기본 7일, 단 실시간 섹션은 N일"로 예외 명문화하고 적재 스키마에 윈도우를 1차원 추가할지 결정. "실시간"이라는 명칭과 D+1 신선도의 괴리도 명시(허용 가능한가).

### M5 · '인기'(∩ 인기)의 정렬 기준이 섹션마다 같은 popularity_score인지 미정

- **위치**: s4 §3 `신규 인기 | … ∩ 인기`(line 44), `실시간 인기 | … ranker=짧은 윈도우 인기`(line 43) ↔ readme.md §5 Ranker B = popularity_score 단일 공식(line 47-53) ↔ FR-R2 `B = PopularityScoreRanker v1 점수 공식`(line 26).
- **모호한 표현**: G1 섹션 필터식에 등장하는 "∩ 인기"의 "인기"가 **B 랭커의 7일 popularity_score와 동일물**인지, 아니면 별도 "인기" 개념(구매수만? 조회수만?)인지 미정. s4는 "짧은 윈도우 인기"라 적어 popularity_score(7일)와 다른 무언가를 암시.
- **해석A vs 해석B**:
  - **개발자 A**: "∩ 인기"는 필터가 아니라 **랭커가 popularity_score로 정렬**하는 것이고, 필터식의 "∩ 인기"는 군더더기 → 무시. → 모든 섹션이 동일 popularity_score로 정렬.
  - **개발자 B**: "∩ 인기"를 **필터 조건**으로 해석(예: 구매수 > 임계) → 신규 섹션에서 인기 임계 미달 스킬을 후보에서 제외 → A보다 후보 좁음, 임계값도 미정.
- **다음 단계 영향**: 필터(후보 선정)와 랭커(정렬)의 책임 경계가 "인기"라는 단어에서 흐려짐 — 이는 S2 개념모델(Filter+Ranker 분리)의 근간을 흔든다. FR-R2의 단일 공식이 모든 섹션에 적용되는지 불명.
- **제안**: 필터식에서 "∩ 인기"를 제거하고(정렬은 랭커 책임임을 명확화) 또는 "인기"를 정량 술어로 치환. **"모든 섹션이 동일한 popularity_score(7일)로 정렬되는가, 섹션별 랭커가 다른가"를 명문화** — 현재 readme는 B 랭커 1종이라 했지만 s4의 "짧은 윈도우 인기"는 섹션별 랭커를 시사(B1·M4와 연쇄).

## 종합

**모호성 렌즈 단일 기준에서도 Blocker 6 / Major 6 / Minor 8.** 이 프로젝트는 "필터 + 랭킹"을 선언하지만, **세 개의 미정의가 시스템 전체를 부유시킨다**:

1. **출력 길이(노출 슬롯 N) 미정 (B1)** — 랭킹의 결과 길이, T1 합격선, 서빙 cap, 적재 row 수가 전부 이 하나에 매달려 있는데 어디에도 값이 없다. 가장 먼저 못 박아야 다른 finding(M1·M4·m8)이 풀린다.
2. **B 랭커의 결정성 미정 (B2 norm + M5 '인기' + M4 윈도우)** — norm 방식·모집단, "∩ 인기"의 의미, "실시간"의 윈도우가 미정이라 **B 랭커가 산출하는 목록 자체가 비결정적**. A/B 실험의 B가 무엇인지 정의되지 않은 채 수용기준 2가 서 있다.
3. **'섹션' 단어의 3중 의미와 적재 키 미정 (B4 + m8)** — 기획 섹션명/`targetSection` enum/`targetSectionTag`/적재 PK가 매핑 없이 같은 "섹션"으로 불려, 서버↔데이터 적재 조인이 구현자마다 다른 키로 깨진다.

추가로 **B3(variant 주입점 형태 미정)**·**B5(합격 기준 전부 초안)**·**B6(신규 N vs 30일)**가 각각 수용기준 3·4와 G1 섹션을 검증 불가/구현 분기 상태로 만든다.

**구조적 관찰**: 근거 문서(s4)가 스스로 "확인 필요"·"모호"·"N 정합 필요"라 플래그한 항목들(1:1 상담 축, 재물운 축, 결혼궁합 분리, 실시간 윈도우, 신규 N)이 **모호성 표지를 단 채로 상위 계약(requirements.md)에서 MUST로 승격**됐다. MUST는 "구현자가 반드시 따라야 할 확정 계약"인데 그 내용이 가설/미확인이면, 구현자는 가설을 확정으로 오인해 제각각 구현한다 — 이것이 이 문서군의 핵심 모호성 패턴이다.

**S5 진입 전 최소 확정 권고**: B1(슬롯 N) → B4(섹션/적재 키 3중 매핑) → B2(norm 방식·모집단) → B3(주입점 인터페이스 모양) 순으로 4개를 못 박으면 나머지 Major는 대부분 따라 풀린다. B5(합격 기준)·M6(KPI 산식)는 기획 합의 트랙으로 병렬 처리하되, **합의 전까지 해당 MUST를 "조건부"로 강등**해 구현자의 가설 확정 오인을 차단할 것.
