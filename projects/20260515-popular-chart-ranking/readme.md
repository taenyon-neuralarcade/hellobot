## 1. 배경

- 홈탭 **인기스킬 섹션**은 현재 운영자가 수작업으로 세팅한 고정 스킬 목록을 모든 사용자에게 동일하게 노출한다. 운영자가 갱신하기 전까지 동일 콘텐츠가 고정 노출된다.
- 홈탭 개선 Phase #1(타 스쿼드)에서 **노출 컨테이너**(`featuredSkillsTabs` 칩/탭 UI, [hellobot-server PR #2414](https://github.com/thingsflow/hellobot-server/pull/2414), 2026-05-18 머지)는 이미 구축됐다. 단 이는 "그릇"일 뿐, 칩이 가리키는 각 섹션의 내용은 여전히 운영 DB 수동 큐레이션이다.
- **Phase #2(이 프로젝트)** = 그 그릇 뒤에 **필터링 + 랭킹**의 실체를 채워, 매일 자동으로 노출 콘텐츠를 결정하는 것.

## 2. 문제 정의

| 문제 | 영향 |
|------|------|
| 수동 큐레이션 → 최신성 결여 | 운영자 갱신 전까지 동일 콘텐츠 고정, 인기 변화 미반영 |
| 운영 부담 | 섹션마다 사람이 스킬을 골라 등록 |
| 섹션-콘텐츠 적합성 미검증 | "사주 섹션에 사주 콘텐츠"가 데이터로 검증된 적 없음 |

## 3. 목표

1. **노출 랭킹 자동화** — 매일 스킬별 인기 점수를 계산하고, 그 순서로 섹션 콘텐츠를 자동 노출한다.
2. **섹션별 필터링** — 콘텐츠 메타(주제·의도·시간성·카테고리)로 각 섹션에 맞는 후보 스킬을 자동으로 거른다.
3. **효과 검증(A/B)** — 자동화(B)와 현행 수동 노출(A)을 비교 가능한 구조로 만든다.
4. **확장 비파괴** — 장기적으로 목적·콘텐츠 특성별 다중 랭킹 알고리즘이 공존할 수 있는 구조를 깨지 않는다.

## 4. 성공 지표 (요약 — 상세 [data-measurement-plan.md](data-measurement-plan.md))

| 구분 | 지표 |
|------|------|
| **Input** | 섹션 CTR (노출 → 클릭) |
| **1차 성과** | 전환율(클릭 → 구매) · 신규 진입율 · 구매 상품 수 |
| **최종 판정** | **총 매출**(가드레일) — ⚠️ 저가 콘텐츠 노출 비중↑ 시 전환율↑·총 매출↓ 가능 → B가 전환율 우위라도 **총 매출 열위면 불합격** |
| **A/B** | 동일 섹션에서 자동화(B) vs 현행(A) 비교 (A/B 버킷팅은 후속 단계) |

## 5. 핵심 개념 — 자동화(B) vs 현행(A)

- **A `AsIsRanker`** (대조군) = **현행 서버의 섹션 목록 산출 방식 전체를 그대로**(어드민 수동 큐레이션·pinned·우선순위 등 변경 없이 노출). ⚠️ A는 **자체 선정 로직**을 가지므로 이번에 구축할 B 후보 풀에 없는 스킬도 포함될 수 있다 — **B의 신규 필터 풀과 동일하지 않다**(C-1 결정). 비교는 *같은 풀의 정렬 차이*가 아니라 **섹션 단위 행동 KPI**(노출→클릭→구매→총 매출).
- **B = 신규 Filter + PopularityScore Ranker** (실험군):
  ```
  B 섹션  =  Filter (메타 조건 → 후보 풀)  +  PopularityScoreRanker (점수 정렬)
  ```
  - **기본 풀 조건(base, 전 섹션 공통 선결 필터)** = **오리지널(`original_type='original'`) ∧ 유료 ≥750원**(✅ 확정). 모든 섹션이 이 풀에서 출발. ⚠️ 현행 '지금 인기'는 minPrice 60(하트)이라 별도 임계 — 750원을 BQ 마트 가격 필드 단위로 바인딩(/dev-data·D-5).
  - **Filter** = base ∧ 섹션 메타축 조건. 메타축: topic / intents / chatbot_content_type (+ targets·temporal은 1차 미사용). (→ [s4](planning/s4-filtering-tagging.md))
  - 'Filter + Ranker' 분해는 **B(및 향후 variant) 기준 모델** — A는 현행 산출을 블랙박스로 보존.
- **랭킹 점수 공식 (v1, 골격)** — 각 항목 정규화 후 가중합 (7일 윈도우):
  ```
  popularity_score = 구매수×0.35 + 조회수×0.1 + 전환율×0.2
                   + 긍정평가비율×0.15 + 매출×0.15 + 신규부스트×0.05
  ```
  - 긍정평가비율 = **`1 − 💩비율`**(별점 부재로 재정의, 0~1), 신규부스트 = **출시일(`open_date`) 30일 이내** 가산(D-4).
  - ✅ 가중치 α=**6항목 확정**, 매출 외부채널=**포함 수용**(CL-07). ⚠️ `norm()` 방식·모집단은 **S5 위임**(CL-04, 미정 시 순위 비결정; 권장=percentile·전체풀).

## 6. 대상 — 인기스킬 섹션 (1차 5종 + 최종단계 7종)

> 1차 = **적합 태깅이 현존하는 5섹션**만 자동화. 커플/솔로 대상 섹션 + 신규 intents 태깅이 필요한 7섹션은 **최종단계로 보류**(D-1·D-2·D-3, 적합 태깅 부재).

### 1차 자동화 (5) — 기본 풀 조건(base) ∧ 아래

| 섹션 | 그룹 | 1차 필터 조건 |
|------|------|----------------|
| 실시간 인기 | G1 | (섹션축 없음 — 기본 풀 전체) + 7일 점수(명칭은 라벨, CL-09) |
| 신규 인기 | G1 | **출시일 `open_date` ≤ 6개월**(N 기본 6, 가변·C-2/D-6) |
| 사주 | G2 | `chatbot_content_type = 사주` |
| 타로 | G2 | `chatbot_content_type = 타로` |
| 재회 | G3 | `intents ∋ 재회` |

### 최종단계 보류 (7) — 적합 태깅 부재

| 섹션 | 보류 사유 |
|------|-----------|
| 커플 궁합 | 커플 대상 구분 태깅 부재 — `intents∋궁합` 있으나 보류(D-1) |
| 커플 결혼 궁합 | 커플 대상 구분 태깅 부재 — `intents∋결혼` 있으나 보류(D-1) |
| 솔로 애정운 | 솔로 대상 + 애정 intents 태깅 부재(D-1) |
| 솔로 결혼운 | 솔로 대상 + 태깅 부재(D-1) |
| 짝사랑/썸 | 적합 태깅 부재(D-1) |
| 재물운 | 적합 태깅 부재(D-1) |
| 1:1 상담 | 적합 태깅 없음(D-3) |

> ⚠️ **커플/솔로 대상 섹션은 `intents`가 있어도 대상 구분 태깅 부재로 전부 최종단계**(D-1, 사용자 2026-06-06). ⚠️ targets·temporal 축은 1차 섹션 필터식 미사용(CL-20).

### 6.1 섹션 ↔ 컨테이너 매핑 (✅ CL-01 코드 확인 2026-06-05)

- 논리섹션은 PR #2414 `featuredSkillsTabs` 칩에 **N:1로 매핑**: 대부분 `targetSection=recommendedTagSkills` + `targetSectionTag`(한글 태그명)로 구분. **신규 섹션타입 불요.**
- **적재/조회 키 = 복합키 `(targetSection, targetSectionTag)`** (FR-API6). 노출 슬롯 N = 7(vertical)/8(horizontal) (FR-API5).
- 실제 태그값 목록·`popularTagSkills` 운영현황은 **운영 DB 덤프** 필요(= FR-F7 값 바인딩). 🚩 `popularTagSkills`는 어드민 enum 누락(운영자 등록 불가) — '신규 인기' 등이 쓰려면 수정.
- ⚠️ **컨테이너 키 매핑(N:1)은 코드 확인됨. 단 섹션→태그 조건식(필터 스펙)은 S5에서 칩 구체화와 함께 재논의** — 일부 섹션은 다축 복합 조건이 될 수 있음(기획 피드백).

## 7. 범위

**포함 (이번 스펙)**
- 랭킹 점수 공식 1종(v1) 정의 + 일배치 산출 + AsIs 대조 랭커.
- **1차 5개 섹션의 필터 규칙 신규 정의**(현재 미존재) — 기본 풀 조건 ∧ 메타축 조건식. (7개 섹션은 최종단계)
- 섹션별 `필터 → 랭킹` 결과 목록의 산출·적재 파이프라인.
- A(현행 산출 전체)·B(신규 필터+랭커) **2종 목록을 산출**하고, 어느 것을 서빙할지 외부 신호로 주입받는 **variant 주입점** 분리. (A·B는 동일 후보 풀이 아님 — C-1)
- 적절성 **사전 검증**(커버리지 T1) 설계.

**제외 (확장 대비만, 이번 비목표)**
- 다중 랭커의 실제 운영(인터페이스 추상화까지만).
- 신규 섹션·신규 필터축 추가(현재 정의된 섹션 조건까지).
- A/B 버킷팅·실험 배포 구현 — **랭킹 알고리즘 적용 이후 별도 단계**(컨테이너 핵클 키18 A/B 종료 후 순차).
- 임시 태그의 정식 메타 승격 — **하이브리드 후속**(1차는 임시 태그 그대로 사용).
- **jp.hellobot 적용 — 1차 KR 전용**, jp 홈탭 인기 섹션은 현행 AsIs 유지(CL-11, NFR-7).
- **필터 규칙 어드민 편집 UI — 1차 시드/DB, 후속**(CL-16). 적절성 T2·T3 검증도 후속(CL-15).
- 섹션 간 전역 dedup — 1차는 중복 허용(CL-13). 개인화·국가별 분기·운영자 오버라이드(단 현행 pinned는 유지 — CL-14).
- **7개 섹션(커플 궁합·커플 결혼 궁합·솔로 애정운·솔로 결혼운·짝사랑/썸·재물운·1:1 상담) — 최종단계**(커플/솔로 대상 구분 태깅 부재·적합 태깅 부재, D-1·D-3).

- **D4** 인기스킬(`recentPurchasedSkills`)은 이미 구매량 반자동 — 자동 랭킹은 이를 일반화하는 방향.
