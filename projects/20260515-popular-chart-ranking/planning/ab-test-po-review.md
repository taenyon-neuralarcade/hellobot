## 0. 요약 (TL;DR)

1. 사용자 제안(2군: 기존 상태 vs 칩 섹션+자동 랭킹)은 **목표 1(섹션의 통합 효과)은 답하지만, 목표 2(자동 랭킹 vs 수동 큐레이션)는 구조적으로 답할 수 없다** — UI 효과와 랭킹 효과가 혼재(confounded)되기 때문.
2. **권장 = 3군 실험(중첩 구현)**: L(기존 legacy) / C-M(칩+수동 큐레이션) / C-A(칩+자동 랭킹). 기존 Hackle 키18을 컨테이너 게이트로 **유지·흡수**하고, 칩 수신군 내부에 ②랭킹 키(신규)를 중첩한다. 두 목표를 한 실험 윈도우에서 모두 답하고, "수동 칩은 지지만 자동 칩은 이기는" 시나리오까지 식별(헤지)한다.
3. 단, 권장안 확정 전 **선행 확인 2건**: ⚠️ (i) 키18의 현 운영 단계(테스터 only인지, 공개 A/B 진행 중인지 — 미확인, 감사 §5) (ii) 3군 분할 검정력(분석 문서). 키18이 이미 판독 직전이면 순차(옵션 c)가 우월할 수 있음.
4. **1차 판정 지표는 유저 레벨 net lift** (홈탭 방문자→구매자 전환율, 유저당 구매 수·매출)로 재정의해야 함 — L군에는 칩 섹션이 없어 "섹션 경유" 지표가 정의되지 않고, 섹션 경유 증가가 타 지면 잠식이면 net은 0이기 때문. 섹션 CTR·섹션 경유 구매는 **B군 내 메커니즘 진단 지표**로 강등.
5. **총매출 가드레일은 확정 정책 그대로 유지** — 전환율 우위라도 유저당 매출 열위면 불합격 (dmp §3).
6. 목표 2의 판정 기준은 superiority가 아니라 **비열등(non-inferiority)으로 제안** — 자동화의 가치에는 매출 외에 운영 부담 제거·최신성이 포함되므로(PRD §2 문제 정의), C-A가 C-M에 비열등 + 가드레일 통과면 ship 가능. (⚠️ 사용자 승인 필요)

## 1. 사용자 제안 요약과 가설 분해

### 1.1 제안 원문 요지

> 기존 phase1의 UI 추가를 포함하여 — **A**: 칩 선택형 인기스킬 섹션이 없는 기존 상태 / **B**: 칩 선택형 인기스킬 섹션 + 노출 항목 랭킹 자동화 적용.
> 목표 1: 자동 랭킹 + 탐색 편한 섹션이 추가되면 섹션 경유 콘텐츠 진입·구매자 수가 늘 것이다.
> 목표 2: 랭킹에 의해 노출되는 스킬이, 수동 선정·변동 적은 스킬 세트보다 구매(횟수·구매자 수)·매출이 좋을 것이다.
> (홈에 다른 지면도 있지만, 정말 유용한 섹션이라면 이 섹션을 통한 구매 지표가 늘 것이다.)

### 1.2 "기존 상태"의 정밀 정의 — A는 빈 홈이 아니다

⚠️ **용어 충돌 주의**: 제안의 "A"는 프로젝트 문서의 "A 랭커(AsIs)"와 **다른 대상**이다. 프로젝트 문서의 A 랭커(C-1·FR-R6)는 *칩 컨테이너 안의 현행 수동 큐레이션 순서*를 뜻하고, 제안의 "A"는 *칩 컨테이너 자체가 없는 상태*를 뜻한다. 혼선 방지를 위해 본 문서는 군 명칭을 새로 정의한다:

| 군 | 명칭 | 실체 ([s5-asis-serving-analysis.md](s5-asis-serving-analysis.md) §1~2 코드 확인) |
|----|------|------|
| **L** | Legacy (제안의 "A") | `featuredSkillsTabs` 미노출. 대신 **legacy `recentPurchasedSkills` 단일 리스트** 노출 — 오리지널 ∧ 유효가 ≥60하트 ∧ 앱구매, **최근 구매순(recency)·분 단위 배치 갱신·pinned 선두**. 키18의 A arm과 동일 |
| **C-M** | Chips + Manual (현행 featured = 프로젝트 문서의 "A 랭커") | 칩 컨테이너 + 현행 순서 — 태그 칩=어드민 큐레이션 priority ASC(정적), recent 칩=recency. 키18의 B arm과 동일 |
| **C-A** | Chips + Auto (제안의 "B" = 프로젝트의 "B 랭커") | 칩 컨테이너 + **1차 5섹션**(실시간·신규·사주·타로·재회)에 popularity_score 자동 랭킹. 나머지 칩은 수동 유지 |

핵심: **L은 "인기 콘텐츠가 전혀 없는 홈"이 아니라, 분 단위로 갱신되는 반자동(최근 구매순) 인기 리스트가 있는 홈**이다. 신선도 면에서는 오히려 수동 큐레이션 칩(C-M, 정적 priority)보다 동적인, 약하지 않은 대조군이다. "기존 상태를 이기는 것"의 난이도를 과소평가하면 안 된다.

또 하나 — **홈 전체 구조 (✅ 사용자 확인 2026-06-10, v1.2 정정)**: 칩이 가리키는 섹션들(사주·타로·재회 등)은 칩 도입 전부터 **홈 세로 스크롤에 단독 섹션으로 분산 노출**되어 있고(뒤쪽 섹션은 깊은 스크롤 필요), 칩 컨테이너가 떠도 **이 단독 섹션들은 하단에 그대로 유지**된다:

- **L** = legacy recent 단일 리스트 + 분산 단독 섹션들 (전부 스크롤로 도달)
- **C(-M/-A)** = 칩 컨테이너(recent 슬롯과 상호배타 1-슬롯 스왑 — `home.ts` L227~232) + **하단에 동일한 단독 섹션들 그대로(중복 노출)**

따라서 L↔C 비교의 본질은 "섹션 추가 vs 부재"가 아니라 **"기존 분산 콘텐츠의 상단 모아보기(스크롤 단축) 추가 + recent 슬롯 교체"**다. phase1(칩 UI)의 가치 가설 = 탐색 비용(스크롤) 단축이며, 칩 콘텐츠 자체는 홈에 새로 들어온 것이 아니라 하단과 중복이다.

**처치 범위 정밀화 (C-A)**: 랭킹 주입 대상은 **칩 lazy 경로만**이다 — 홈 1차 응답의 단독 섹션 직접 노출 경로는 미주입([감사 §3](s5-asis-implementation-audit.md) "②랭킹 B의 주입 대상은 lazy 경로만"). 즉 C-A에서도 **하단 단독 섹션들은 수동 그대로**이고, 같은 '사주'가 위 칩(랭킹순)과 하단 단독 섹션(수동순)에서 **서로 다른 목록·순서로 동시 노출**될 수 있다(함의 → §5.8).

### 1.3 가설 분해 — 제안 구조의 혼재(confound) 문제

| 가설 | 내용 | 식별에 필요한 대비 |
|------|------|--------------------|
| **H1 (통합 효과)** = 목표 1 | 칩 섹션+자동 랭킹 패키지가 기존 상태보다 구매자·매출을 늘린다 | **L vs C-A** ← 제안의 2군이 답함 |
| **H1′ (UI/컨테이너 효과)** | 칩 섹션 자체(수동 큐레이션이어도)가 기존보다 낫다 | **L vs C-M** ← **키18이 이미 LIVE로 묻고 있는 질문** |
| **H2 (랭킹 효과)** = 목표 2 | 자동 랭킹이 수동 큐레이션보다 구매·매출이 좋다 | **C-M vs C-A** ← 제안 2군으로는 **식별 불가** |

제안된 2군(L vs C-A)에서 차이가 나와도 그것이 **칩 UI 때문인지 자동 랭킹 때문인지 분리할 수 없다**. 목표 2는 이 비교만으로는 영원히 답해지지 않는다. 반대로 차이가 안 나와도 "UI는 효과 있는데 랭킹이 깎아먹은 것"인지 "둘 다 무효"인지 모른다. 즉 **2군 설계는 목표 1 전용이고, 목표 2는 별도 대비(C-M vs C-A)가 반드시 필요하다.**

## 2. 실험 구조 옵션 비교와 권장안

### 2.1 옵션 정의

- **(a) 2군 통합 (제안 원안)**: L vs C-A. 목표 2는 별도 후속 실험(ship 후 칩 내 수동 vs 자동 holdback)으로 미룸. 구현상 키18의 B arm 의미가 "수동 칩→자동 칩"으로 바뀌므로 **키18 실험으로서의 판독은 종료·재선언** 필요.
- **(b) 3군 (중첩 구현 권장)**: L vs C-M vs C-A. 구현은 신규 3-way 키가 아니라 **키18(컨테이너 게이트) 유지 + 칩 수신군 내부에 ②랭킹 키(신규) 중첩**이 자연스럽다 — 감사 §4-3-1(②variant 신호=신규 키)·§4-3-5(②모집단=①B군 내) 옵션과 정확히 정합. 키18 50/50 유지 시 셀 배분 L=50 / C-M=25 / C-A=25 (⚠️ 배분은 분석 문서에서 최적화 — 키18 배분 변경은 키18 자체 판독 리셋을 수반).
- **(c) 순차 (기존 프로젝트 계획)**: 키18(L vs C-M) 판독·종료 → 칩 100% 공개 → ②랭킹 A/B(C-M vs C-A, 전원 50/50). FR-AB4·S2 결정("컨테이너 키18 종료 후 별도·순차")의 원안.

### 2.2 트레이드오프 비교

| 축 | (a) 2군 통합 | (b) 3군 중첩 | (c) 순차 |
|----|--------------|--------------|----------|

| **리스크 헤지** | 없음 — 실패 시 원인 미상 | **있음** — UI 실패/랭킹 실패 분리, 랭킹 자산 독립 평가 | 부분 — 키18 패배 시 ②가 설 자리(칩) 자체가 사라짐 |

### 2.3 권장안 — **(b) 3군 중첩** ✅

**권장: 옵션 (b) 3군 — 키18을 컨테이너 게이트로 유지(흡수)하고 칩 수신군 내부에 ②랭킹 키를 중첩, L / C-M / C-A를 동시 운영한다.**

근거:

1. **사용자의 두 목표를 한 윈도우에 모두 답한다.** 제안의 헤드라인 질문(L vs C-A)은 직접 대비로 그대로 살아 있고(목표 1), C-M 셀 하나를 추가하는 것만으로 목표 2(C-M vs C-A)가 같은 기간·같은 시즌에서 식별된다. 사용자의 의도를 깎는 게 아니라 **중간 셀 하나로 의도를 완성**하는 구조다.
2. **"수동 칩은 지는데 자동 칩은 이기는" 시나리오에 대한 헤지.** L(legacy)은 분 단위 recency 갱신이라 신선도에서 정적 수동 큐레이션(C-M)보다 유리할 수 있다 — 즉 키18에서 C-M이 패배할 개연성이 실재한다. 순차(c)로 가면 그 순간 칩이 롤백되어 **이 프로젝트의 자동 랭킹은 살 곳(컨테이너)을 잃는다**. 3군이면 C-M이 져도 C-A vs L이 직접 측정되어 있어, "자동 랭킹이 칩을 구하는지"를 알 수 있다. 자동화 랭킹이 정확히 '최신성 결여'(PRD 문제 1)를 고치는 처방이므로 이 시나리오는 충분히 그럴듯하다.
3. **기존 계획·구현과의 정합 비용 최소.** 중첩 구현은 감사 §4-3이 이미 식별한 설계 항목(②variant 신규 신호, 모집단=①B군 내) 그대로다. 키18 메커니즘·게이팅 코드 패턴 재사용(감사 §4-2). 변경되는 것은 "키18 *종료 후* 순차"라는 시퀀싱 결정 하나뿐(→ §7 문서 갱신).
4. **2단 롤백이라는 운영 안전판.** ②키 off → 전원 C-M 복귀(랭킹만 끔), 키18/플래그 off → 전원 L 복귀(컨테이너까지 끔). FR-API3의 "기본값=A(현행), B는 명시 주입"과 정합 — 자동 랭킹은 언제든 무중단으로 꺼진다.

**권장안의 전제 — 선행 확인 2건 (확정 전 필수):**

- ⚠️ **(전제 1) 키18 현 운영 단계 확인** — 감사 §5 미확인 항목: FeatureFlag 2종 값 + 키18 분배 현황. **public-enabled가 아직 OFF(테스터 only)면 공개 A/B 데이터가 아예 없어** 3군을 처음부터 설계하는 데 보호할 sunk 판독도 없음 → (b) 채택이 더욱 자연스럽다. 반대로 이미 공개 A/B가 깊게 진행되어 곧 판독 가능하면, 키18 완주 → (c) 순차가 우월할 수 있다 (목표 1은 키18+②의 합성으로 답함).
- ⚠️ **(전제 2) 검정력** — C-M vs C-A(25/25, 효과 크기 최소로 예상)가 수용 가능한 기간 내 판독 가능한지 → [ab-test-analysis-design.md](ab-test-analysis-design.md) 산정 결과로 확정. 불충분하면 폴백 = **(c) 순차** (각 단계 50/50 — 목표 2 검정력 최상). (a)는 폴백으로도 비권장 — 목표 2를 영구 미답으로 두는 구조이기 때문. (a)를 택하는 경우는 "목표 2를 명시적으로 다음 분기로 미룬다"는 사용자 결정이 있을 때뿐이며, 그 경우에도 키18 종료·재선언 절차가 필요하다.

**모집단 (전 옵션 공통)**: 칩 컨테이너 지원 클라이언트(Android ≥2.43.0 / iOS ≥2.54.0 / 웹)의 KR 홈탭 방문자. 구버전 앱은 게이팅상 무조건 L 경로(키18 분배 자체를 안 탐)이므로 **실험 분석 모집단에서 제외**(§5.3). 버킷 단위 = **유저, 인기스킬 섹션 일괄** (FR-AB3 확정과 정합 — 한 유저는 5섹션 전체가 동일 셀).

## 3. 성공 지표 체계 — 레벨 재정의

### 3.1 왜 재정의가 필요한가

기존 dmp §3 KPI 인벤토리(섹션 CTR → 전환율 → 신규 진입율 → 구매 상품 수 → 총매출)는 **섹션 레벨 중심**이다. 이 체계는 ②(칩 내부 C-M vs C-A) 비교엔 맞지만, L군이 들어오는 순간 두 가지가 깨진다:

1. **L군에는 칩 섹션이 없다** — "섹션 노출·CTR·섹션 경유 구매"가 L에서 정의되지 않는다. (L의 legacy 리스트에 대해 같은 지표를 만들 수는 있으나, 지면 위치·크기·UI가 달라 칩 섹션과의 직접 비교는 사과-오렌지 비교가 된다.)
2. **잠식(cannibalization)을 못 본다** — 사용자도 인지하듯 홈에는 다른 지면들이 있다. 섹션 경유 구매가 늘어도 타 지면(검색·배너·다른 섹션) 구매를 빼앗은 것이면 **net lift = 0**이고, 그 경우 ship할 이유가 없다. 섹션 귀속 지표는 구조적으로 잠식을 성공으로 오인한다.

### 3.2 제안 지표 체계 (실험 판정용)

| 층 | 지표 | 정의 | 적용 |
|----|------|------|------|
| **1차 판정 (유저 레벨 net)** | ① 구매자 전환율 ② 유저당 구매 수 ③ **유저당 매출** | 홈탭 방문(=실험 노출) 유저 기준, **전 지면 합산**(섹션 귀속 아님) | 모든 셀 간 대비 (L vs C-A, C-M vs C-A, L vs C-M) |
| **가드레일 (확정 정책 유지)** | **유저당 매출 비열등** | "전환율 우위라도 총매출 열위면 불합격" (dmp §3 확정) — 실험 단위에서는 유저당 매출로 정규화해 판정 | 모든 ship 판정의 거부권 |
| **운영 가드레일** | 홈탭/lazy 엔드포인트 오류율·지연, C-A 폴백 발동률, 섹션 숨김(CL-10) 발동률 | 실험과 무관하게 경험이 깨지지 않는지 | 모니터링·중단 기준 |
| **메커니즘 진단 (B군 내 — C-M·C-A 한정)** | 섹션 CTR(노출→클릭, CL-02 신규 이벤트), 섹션 경유 전환율·구매·매출 share, 칩별 분해(`(targetSection, targetSectionTag)` 단위), 신규 진입율(⚠️ 정의 미정 CL-26), 타 지면 구매 share 변화(잠식 추적) | "왜 이겼나/졌나"를 설명 | ship 판정엔 미사용 — 해석·iterate 방향 결정용 |
| **조작 점검 (manipulation check)** | C-M vs C-A 노출 목록 괴리도(오버랩·순위 상관) | 자동 랭킹 목록이 수동과 거의 같으면 효과가 없는 게 당연 — "처치가 실제로 달랐는가" | C-A 셀 진단 |

**의사결정 원칙 명문화**: *"섹션 경유 구매 증가는 성공이 아니다. 유저 레벨 net 증가가 1차다."* — 섹션 진단 지표가 좋아도 net이 평평하면 잠식으로 해석하고 ship하지 않는다.

### 3.3 기존 dmp와의 정합 (제안 — 문서 갱신은 사용자 승인 후)

- dmp §3 표에 **"판정 레벨(유저·net)" 행을 신설**하고, 기존 섹션 레벨 지표는 "진단" 역할로 명시 구분. 기존 표를 버리는 게 아니라 **층을 부여**하는 갱신이다.
- dmp의 미확정 2건은 진단 지표 정의로서 여전히 확정 필요: **신규 진입율 분자/분모**(유저-신규 vs 메타-신규 택1), **구매 상품 수 단위**(절대 vs 유저당 — 실험 맥락에선 유저당 권장). (CL-26 — 위임 금지 원칙대로 dmp에서 확정)
- 측정 성립 조건 = **CL-02 (블로커 확정)**: 노출(impression) 이벤트 + variant 차원 + (targetSection+tag) 단위 식별 전부 신규 필요, **소급 불가**, B 출시와 동시 배포. 추가 제안 — **측정의 이원화**: 1차 판정(유저 레벨)은 클라 이벤트 없이도 **서버 측 배정 로그(유저×셀) + 구매 데이터 조인**으로 성립 가능하게 설계하면, 앱 릴리즈 트레인(노출 이벤트 탑재 버전 보급)이 1차 판독을 게이트하지 않는다. 클라 노출 이벤트는 진단 층 활성화 조건. (스펙 상세 = event-spec(/dev-data)·분석 문서)

## 4. 의사결정 프레임 — ship / iterate / kill

### 4.1 판정 매트릭스 (3군 기준, 1차 지표 × 가드레일)

판정 시점: 사전 합의된 실험 기간 종료 시(기간·novelty 제외 규칙은 분석 문서 산정 + §5.1). **판정 지표·기간·아래 매트릭스는 실험 시작 전 고정(pre-registration)** — 시작 후 변경 금지.

| # | L vs C-A (목표 1·통합) | C-A vs C-M (목표 2·랭킹) | 매출 가드레일 | 판정 | 후속 |
|---|------|------|------|------|------|
| 1 | 승 | 승 또는 비열등 | 통과 | **Ship C-A 100%** | 칩+자동 랭킹 전면화 → 7섹션 확장(태깅 선행)·jp 적용 검토(CL-11 후속)·②키 정리 |
| 2 | 승 | **열등** | 통과 | **Ship C-M** (칩은 유지, 자동 랭킹 보류) | 랭킹 iterate — norm/가중치(CL-04)·태깅 품질(T2/T3)·콜드스타트(CL-08) 개선 후 ② 재실험 |
| 3 | 무유의 | 비열등 | 통과 | ⚠️ **사전 선언 필요** — "비열등이면 운영비 절감·최신성 확보 명분으로 C-A ship" vs "무유의면 보류" | 사용자 정책 결정(§6-5). ship 시 진단 지표로 개선 루프 |
| 4 | 승 (전환율) | — | **유저당 매출 열위** | **불합격 (확정 정책)** — ship 금지 | 저가 노출 비중 점검 → 점수의 매출 가중·base 임계 iterate 후 재실험 |
| 5 | 패 | 패 | — | **Kill — L 복귀** (2단 롤백: ②키 off → 키18/플래그 off) | 학습 정리: legacy recency의 강함 확인, 칩 UX 자체 재설계 검토 |
| 6 | 패 | 승 | — | 칩 컨테이너(UI)가 문제, 랭킹 알고리즘은 유효 | 컨테이너 UX iterate. **랭킹 자산은 보존** — legacy 단일 리스트에 점수 랭킹 적용 등 대안 지면 검토 (3군이라서만 얻는 학습) |

> 표기: "승/패/무유의"의 통계적 정의(유의수준·검정 방식·다중비교 보정 — 3군은 대비가 3개)는 분석 문서에서 확정. "비열등"의 마진도 분석 문서.

### 4.2 목표 2의 판정 기준 — 비열등 원칙 (제안)

목표 2를 "자동이 수동을 **이겨야** 한다"로 두면 자동화의 가치를 과소평가한다. PRD §2의 문제는 매출 부진이 아니라 **최신성 결여 + 운영 부담 + 적합성 미검증**이다. 자동화는 이기지 않아도 — 같은 성과를 사람 없이, 매일 갱신되는 상태로 내면 — 가치가 있다. 따라서:

> **C-A가 C-M에 유저당 매출 기준 비열등(non-inferior)이고 가드레일을 통과하면 ship 가능** — superiority는 충분조건이지 필요조건이 아니다. (⚠️ 잠정 — 사용자 승인 필요, §6-4)

### 4.3 성공/실패 시 경로 요약

- **성공(행 1) 경로**: C-A 100% → 잔여 7섹션(커플/솔로 등)은 대상 태깅 구축 후 확장(D-1·D-3 해소 선행) → jp 적용 검토(CL-11) → 키18·②키 정리, legacy 리스트 sunset 논의.
- **실패 시 학습 정리 방향**: 3군이므로 실패가 **어느 층(컨테이너 vs 랭킹)의 실패인지 분리 기록** 가능. 진단 층(칩별 CTR·목록 괴리도·잠식 추적)으로 "노출은 됐는데 안 샀는지 / 노출 자체가 안 됐는지 / 목록이 수동과 같았는지"를 사후 보고서에 남긴다 — 차기 랭킹 v2(가중치·개인화)·칩 UX 개선의 입력.

## 5. 제품 리스크 점검과 운영 정책

### 5.1 Novelty effect (신규성 효과)

칩 UI는 새 지면이라 초기 클릭이 일시적으로 부풀 수 있다. 대응: **판정은 후반 안정 구간 기준**(예: 1주차 제외 또는 주차별 효과 안정성 확인 — 구체 규칙은 분석 문서), CTR이 아닌 구매·매출 중심 판정(이미 §3 구조가 그러함). 참고: 키18이 이미 공개 분배 중이었다면 C-M/C-A 유저의 칩 UI novelty는 일부 소진된 상태고, C-A의 novelty는 "랭킹 변화"분만 남는다 — 키18 단계 확인(§2.3 전제 1)이 여기서도 입력이 된다.

### 5.2 대조군 운영 정책 — C-M 큐레이션 동결 금지

실험 기간 중 운영자가 C-M(수동 큐레이션) 갱신을 멈추면 C-M이 인위적으로 약해져 **C-A에 유리한 편향**이 생긴다(자동화 쪽이 "이기게 되어 있는" 실험은 의사결정 가치가 없다). 정책 제안:

- **평소 케이던스 그대로 큐레이션 지속** (동결 금지·실험 의식한 과잉 큐레이션도 금지 — "하던 대로").
- **칩 구성(어떤 칩·순서·라벨·showRanking)은 실험 기간 동결** — 칩 구성은 셀 간 공유 설정이라 변경 시 처치 정의 자체가 흔들린다. 부득이한 변경은 일자 기록 필수.
- 큐레이션 변경 이력(태그 junction priority 변경)을 주기 기록(주간 스냅샷 등)해 분석 시 맥락으로 제공.
- pinned 운영은 전 셀 공통 기능으로 평소대로 유지 (C-A도 pinned 상위 고정 보존 — FR-R10 확정과 정합).

### 5.3 구버전 앱 (컨테이너 미지원)

Android <2.43.0 / iOS <2.54.0은 게이팅상 키18 분배 전에 L 경로로 빠진다(코드 확인 — 감사 §1). 즉 **처치 수신이 불가능하므로 분석 모집단에서 제외**한다 (포함하면 효과 희석만 발생). 운영 시사점: 신버전 보급률이 낮으면 실험 모집단·검정력이 줄어든다 — 보급률 확인을 분석 문서 입력으로.

### 5.4 혼합 경험 — "C-A는 부분 자동화"다

1차 자동화는 5섹션뿐이다. C-A 유저가 보는 칩 중 자동 랭킹은 5개 칩에만 적용되고 나머지 칩은 C-M과 동일(수동)하다. 함의:

- C-A vs C-M 차이는 **칩 부분집합에서만 발생** → 효과 희석. 칩별 분해 진단(§3.2)이 필수.
- **활성(첫 번째) 칩이 자동화 대상 5섹션에 포함되는지**가 노출량을 좌우 — ⚠️ 현재 운영 칩 구성 실값은 미확인(운영 DB 덤프 잔여, 감사 §5). 실험 시작 전 확인하고, 필요하면 시작 *전에* 칩 구성을 확정(시작 후 변경 금지 — §5.2).
- 헤드라인 결과는 "5섹션 부분 자동화의 효과"로 해석·보고해야 하며, 7섹션 확장 시 효과가 달라질 수 있음을 전제.

### 5.5 KR 전용 · 시즌성

- **KR 전용**(CL-11 확정): 결과는 KR에만 일반화. jp는 현행 유지, 성공 시 후속 검토.
- 운세 서비스는 시기 수요 변동(신년·시험·연애 시즌 등)이 크다. 동시 무작위 대조라 **수준 변동 자체는 통제**되지만 — (i) 실험 기간은 7일 배수(요일 주기), (ii) 대형 시즌 이벤트·마케팅 푸시 주간과 시작·판독 시점이 겹치지 않게 조정, (iii) 효과×시즌 상호작용(예: 신년엔 사주 수요 폭증으로 랭킹 효과가 과대) 가능성은 보고서에 한계로 명시. 기간 산정은 분석 문서.

### 5.7 일정 의존 — 실험 시작 게이트

개발 마감 6/12은 **서버·데이터 측 B 구현**의 마감이고, 실험 시작은 별도 게이트를 가진다: (i) CL-02 측정 체계 — 서버 배정 로그는 B와 동시, 클라 노출 이벤트는 **앱 릴리즈 + 보급** 필요(§3.3 이원화로 1차 판독은 비차단 가능), (ii) 키18 단계 확인·실험 구조 확정(§6-1), (iii) T1 커버리지 사전 검증 + 섀도 모드(§5.6), (iv) ②키 신설·QA. **"6/12 배포 = 실험 시작"이 아님**을 일정 커뮤니케이션에 명시 권장.

### 5.8 동일 콘텐츠 이중 노출 — 칩 ↔ 하단 단독 섹션 (v1.2 정정 반영)

칩 컨테이너는 하단 단독 섹션들과 콘텐츠가 중복된다(§1.2). 함의 3가지:

1. **잠식 1순위 후보 = 하단 중복 섹션.** B군에서 칩 경유 소비 증가의 가장 유력한 출처는 "하단까지 스크롤해서 소비하던 트래픽"의 위치 이동이다. 이 내부 이동 자체는 실패가 아니라 접근성 개선의 정상 작동이며, 판정은 유저 레벨 net(§3.2)이 담당한다. 다만 진단 분해([분석 문서 §9](ab-test-analysis-design.md))에서 **"하단 동일 콘텐츠 섹션" 행을 별도 분리**해 "칩 +X / 하단 −Y / 기타 지면 −Z"를 나란히 봐야 한다. 하단 단독 섹션은 **L·C 양군에 공통 존재**하므로 군 간 직접 비교가 가능한 지면이다 — "칩이 하단 트래픽을 흡수했는가"를 정량화할 수 있다.
2. **UX 정합 관찰 항목.** C-A에서 동일 섹션이 위(랭킹순)·아래(수동순) 다른 목록으로 보이는 것은 1차 범위의 의도된 상태다(처치=칩 lazy 한정 — §1.2). 다만 "같은 '사주'인데 위아래 순서가 다르다"는 혼란 가능성은 **출시 전 섀도 모드 눈검수(§5.6) 관찰 항목**에 포함한다. 거슬리면 후속 단계(하단 섹션 랭킹 일원화 또는 하단 정리)로 검토 — 실험 기간 중 변경은 금지. ⚠️ 만약 하단 단독 섹션까지 랭킹 적용이 *의도*라면 이는 설계 변경(주입 범위 확대 — 감사 §3 전제 변경)이므로 사용자 확인 필요.
3. **운영 정책 단순화 (2군 확정 반영).** 하단 단독 섹션 큐레이션은 **양군 공통 지면**이므로 평소 케이던스로 운영하면 양군에 동일하게 작용해 편향이 없다 — §5.2의 'C-M 동결 금지' 우려는 2군+공통 지면 구조에서 자연 해소된다. 칩 구성 동결(§5.2)은 유지.
