## 0. 요약 (TL;DR)

- **식별 문제**: 사용자 제안 2군(무칩 vs 칩+자동랭킹)은 **UI 추가 효과와 랭킹 자동화 효과가 혼재** — 목표 1(패키지 총효과)만 답하고 목표 2(자동 vs 수동)는 식별 불가. §1에서 3개 구조 옵션의 통계적 함의를 비교. 구조 선택은 ⚠️ 사용자 결정(PO 문서 연계).
- **1차 지표(권장)**: 유저 레벨 net 2종 — ① **실험 기간 구매자 전환율**(우위 검정) ② **유저당 매출**(비열위 가드레일 겸용). 두 조건 동시 충족 시 ship(접합 규칙 — α 보정 불요).
- **베이스라인 실측 ✅**: 최근 4주 KR 홈탭 방문 90,066명 / 구매자 30,227명(방문 대비 ≈30% 수준) / 구매자당 매출 ₩19,063(CV 1.08 — 헤비테일 약함, 전체 유저 CV ≈2.5).
- **검정력**: 2군 4주면 전환율 상대 MDE ≈3.2%, ARPU ≈5.2%(CUPED 시 ≈4.5%), ARPU 비열위 마진 5% 검정 가능. 3군이면 동일 기간 MDE 약 1.35배 악화.
- **권장 운영**: 고정 기간 **4주**(요일 주기 ×4 + novelty 분해 여유), peeking 금지, 중간 모니터링은 SRM·가드레일·버그 한정.
- **최대 블로커**: CL-02 — 노출 이벤트·variant 차원 **신규 필수, 소급 불가** → 실험 시작 전 배포+검증 ≥1주(§5·§13).

## 1. 실험 구조 — 식별 문제와 옵션

### 1.1 군(arm) 명명 — 혼동 방지

"A/B"가 세 겹으로 과적("①축 핵클 variation A/B" / "②축 AsIs/인기점수" / "사용자 제안 A/B")이므로, 본 문서는 **상태 기호**로 통일한다.

| 기호 | 상태 | 기존 문서 대응 |
|------|------|----------------|
| **L** (Legacy) | 칩 컨테이너 없음 — 기존 단일 '지금 인기'(`recentPurchasedSkills`, recency 순) | 사용자 제안의 "A". ①축(Hackle 키18)의 variation A와 동일 상태 |
| **M** (Manual) | 칩 컨테이너(`featuredSkillsTabs`, 5칩) + **현행 수동 큐레이션 순서**(태그=어드민 priority / recent=recency) | 프로젝트 기존 문서의 "A(AsIsRanker)" — [감사 §2](s5-asis-implementation-audit.md)의 엄밀 정의 |
| **R** (Ranked) | 칩 컨테이너 + **자동 랭킹**(PopularityScore, base 풀 ∧ 섹션 필터, 점수순) | 사용자 제안·기존 문서의 "B" |

주의: 사용자 제안의 A("칩 섹션이 없는 기존 상태")는 **빈 화면이 아니라 L** — 상호배타 1-슬롯 스왑으로 legacy '지금 인기' 단일 섹션이 그 자리에 노출된다([서빙 분석 §1-②](s5-asis-serving-analysis.md)).

### 1.2 식별 문제

사용자 제안 = **L vs R 2군**. 효과 분해:

```
Δ(L→R) = Δ(L→M)[UI·칩 컨테이너 효과] + Δ(M→R)[랭킹 자동화 효과] + 상호작용
```

| 사용자 목표 | 필요한 대비 | L vs R로 답 가능? |
|------|------|------|
| 목표 1 — 섹션을 통한 진입·구매자 증가 (패키지 총효과) | L vs R | **O** |
| 목표 2 — 자동 랭킹이 수동 선정 대비 구매·매출 우위 | **M vs R** (UI 고정) | **X — 식별 불가** |

기존 결정과의 관계: 프로젝트의 기존 ②랭킹 A/B 정의는 **M vs R**(featuredSkillsTabs 안에서 AsIs ↔ 인기점수, [status.md 확정](../status.md))이고, L vs M은 ①컨테이너 A/B(Hackle 키18)로 **이미 LIVE**. 사용자 제안(L vs R)은 이 둘을 합친 총효과 실험이다 — readme §7의 "①종료 후 ② 순차" 결정과 긴장 관계(제품 판단 → PO 문서).

### 1.3 구조 옵션별 통계적 함의

| | **옵션 1 — 순차 2단계** | **옵션 2 — 3군 병렬** | **옵션 3 — 2군 통합 (사용자 원안)** |
|---|---|---|---|
| 구성 | ①축(L vs M, 키18 LIVE) 판독·종료 → ②축(M vs R) 2군 | L / M / R 동시 (키18 위에 ②키 중첩 또는 신규 3-arm) | L vs R 2군 |
| 목표 1 식별 | **△ 합성** — Δ(L→M)+Δ(M→R) 근사. 두 실험이 다른 기간 → 기간 교란·비가산성, 단일 무작위 비교 아님 | **O 직접** (L vs R 대비) | **O 직접 · 풀 검정력** |
| 목표 2 식별 | **O 직접 · 풀 검정력** (단계당 50:50) | **O 직접** (M vs R 대비) | **X** — B군 내 관찰 분해만(§1.4, 인과 아님) |
| 군당 표본 (동일 트래픽) | 단계당 n/2 | **n/3 — 50:50 대비 군당 2/3** | n/2 |
| 동일 MDE 달성 기간 | 단계당 기준 1× (총 캘린더 = 2단계 합) | 분할 ×1.5 + 1차 대비 2개 α보정(Holm, ×1.21) ⇒ **≈1.8×** (MDE 고정 시), MDE 환산 ≈×1.35 악화 | 1× |
| 다중비교 | 단계당 1차 대비 1개 — 보정 불요 | 1차 대비 2개(목표1·목표2) — α=0.025/개 또는 Holm | 1차 대비 1개 — 보정 불요 |
| 배정·SRM 복잡도 | 낮음 (단계당 2군) | 높음 — 키18 분배율(⚠️ 운영 미확인) 위 중첩 시 군 비율 통제·2단 SRM 필요 | 낮음 — 단 키18을 ②신호와 통합 정리 필요 |
| 측정 리스크 | ⚠️ ①축 판독 가능성 자체가 불확실 — 키18 분배가 이벤트에 안 실림(CL-02-③) → ①효과는 핵클 콘솔 지표 의존(연동 범위 미확인). ①판독 불가 시 목표1 합성도 약화 | CL-02 해소 시 세 군 모두 BQ로 동일 판독 | 동일 |
| 기존 결정 정합 | **최고** — S2 "①종료 후 ②순차"·감사 §4-3-5와 일치 | ①을 사실상 연장 — S2 결정 수정 필요 | ②축 정의(M vs R) 폐기, M 운영 중단 — 기존 문서 다수 수정 |

**분석가 권고(통계 관점만)**:
- 목표 2(랭킹 파이프라인 투자 지속 판단)가 핵심 의사결정이면 **M vs R 대비가 풀 검정력으로 확보되는 옵션 1 또는 2**가 필요하다. 옵션 3은 목표 2를 영구히 답하지 못한다(§1.4).
- 캘린더 총기간을 짧게 묶고 두 질문을 같은 기간·같은 모집단에서 답해야 하면 옵션 2(비용 = MDE ×1.35 또는 기간 ×1.8). 단계별 풀 검정력·낮은 운영 복잡도·기존 결정 정합이 우선이면 옵션 1.
- 최종 선택 = ⚠️ **사용자 결정** ([PO 문서](ab-test-po-review.md)의 제품 트레이드오프와 함께).

### 1.4 옵션 3 선택 시 — R군 내 관찰 분해의 한계 (사전 고지)

옵션 3에서 목표 2를 "관찰 분석으로 보완"하는 안은 다음 한계를 갖는다. **어느 것도 인과 추정이 아니다.**

1. **동기간 반사실 부재** — 같은 기간의 "칩+수동(M)" 성과가 없으므로, 역사적 M 성과(①실험 기간)와 비교하면 시즌·프로모션·유저 구성 변화가 전부 교란.
2. **R군 내 칩·랭크별 CTR/구매**는 자동 랭킹의 *운영 품질* 진단이지 수동 대비 *우월성* 증거가 아님.
3. **목록 차집합 분석**(자동 목록에만 있는 스킬 vs 수동 목록에만 있는 스킬의 성과)은 position bias·selection bias(애초 점수가 높아 선정된 스킬)로 오염 — 탐색적 라벨로만 보고.

### 2.2 분석 모집단 등록과 dilution

배정(전 유저 일괄)과 분석 모집단은 분리한다. 효과는 섹션을 *볼 수 있는* 유저에게만 발생하므로, 미방문 유저를 분모에 넣으면 효과가 희석(dilution)되어 동일 MDE에 더 큰 표본이 필요하다.

| 모집단 후보 | 정의 | 판정 |
|------|------|------|
| (i) 배정 전체 | KR 적격 유저 전부 | 순수 ITT — dilution 최대(홈 미방문 유저 다수 포함), 비권장 |
| **(ii) 홈탭 진입자 (권장)** | 실험 기간 중 홈탭 1회 이상 진입(`view_tab_at_home` ✅ 실재 확인 — 또는 서버 홈 API 호출) | **양군 대칭 트리거** — 홈 진입 시점에는 아직 variant 콘텐츠가 행동에 영향을 못 미침(최초 진입 기준). 섹션이 홈탭 상에 있으므로 진입=노출 기회 |
| (iii) 섹션 영역 도달자 | 섹션 뷰포트 노출(신규 이벤트) 발생 유저 | L군(옵션 3)엔 동일 개념 부재 → **비대칭, 사용 금지**. M vs R(옵션 1·2)에선 정의 가능하나 노출 성향이 처치의 영향을 받을 수 있어(재방문 증가 등) collider 위험 — 진단용으로만 |

- **등록 규칙(권장)**: 실험 기간 중 **첫 홈탭 진입 시점**에 등록(enrolled_at) — 이후 모든 지표 윈도우는 등록 시점부터 실험 종료까지(또는 고정 k일 — §14). 미등록 배정 유저는 분석 제외.
- **2차 dilution**: 홈탭 진입자 중 섹션 영역 미도달(스크롤 위치)·칩 데이터 lazy 미호출 유저 비율 r = ⚠️ 실측 필요(노출 이벤트 배포 후 산출 가능). r은 메커니즘 해석(노출 기회 대비 효과)에 사용하고 모집단 정의에는 쓰지 않는다.
- **신규 가입자**: rolling 등록 허용 권장(가입 즉시 해시 배정 → 첫 홈 진입 시 등록). '신규 인기' 섹션 가설에 신규 유저가 핵심이므로 제외하면 일반화 손상. ⚠️ 포함 여부 사용자 확인(§14).

### 2.3 SRM (Sample Ratio Mismatch) 체크 설계

| 항목 | 설계 |
|------|------|
| 검정 | 등록 유저 수의 군별 비율 vs 기대 비율 — χ² 적합도 검정 |
| 주기 | 일 1회 자동(일별 신규 등록 + 누적), 대시보드화 |
| 임계 | p < 0.001 → 알람·효과 판독 동결(원인 규명 전까지) |
| 절단 | 전체 + 세그먼트별(플랫폼, 앱 버전, 신규/기존, 일자) — 부분 SRM 탐지 |
| 주요 원인 후보 | Hackle SDK 실패 폴백 쏠림(§2.1 — `sdk_fail_flag` 분리로 예방), 버전 게이트 비대칭 배포, 클라 로깅 유실 플랫폼 편차, 봇/테스터 유입 |
| 사전 점검 | **AA 테스트(권장 1주) 또는 최소 1일 드라이런** — 동일 처치 2군으로 배정·로깅·SRM 파이프라인 검증(§13) |

|---|---|---|
| 정의 | 분자 = 기간 중 "그 유저가 과거 90일간 진입(`open_skill_description`) 이력 없는 스킬"에 진입한 유저 수(보조: 유저당 신규-스킬 진입 건수) · 분모 = 분석 모집단 | 분자 = 출시(`open_date`) 30일 내 스킬 진입 유저 수 · 분모 동일 |

**(2) 구매 상품 수 — 제안: 유저당 distinct 스킬 수**

- 후보: 절대 카운트 / 노출당 / **유저당(제안)**.
- 이유: A/B는 유저 무작위 — 유저 정규화 지표만 군 크기와 무관하게 검정 정합. 절대 카운트는 군 크기 종속, "노출당"은 L군(옵션 3)에 노출 개념이 없어 비대칭.
- 운영 보고용으로 섹션 레벨 절대 카운트 병기 가능(판정 지표 아님).
- 베이스라인 ✅: 구매자당 distinct 1.22개/4주(구매 1.89건 대비 — 반복 구매 비중 큼 → 다양성 개선 여지 큼).

### 4.4 1차 지표 판정 규칙 (다중비교 처리 포함)

```
Ship 조건 = [전환율: R(또는 옵션3의 R) > 대조군, 양측 α=0.05 유의]
          AND [ARPU: 비열위 — 단측 95% CI 하한 > −δ_NI (δ_NI 제안 5%, §6.3)]
```
- **접합(intersection-union) 규칙** — 두 조건 모두 통과해야 ship이므로 각각 α=0.05로 검정해도 전체 1종 오류 ≤0.05. **α 분할 불요.**
- 2차·진단·세그먼트 지표는 전부 **탐색적** — 판정에 불사용, 보고 시 BH-FDR 10% 라벨.
- 옵션 2(3군)만 예외: 1차 대비가 2개(목표1 L vs R, 목표2 M vs R) → **Holm 보정**(α=0.025 시작).

### 6.2 공식

```
비율(전환율):   n/군 = (z₁₋α/₂ + z₁₋β)² · 2·p̄(1−p̄) / (m·p)²     (m = 상대 MDE)
연속(ARPU):     n/군 = (z₁₋α/₂ + z₁₋β)² · 2·CV² / m²
비열위(ARPU):   n/군 = (z₁₋α + z₁₋β)² · 2·CV² / m_NI²            (단측, 진차 0 가정)
α=0.05, power=0.8 → (1.96+0.8416)² = 7.85 (비열위 단측: (1.645+0.8416)² = 6.18)
```

### 6.3 MDE × 기간 매트릭스 (✅ 실측 기반)

전제: KR 홈탭 진입 누적 유저 — 1주 32k(실측) · 2주 ≈54k(보간) · 4주 90k(실측) · 6주 ≈122k · 8주 ≈153k(외삽, 주당 +16k) → **버전 게이트 80% 가정**(⚠️) 적용 후 군별 분할. p=0.30, CV_all=2.49 기준. 양측 α=0.05, power 0.8.

| 기간 | 적격 누적(80%) | **2군** 군당 n | 전환율 MDE | ARPU MDE | ARPU MDE+CUPED† | **3군** 군당 n | 전환율 MDE‡ | ARPU MDE‡ |
|------|------|------|------|------|------|------|------|------|
| 2주 | ~44k | 22k | 4.1% | 6.7% | 5.8% | 14.7k | 5.5% | 9.0% |
| **4주 (권장)** | **~72k** | **36k** | **3.2%** | **5.2%** | **4.5%** | 24k | 4.3% | 7.0% |
| 6주 | ~98k | 49k | 2.7% | 4.5% | 3.9% | 32.5k | 3.7% | 6.0% |
| 8주 | ~122k | 61k | 2.4% | 4.0% | 3.5% | 40.7k | 3.3% | 5.4% |

† CUPED ρ=0.5 가정(분산 25%↓ → MDE ×0.866). ρ ⚠️ 사전 실측으로 확정.
‡ 3군 = 군당 표본 1/3 + 1차 대비 2개 Holm 보정(MDE ×1.10) 포함.
- p 민감도(4주 2군): p=0.25 → 전환율 MDE 3.6% / p=0.20 → 4.2%.
- 전환율 p는 윈도우가 길수록 커져(누적 전환) 장기간일수록 위 표보다 유리 — 2주 행은 다소 낙관(2주 윈도우 p < 0.30) ⚠️.

**ARPU 비열위 검정력** (단측 α=0.05, power 0.8, CV 2.49):

| 비열위 마진 δ_NI | 필요 군당 n | 도달 시점 (2군) |
|------|------|------|
| 10% | 7.7k | <2주 |
| **5% (제안)** | **30.8k** | **4주 (36k ✓)** |
| 3% | 85.5k | ~10주 — 비현실 |

→ **δ_NI = 상대 5% 제안** (⚠️ 사용자 결정). 의미: "ARPU가 5% 넘게 나빠지지 않았음"까지만 보증 — 그보다 작은 매출 하락은 통계적으로 배제 불가함을 판정 시 인지(§12).

### 6.4 매출 지표의 구조적 저검정력 (명시)

- 동일 표본에서 **ARPU MDE(5.2%)는 전환율 MDE(3.2%)의 약 1.6배** — 영과잉+연속 분산(CV 2.49 vs 비율 분산) 때문. 본 서비스는 가격 상한 구조로 구매자 CV가 1.08로 낮아 일반 freemium(CV 4~8)보다는 양호하나, 구조는 동일.
- 유저당 **구매 횟수**는 더 나쁨 — 구매자 횟수 CV 2.73(SD 5.16/평균 1.89), 전체 유저 CV ≈5.2 → 4주 2군 MDE ≈11%. winsorize 필수(§7), 2차 지표로만.

## 7. 검정 방법론

| 지표 유형 | 1차 검정 | 보조·강건성 | 비고 |
|------|------|------|------|
| 비율(전환율 등) | 두 비율 z-검정(≡χ²) | Wilson/Newcombe CI | |
| 연속(ARPU·구매횟수) | **Welch t** — 상위 **0.5% winsorize**(0.1~1% 민감도 병행) | **Mann-Whitney 병기**(분포 이동 — 평균 검정 아님을 해석에 명시) + **bootstrap percentile CI**(평균차, ≥10k 재표집) | log(1+x)는 영과잉 해석 왜곡 — winsorize 우선, log는 보조만 |
| 비열위 | 단측 95% CI 하한 vs −δ_NI | bootstrap CI 교차 확인 | §6.3 |

**CUPED (적용 권고)**
- 공변량 X = **실험 직전 28일 동일 지표**(사전 구매 여부 / 사전 매출). `Y_adj = Y − θ(X − X̄)`, θ = cov(X,Y)/var(X). 1차 지표 2종 모두 적용.
- 기대 효과: 분산 감소 = ρ². 구매·매출의 4주↔4주 자기상관 ρ는 통상 0.4~0.6 → **분산 16~36%↓ (기간 환산 동일 비율 단축)**. ⚠️ ρ는 사전 기간 두 윈도우로 실측 후 확정(§13) — 실측 전 본 문서 수치는 ρ=0.5 가정.
- 신규 가입자(사전 기간 부재): X=0 + 신규 더미 추가(또는 층화 CUPED).
- 전환율(이진)에도 선형 CUPED 적용 가능(사전 구매 더미) — 보수적으로 미조정 결과 병기.

**다중비교 방침** — §4.4와 동일: 1차 = 접합 규칙(보정 불요, 옵션 2만 Holm) / 2차·진단·세그먼트 = 탐색적(BH-FDR 10% 라벨, 판정 불사용).

## 9. 잠식(cannibalization) 분석 설계

**목적**: R군의 신규 섹션 기여가 타 지면에서 옮겨온 것인지(zero-sum) 순증인지 분해. 1차 지표가 net이므로 판정은 1차가 담당 — 본 분석은 **해석용**.

**net 분해 표 (설계)** — 군별 유저당 귀속 구매·매출을 지면별로:

| 지면 | 식별 이벤트 | 가용성 |
|------|------|------|
| 인기 섹션 (L: 지금인기 / M·R: 칩 컨테이너) | `touch_home_section_item`(`section_name` + E2 확장 tag) → §5 귀속 | 기존 이벤트로 홈 내 클릭 식별 ✅ (주간 25k 클릭 실측) — 칩·variant 차원은 E2 배포 후 |
| 홈 내 타 섹션 (배너·카테고리·무료 등) | `touch_home_section_item`(타 `section_name`), `touch_featured_banner`, `touch_home_section_more_item` ✅ 실재 확인 | **홈 내 분해 가능** — 단 §5 동일 귀속 규칙 적용 필요 |
| 홈 외 지면 (검색·채팅탭·푸시 딥링크 등) | ⚠️ 식별 이벤트 미확인 — E5(`open_skill_description` source) 구현 시 직접 식별 | E5 미구현 시 **잔차로 처리**: 홈외 귀속 = 전체 구매 − 홈 내 귀속 합 |
| 합계 | = 1차 지표(전체 구매·매출) | 분해 합 ≡ net 정합 체크 |

**산출물**: `군 × 지면` 매트릭스 — R군 신규 섹션 기여 +X vs 타 지면 합 −Y를 나란히, net(X−Y)이 1차 Δ와 일치하는지 교차 검증. 지면별 차이의 유의성은 탐색적 라벨.

**Q4. 4주 유저 레벨 매출 분포** (동기간, KR 구매자)

```sql
WITH u AS (
  SELECT user_id, SUM(revenue_krw) AS rev, COUNT(*) AS purchase_cnt, COUNT(DISTINCT menu_seq) AS distinct_skills
  FROM `hellobot-f445c.hlb_mart.mart_use_skill_se`
  WHERE event_date BETWEEN "2026-05-11" AND "2026-06-07" AND user_country = "South Korea"
    AND event_name IN ("pay_for_contents","pay_under_750")
  GROUP BY 1
)
SELECT COUNT(*) AS buyers_4w, ROUND(AVG(rev)) AS mean_rev, ROUND(STDDEV(rev)) AS sd_rev, ROUND(STDDEV(rev)/AVG(rev),2) AS cv_buyers,
  ROUND(APPROX_QUANTILES(rev,100)[OFFSET(50)]) AS p50, ROUND(APPROX_QUANTILES(rev,100)[OFFSET(90)]) AS p90,
  ROUND(APPROX_QUANTILES(rev,100)[OFFSET(99)]) AS p99, ROUND(MAX(rev)) AS max_rev,
  ROUND(AVG(purchase_cnt),2) AS avg_purch, ROUND(STDDEV(purchase_cnt),2) AS sd_purch, ROUND(AVG(distinct_skills),2) AS avg_distinct
FROM u
```
