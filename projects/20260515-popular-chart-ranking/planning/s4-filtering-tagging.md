## 한 줄

섹션을 채우는 **필터 = (시간성 · 주제 · 의도+대상) 메타축의 조건식**으로 정의한다. 축의 원천은 둘 — **임시 태그(`taenyon_temp_skill_tag_info_v2`: topic·intents·temporal)** 와 **공식 카테고리(`mart_fixed_menu_server.chatbot_content_type`)**. 섹션→필터 규칙은 **현재 미존재 → 신규 설계 대상**이며, "현 태깅이 적절한가"는 **커버리지·정합·행동데이터·태그품질 4종 테스트**로 검증한다.

## 1. 태그 메타 인벤토리 (가용 축)

| 축 | 원천 | 입도 | 성격 | 비고 |
|----|------|------|------|------|
| **topic** (주제) | `google_sheet_sync.taenyon_temp_skill_tag_info_v2` | 스킬×1 | 점술 주제(사주/타로/…) | **임시 태깅** — 프로덕션 메타 아님 (DF-S4-1) |
| **intents** (의도) | 〃 | 스킬×N(다중 추정) | 관계/목적(재회/궁합/짝사랑/…) | 〃 |
| **temporal** (시간성) | 〃 | 스킬×1 | 실시간/신규/상시 성격 추정 | 〃 · 값 정의 확인 필요 |
| **chatbot_content_type** (카테고리) | `hlb_mart.mart_fixed_menu_server` ← `chatbot`+`chatbot_category`+`chatbot_category_relation` join | 스킬×1 | **공식** 챗봇 카테고리 | 서버 DB 정규 데이터 |
| **targets** (대상) | 서버 `FixedMenu.targets` (S1 확인) | 스킬×N | 솔로/커플 등 | 섹션 4종이 이 축 사용 |

> **핵심 긴장(=DF-S4-1)**: 섹션 세분화에 필요한 풍부한 축(topic/intents)은 **임시 태그(구글시트 동기화)** 에 있고, 정규/안정 축(chatbot_content_type)은 **세분도가 부족**할 가능성. → 임시 태그를 **프로덕션 승격**할지, **공식 카테고리로 흡수/매핑**할지가 S5 구조를 가르는 1순위 결정.

## 2. 섹션 분류 — 3개 구동 그룹

12개 섹션은 "무엇이 후보 풀을 거르는가"에 따라 3그룹으로 갈린다:

| 그룹 | 구동 축 | 섹션 |
|------|---------|------|
| **G1 시간성** | temporal / 등록일 | 실시간 인기 · 신규 인기 |
| **G2 주제** | topic / content_type | 사주 · 타로 · 1:1 상담 |
| **G3 의도(+대상)** | intents (+ targets) | 솔로 애정운 · 짝사랑/썸 · 커플 궁합 · 재물운 · 커플 결혼 궁합 · 재회 · 솔로 결혼운 |

→ G3가 7/12로 **이번 필터 설계의 본체**. 그리고 G3 중 4개(솔로 애정운·커플 궁합·커플 결혼 궁합·솔로 결혼운)가 **의도 ∩ 대상** 복합 조건.

## 3. 섹션 → 필터식 (1차 가설 — 값 바인딩은 §6 데이터 확인 후 확정)

> 표기: `axis = v` 단일, `axis ∋ {…}` 다중 포함. 값 문자열(예: '애정운'/'연애운')은 **가설** — 실제 태그 값으로 치환 필요.

| 섹션 | 그룹 | 필터식 (가설) | 비고/모호 |
|------|------|--------------|-----------|
| 실시간 인기 | G1 | 전체 노출가능 풀 (필터 ≈ 없음) · ranker=짧은 윈도우 인기 | "실시간"은 **랭커 윈도우** 문제일 수 있음 → S3/S5 |
| 신규 인기 | G1 | `등록일(menu_create_at_date) ≤ N일` ∩ 인기 | S3 신규부스트와 동일 원천 — N 정합 필요 |
| 사주 | G2 | `topic = 사주` | content_type=사주와 택1/교차 (DF-S4-1) |
| 타로 | G2 | `topic = 타로` | 〃 |
| 1:1 상담 | G2 | `content_type = 상담(1:1)` | **주제가 아니라 콘텐츠 형식** — content_type 축이 더 적합 |
| 솔로 애정운 | G3 | `intents ∋ 애정운` ∩ `targets ∋ 솔로` | 대상 축 출처 = DF-S4-2 |
| 짝사랑/썸 | G3 | `intents ∋ {짝사랑, 썸}` | |
| 커플 궁합 | G3 | `intents ∋ 궁합` ∩ `targets ∋ 커플` | 대상 축 = DF-S4-2 |
| 재물운 | G3 | `intents ∋ 재물` (또는 `topic = 재물`) | **의도 vs 주제 모호** — 연애계열과 축이 다를 수 있음 |
| 커플 결혼 궁합 | G3 | `intents ∋ 결혼궁합` ∩ `targets ∋ 커플` | '궁합'과 '결혼궁합' 분리 여부 확인 |
| 재회 | G3 | `intents ∋ 재회` | |
| 솔로 결혼운 | G3 | `intents ∋ 결혼운` ∩ `targets ∋ 솔로` | '결혼운'과 '결혼궁합' 분리 여부 확인 |

> 일반화: **`section_filter(skill) = Π match(axis_i, values_i)`** (축 조건의 AND, 한 축 내부는 OR). 이 "섹션=필터 술어 집합"을 데이터화하는 스키마는 S5.

## 4. 적절성 테스트 설계 (사용자 요청 ② — "어떻게 테스트할지 방향")

"현재 태깅한 필터링이 적절한가"를 **배포 전 사전검증 → 배포 후 행동검증** 순으로 4종:

| # | 테스트 | 질문 | 방법 / 지표 | 시점 |
|---|--------|------|------------|------|
| **T1 커버리지** | 필터가 충분한 풀을 만드나 | 섹션별 후보 스킬 수, **미태깅률**(topic/intents/temporal 결측 %), 빈/과소 섹션 탐지 | 사전 (BQ 집계) |
| **T2 정합(오분류)** | 엉뚱한 섹션에 들어가거나 빠지나 | **골든셋**(사람 라벨 스킬×섹션) 대비 precision/recall, 섹션 간 중복도 | 사전 (샘플 라벨링) |
| **T3 태그 품질** | 임시 태그 자체가 믿을 만한가 | temp `topic` × 공식 `chatbot_content_type` **교차표 충돌율**, 동일 챗봇군 태그 일관성 | 사전 (BQ 교차) |
| **T4 행동 정합** ★ | "유저가 원하는 걸 거른다"가 사실인가 | ① 섹션 노출→그 섹션 콘텐츠 **구매 전환율** ② 과거 같은 topic/intent 구매자가 그 섹션에서 **CTR↑** 인가(태그-선호 상관) ③ **A/B**: 자동필터(B) vs 수동큐레이션(A) 섹션 CTR·구매 비교 | ①②=사전(과거로그) / ③=사후(S2 A/B) |

- **사전 즉시 가능(T1·T2·T3)** → `/dev-data` 과업. 합격선(예: 섹션당 후보 ≥ 슬롯×3, 미태깅률 ≤ X%, 교차 충돌율 ≤ Y%)은 **DF-S4-4**.
- **T4 ③** 가 최종 판정 — 단 S2에서 A/B는 알고리즘 적용 이후 단계로 분리했으므로, T4①②(과거 로그 상관)로 사전 신뢰도를 먼저 확보.

## 6. /dev-data 위임 조사 항목 (값·커버리지 실측 — 인증 복구 후)

1. `taenyon_temp_skill_tag_info_v2` 스키마 + **distinct topic / intents / temporal 값 목록** + 스킬 커버리지(태깅된 스킬 수 / 전체).
2. `mart_fixed_menu_server.chatbot_content_type` **distinct 값** + 빈도.
3. **교차표**: temp topic × chatbot_content_type (T3 충돌율).
4. 각 §3 섹션 가설 필터식의 **후보 스킬 수**(T1) + 섹션 간 중복(T2 예비).
5. `FixedMenu.targets` 값 분포(솔로/커플 등) — DF-S4-2.

> 이 5개가 §3 값 바인딩과 §4 T1~T3을 **수치로 확정**시킴. 설계(본 문서)는 이미 진행 가능, 실측은 병렬.

## 7. S5(아키텍처)로 넘기는 것

- **통합 필터 정의 레이어** 스키마 (S1: 현재 부재) — "섹션 = 필터 술어 집합 + 랭커 바인딩(S2)"을 어떻게 저장/평가할지.
- 태그 소스 확정(DF-S4-1) 후의 **메타 적재 파이프라인**(임시 시트 → 프로덕션 or 카테고리 매핑).
- 필터 평가 위치(서버 쿼리 vs 사전계산 풀) + 랭킹(S3)·variant 주입점(S2)과의 결합.

## 8. S4 결론

필터는 **3축 그룹(시간성/주제/의도+대상)** 으로 모델링되고, 섹션 7/12가 의도(+대상) 복합 조건으로 G3가 본체다. 섹션→필터 규칙은 **신규 설계 대상**이며 §3에 1차 가설을 제시했다(값 바인딩은 /dev-data 실측 후 확정). 적절성은 **사전 3종(커버리지·정합·태그품질) + 사후 행동/A·B**로 검증한다. **DF-S4-1 = 하이브리드 확정(2026-06-05)**: 1차는 임시 태그 그대로, 정식 승격/카테고리 정합은 실측 후 후속.

> **S5 함의(하이브리드)**: 필터 메타(임시 태그)와 랭킹 스코어(S3)가 **둘 다 BQ**에 있으므로, 1차 자연 설계 = **일배치가 BQ에서 섹션별 `필터→랭킹` 적용 후 `섹션→[정렬된 menu_seq]` 결과 목록을 서비스 DB에 적재**(서비스 DB에 태그 메타를 옮길 필요 없음 = 빠른 1차). 승격 seam = 후속에 태그 메타를 정식/서버로 이동해 서버측 필터 평가로 전환. A 랭커(AsIs)=현행 서버 순서, B만 배치 산출. (상세 S5)
