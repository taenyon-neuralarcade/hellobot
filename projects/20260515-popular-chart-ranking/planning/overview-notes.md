# 인기스킬 섹션 노출 자동화 — 개괄 정리 (사용자 원본 노트)

> 출처: 사용자 작성 `temp.md` (UTF-8 재작성·이관, 2026-06-04). 원본은 인코딩 깨짐(latin1)으로 삭제.
> 성격: 요구사항·아이디어 소스. 가공된 실행 스텝은 [execution-steps.md](execution-steps.md) 참조.

## 목표

- 현재 인기 스킬 섹션에 스킬들이 **운영자가 수작업으로 세팅한 스킬 목록**이 모든 사용자에게 **고정적으로 노출**되고 있음.
- 콘텐츠 노출은 운영자가 업데이트하기 전까지 동일한 콘텐츠만 노출되고 있음.
- **매일 스킬의 노출 랭킹을 계산하고, 해당 랭킹을 기반으로 노출되는 콘텐츠를 결정하여 노출을 자동화**하는 것이 목표.

## 아이디어

- **노출 자동화 = 필터링 + 랭킹**
- 현재 스킬 목록은 여러 종류가 있을 수 있음:
  - 실시간 인기 / 신규 인기 / 사주 / 타로 / 1:1 상담 / 솔로 애정운 / 짝사랑·썸 / 커플 궁합 / 재물운 / 커플 결혼 궁합 / 재회 / 솔로 결혼운
- 현재 방향성은 **랭킹은 1종**으로 하려고 함.
- 결국 **필터링으로 각 스킬 목록을 결정**할 것.
- 필터링은 콘텐츠의 메타 정보로 될 것:
  - 카테고리 — 사주/타로
  - 관심사 — topic
  - 의도 — intents

### 데이터 분석 필요 항목

- 이전 분석 결과로 보면, 필터링은 사용자가 원하는 콘텐츠를 쉽게 찾을 수 있게 걸러서 보여주는 것:
  - 사주 사용자는 사주 콘텐츠에 관심
  - 타로 사용자는 타로 콘텐츠에 관심
  - 관심사별: 연애 → 연애 / 결혼 → 결혼·연애 / 재물 → 재물 / 가족 → 가족
- **현재 태깅한 필터링이 적절한가?** (검증 대상)

## 요구사항 — 이걸 하려면

- **랭킹 알고리즘은 일단 1개** > 스킬 목록을 도출하면 그 스킬들의 순서를 랭킹순으로 정렬되어야 한다.
  - 이거 맞나? 여러 개 되어야 하는 거 아냐?
  - 이번 스펙은 아니지만, 알고리즘도 여러 개를 동시에 유지하며 사용할 수 있게 확장될 수 있다.
  - 랭킹 알고리즘을 어떻게 구현할 것인가? (아키텍처 관점)
- **콘텐츠들을 필터링하기 위한 카테고리·태깅** > 현재 설정된 섹션들에 해당하는 스킬을 필터링할 수 있어야 한다.
  - 현재 저 필터링을 할 수 있도록 메타 정보들이 존재하는가?
  - 스킬 태깅 정보는 어떻게 관리할 것인가? (아키텍처 관점)
- **리스트별로 필터링하는 조건이 다름** — 이걸 어떻게 관리해야 하나? > 섹션들의 필터링 조건은 우선 현재 정의된 것까지만 범위로 한다.
  - 아마도 API별로 필터링 알고리즘이 다를 것 같은데, 현재는 API별로 하드코딩되어 있을 것 같음.
