## 한 줄

**노출 섹션 = 필터(후보 풀 결정) + 랭커(정렬 전략).** 랭커는 교체 가능한 전략으로 두고, A/B는 "어떤 랭커를 적용할지"의 선택(A=현재 그대로, B=자동 스코어)으로 모델링한다. 랭커를 인터페이스로 분리해 장기적으로 N종 알고리즘 공존을 비파괴로 허용한다.

## 사용자 확정 요구사항 (2026-06-04)

1. **랭킹 1종만 이번 적용** — "인기차트 선정 로직" 문서의 스코어링.
2. **A/B 진행** — A = 현재 상태 그대로 노출 / B = 자동 랭킹 적용. **API 레벨에서 2종 목록을 경우에 따라 다르게 서빙**.
3. **장기 다중 랭커 공존** — 목적·필터링된 콘텐츠 특성에 따라 알고리즘이 달라질 가능성 높음. **여러 알고리즘 공존 확장을 방해하지 않는 선에서 구조 설계**.

## 개념 모델

```
섹션/칩  =  Filter  +  Ranker
            │           │
            │           └─ 후보 풀 → 정렬·상위 N  (교체 가능한 전략)
            └─ 메타 조건(카테고리/태그/...) → 후보 스킬 풀
```

- **Filter (필터)**: 섹션의 메타 조건으로 "이 섹션에 들어갈 수 있는 후보 스킬 풀"을 결정. (S4에서 구체화)
- **Ranker (랭커)**: 후보 풀을 정렬해 상위 N을 산출하는 **전략(strategy)**. 이번 스펙에서 2개:
  - `AsIsRanker` = **A (대조군)** — 현재 동작 그대로 (태그 섹션=수동 큐레이션 순서, recentPurchasedSkills=구매 반자동).
  - `PopularityScoreRanker v1` = **B (실험군)** — 인기차트 문서 스코어. (S3에서 구체화)
- **Variant Resolver (변형 선택)**: 요청/유저 버킷에 따라 어느 랭커를 적용할지 결정 → API가 그 결과 목록을 서빙.

## A/B = 랭커 전략 선택 (두 레이어 구분)

S1에서 확인된 것과 혼동 주의 — A/B가 **두 층**에 존재:

| 레이어 | 무엇을 가르나 | 메커니즘 | 소관 |
|--------|--------------|----------|------|
| ① 컨테이너 롤아웃 | featuredSkillsTabs(신규 칩 UI) vs legacy recentPurchasedSkills | FeatureFlag + 핵클 **키 18** | PR #2414 (기존) |
| ② **랭킹 실험** | 같은 목록을 **현재 순서(A)** vs **자동 랭킹(B)** | **별도·순차** (①  종료 후 별도 실험 배포) | **이 프로젝트** |

**결정 (2026-06-04)**: ②는 ① 컨테이너 A/B가 **종료된 뒤 별도 실험으로 순차 배포**한다 (중첩·2x2 아님). 따라서 **이번 단계에서는 A/B 구현·버킷팅 방식을 설계하지 않는다** — A/B 구분 방식은 **랭킹 알고리즘 적용 이후 단계**에서 별도 진행. 적용 단위 의도는 **유저 버킷 단위·일괄**(한 유저는 인기스킬 섹션 전체를 A or B로 통일).

→ S2의 구조적 요구는 단 하나로 압축됨: **시스템이 동일 후보 풀에 대해 A(AsIs)·B(자동) 두 목록을 산출할 수 있고, "어느 것을 서빙할지"를 외부 신호로 주입받는 지점(variant 주입점)만 깔끔히 분리**되어 있을 것. 그 신호의 출처·구현은 후속 단계 책임.

## 확장성 (다중 랭커 비파괴)

- **랭커 인터페이스 1개**로 추상화 (입력=후보 풀+컨텍스트, 출력=정렬 목록). 이번엔 `AsIs` + `PopularityScore v1` 둘만 구현.
- **섹션 → 랭커 바인딩을 데이터/설정화**하면 알고리즘 추가가 구조를 깨지 않음. (실제 스키마는 S5)
- 단서: PR #2414가 칩에 `showRanking`(bool)을 이미 둠 → 장기적으로 `ranker_id`로 일반화할 여지. (S5에서 판단)

## 이번 스펙 범위 경계

| 항목 | 이번 스펙 | 범위 밖 (확장 대비만) |
|------|-----------|----------------------|
| 랭커 | 자동 1종(+대조 AsIs) | 다중 랭커 "구현" |
| 추상화 | 랭커 인터페이스 + 비파괴 확장 가능 구조 | 섹션별 다른 랭커 실제 운영 |
| 필터 | 현재 정의된 섹션 조건까지 | 신규 섹션·신규 필터축 |

## 결정 요소 (S2) — 결과

- **DF-S2-1 — A/B 적용 단위**: ✅ **유저 버킷 단위·일괄** (한 유저는 인기스킬 섹션 전체를 A or B로 통일).
- **DF-S2-2 — 기존 핵클 키18과의 관계**: ✅ **별도·순차**. 컨테이너 A/B 종료 후 별도 실험 배포. **A/B 구현·버킷팅은 이번 단계 범위 밖** (알고리즘 적용 이후 결정).
- **(S3/S4로 이월) DF-S2-3 — B의 자동화 범위**: 정렬(랭킹)만 자동화인가, 후보 선정(필터링)도 포함인가. → S3 랭킹·S4 필터 구체화 시 확정.

## S2 결론

개념 모델 = **섹션 = 필터(후보 풀) + 랭커(교체 가능 전략)**. 이번 스펙은 랭커 2종(AsIs=A, PopularityScore v1=B)을 산출 가능하게 만들고, "어느 랭커를 서빙할지"는 **variant 주입점만 분리**해 둔다(A/B 메커니즘 자체는 후속). 랭커는 인터페이스로 추상화해 다중 알고리즘 공존을 비파괴로 허용하되, 다중 랭커 운영은 이번 범위 밖.

## S5(아키텍처)로 넘기는 것

랭커 인터페이스 실제 시그니처, 섹션→랭커 바인딩 스키마(`showRanking` 일반화 여부), variant 주입점 위치, 2종 목록 산출·캐싱 방식. (variant 신호의 출처·버킷팅은 알고리즘 이후 별도 단계)
