# 02. 결제·쿠폰 취소 가이드 — 사용한 쿠폰의 취소 관리

> **상태**: 🟡 초안 (2026-06-02) · **대상**: 내부 운영팀 · **도구**: AdminJS `CoopMarketingCouponUsage`
> 선행 문서: [README — 개요·구조·용어](README.md)

사용 완료된 카카오 선물하기 쿠폰을 운영자가 취소하고, 지급된 상품(하트/스킬)을 회수하며, DB 불일치를 정합화하는 절차입니다.

---

## 1. 취소 3종 구분 (먼저 이해)

| 종류 | 누가/언제 | 동작 | 회수 |
|------|----------|------|------|
| **사용취소 (L2)** | **운영자 수동** (AdminJS) | 쿠프마케팅에 사용 취소 요청 | 지급 상품 회수 동반 |
| **망취소 (L3)** | 서버 자동 | L1(사용승인) 타임아웃 시 자동 원복 | 상품 미지급 상태라 회수 불필요 |
| **자동 보상 취소 (L2)** | 서버 자동 | L1 성공 후 상품 지급 실패 시 | 상품 미지급 상태라 회수 불필요 |

> 사용자가 **정상 사용한** 쿠폰은 원칙적으로 취소 불가(콘텐츠형 상품). 운영자 사용취소는 CS·오류 대응 등 예외 처리용입니다.

---

## 2. 운영자 사용취소 절차 (L2)

1. AdminJS → `CoopMarketingCouponUsage` → 대상 행 검색
   - 검색 키: `coupon_code` 또는 `user_seq`
   - 상태 확인: `status = used` 인 행만 취소 대상
2. **취소(사용취소) 액션** 실행 <!-- 정확한 버튼 라벨 어드민 확인 필요 -->
3. 확인(Guard) 메시지에서 회수 가능 여부 확인 후 진행
   - 하트: 충전 수량 + 현재 잔여 하트 (잔액 부족 시 경고)
   - 스킬: 이용권 사용 여부 (사용 완료면 경고)
4. 실행 시 서버가 순서대로 처리:
   - **L2 취소 API 호출** (쿠프마케팅) — `brand_auth_code` 사용
   - **상품 회수** (아래 §3)
   - `status → canceled`, `canceled_at` 기록, `recovered_at` 기록

> 회수 불가 상태(잔액 부족·이미 사용 등)에서도 취소는 진행 가능. 그 경우 운영자가 부족분을 인지하고 별도 조치합니다(§5).

---

## 3. 상품 회수 동작

| 상품 | 회수 방식 | 부분 회수 |
|------|----------|----------|
| 하트 충전권 | `useHeart` 로 충전분 차감 + 회수 로그 | 잔액 부족 시 **남은 만큼만 차감** 후 `recovered_at` 마킹 |
| 스킬 교환권 | 발급된 100% 할인 쿠폰 삭제(`Coupon.delete`) | 사용 완료된 이용권은 회수 불가 — 경고 후 진행 |

- `recovered_at` 은 **회수 1회만 수행**하도록 막는 가드입니다(ISS-056). 이미 `recovered_at` 이 찍힌 행은 재취소해도 상품을 다시 차감하지 않습니다.

---

## 4. ⚠ 연타 금지 (ISS-056 배경)

과거(2026-05-06) 운영 사용취소 버튼을 **연속 클릭**했을 때 하트가 클릭 횟수만큼 중복 회수되는 사고가 있었습니다.

- 원인: 쿠프마케팅 L2 가 이미 취소된 쿠폰에 `8005`(이미 취소) 를 반환하는데 이를 조용히 무시 + 회수 가드(`recovered_at`) 부재.
- 조치(해결, 2026-05-20 패치 / 5/21 배포):
  - L2 응답 `0000`(성공) 과 `8005`(이미 취소) **둘 다 정상 처리(멱등)** → `canceled_at = COALESCE(canceled_at, NOW())`
  - `recovered_at` 컬럼 추가 → 회수는 1회만
- **운영 수칙**: 취소는 **한 번만** 클릭하고 결과를 기다립니다. 가드가 들어왔지만, 불필요한 연타는 여전히 피합니다.

## 5. 잔액 부족 부분 회수 시 운영자 조치

하트 회수 시 사용자가 이미 하트를 써서 잔액이 부족하면 **가능한 만큼만 차감**되고 `recovered_at` 이 찍힙니다.

- 어드민/로그에서 실제 차감량과 원래 충전량의 차이를 확인
- 부족분 처리 정책(추가 회수 보류 / CS 안내 등)은 운영 기준에 따름 <!-- 정책 확정 필요 -->
- 회수가 완료로 마킹되므로, 추가 보정이 필요하면 §6 정합화로 수동 처리

## 6. 정합화 (Reconciliation)

DB 상태와 실제 취소/회수 상태가 어긋난 경우(과거 사고분, 부분 실패 등) SQL 로 보정합니다.

**A. 진단 — 멈춰 있는(취소 미완료) 행 찾기**
```sql
SELECT seq, user_seq, coupon_code, status, canceled_at, recovered_at
FROM coop_marketing_coupon_usage
WHERE status = 'used' AND canceled_at IS NULL
  AND coupon_code = :coupon_code;   -- 또는 기간/유저 조건
```

**B. 보정 — 수동 정합화 (개발 검토 후 실행)**
```sql
UPDATE coop_marketing_coupon_usage
SET status = 'canceled',
    canceled_at = :canceled_time,
    recovered_at = :recovered_time   -- 실제 회수 시점(heart_log 등으로 확인)
WHERE coupon_code = :coupon_code;
```

> ⚠ 정합화 SQL 은 **운영 DB 직접 변경**입니다. 운영자가 단독 실행하지 말고 `/dev-server`/개발과 함께 영향 범위 확인 후 실행하세요. 회수 시점(`recovered_time`)은 `heart_log` 의 최초 회수 기록으로 확인합니다.

## 7. 에러·엣지 케이스

| 상황 | 동작 / 대응 |
|------|------------|
| 취소 후 같은 쿠폰 재사용 | `(user_seq, coupon_code)` UNIQUE — UPSERT 로 기존 행 갱신(재사용 지원). 별도 오류 아님 |
| L2 가 `8005`(이미 취소) 반환 | 정상(멱등) 처리됨 — 추가 조치 불필요 |
| L1+L3 동시 실패(망취소도 실패) | 자동 복구 불가(ISS-052) — `coop_marketing_api_log`/winston 로그로 확인 후 CS·개발 수동 대응 |
| L2 호출 자체 실패(통신 오류 등) | 취소 미완료 — 재시도 또는 개발 문의. `status` 가 `used` 로 남음 → §6 진단으로 추적 |
